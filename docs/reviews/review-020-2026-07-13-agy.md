## Некритичные замечания

### [MINOR] Событие `settings-changed` не является единым контрактом для всех настроек
- **Замечание**: `useAppSettings` перезагружает единый DTO только после `settings-changed`, но значительная часть команд, изменяющих конфигурацию, это событие не отправляет. Например, аудио-команды находятся в `src-tauri/src/commands/playback.rs:93-160`, AI-настройки — в `src-tauri/src/commands/ai.rs:15-116`, прокси — в `src-tauri/src/commands/proxy.rs:206-265`, тогда как событие явно отправляется лишь отдельными командами (`playback.rs:396`, `webview.rs:101`, `commands/mod.rs:192-261`).
- **Файл**: `src-tauri/src/commands/ai.rs:15`
- **Решение**: Зафиксировать правило: любая команда, меняющая persisted settings, публикует одно типизированное событие с изменённым разделом/версией; либо перейти на единый command-layer, который автоматически публикует событие после успешной записи.

### [MINOR] Два разных менеджера конфигурации используют разные гарантии согласованности
- **Замечание**: Основной `SettingsManager` держит in-memory cache и общий lock для `settings.json` (`src-tauri/src/config/settings.rs:841-955`), а `WindowsManager` каждый раз читает/перезаписывает `windows.json` без cache и общей блокировки (`src-tauri/src/config/windows.rs:242-270`, setters `301-345`). При этом единый DTO собирается из обоих источников в `src-tauri/src/commands/mod.rs:105-155`.
- **Файл**: `src-tauri/src/config/windows.rs:242`
- **Решение**: Вынести общий persistence primitive (cache, lock, atomic write, validate/migrate) и использовать его для обоих файлов.

### [MINOR] Runtime TTS state и persisted settings имеют несколько источников истины
- **Замечание**: TTS runtime хранится в `AppState::tts_config` (`src-tauri/src/state.rs:83-85`), persisted значения — в `SettingsManager`, а команды используют их непоследовательно. Например, `set_openai_api_key` сначала меняет runtime и инициализирует TTS, затем сохраняет файл (`src-tauri/src/commands/ai.rs:574-589`); `set_fish_audio_api_key` делает то же (`ai.rs:668-683`), но другие getters читают непосредственно `SettingsManager` (`ai.rs:594-598`, `ai.rs:688-719`).
- **Файл**: `src-tauri/src/commands/ai.rs:574`
- **Решение**: Валидировать и атомарно сохранить новый snapshot в файл, затем применять его к runtime, либо иметь транзакционный сервис `update_settings_and_runtime`. Запретить прямое изменение `tts_config` из команд.

### [MINOR] Unified DTO сосуществует с разрозненными IPC-чтениями и обходом типового контракта
- **Замечание**: Приложение имеет `get_all_app_settings` и типизированный `AppSettingsDto`, но часть UI продолжает вызывать отдельные getters (`HotkeysPanel.vue:19`, `AudioPanel.vue:245`), а `useTelegramAuth.ts:280` использует `invoke<any>('get_all_app_settings')`.
- **Файл**: `src/composables/useTelegramAuth.ts:280`
- **Решение**: Разделить IPC на явно выбранные read-models: либо один DTO для начальной загрузки и реактивного обновления, либо типизированные доменные composables. Убрать `any`, вынести тип `AppSettingsDto` в общий слой.

### [MINOR] Перезагрузка настроек может потерять событие во время активной загрузки
- **Замечание**: `createAppSettings.load()` сразу выходит, если `isLoading.value` уже установлен (`src/composables/useAppSettings.ts:54-58`). Обработчик `settings-changed` вызывает `reload()` без очереди (`useAppSettings.ts:113-117`).
- **Файл**: `src/composables/useAppSettings.ts:54`
- **Решение**: Использовать coalescing-флаг `reloadRequested` или serial queue: если событие пришло во время загрузки, выполнить ровно один reload после завершения текущего запроса.

### [MINOR] Блокирующие синхронные операции ввода-вывода в Tauri-командах (Main Thread Blocking)
- **Замечание**: Почти все setter-команды в `ai.rs` (например, `set_ai_provider`, `set_openai_api_key`, `set_openai_voice`, `set_fish_audio_api_key` и др.) объявлены как синхронные (`pub fn` вместо `pub async fn`). Они выполняют синхронные дисковые операции через `settings_manager` (чтение, парсинг и запись JSON) под глобальным мьютексом `settings_write_lock()`. В Tauri синхронные команды выполняются на UI-потоке, что может вызывать фризы UI при задержках ввода-вывода на диске.
- **Файл**: `src-tauri/src/commands/ai.rs:15`
- **Решение**: Перевести setter-команды на `pub async fn` и производить запись неблокирующим образом (через async-версию записи файла или `tokio::task::spawn_blocking`).

### [MINOR] Паника и крах приложения при повреждении конфигурационных файлов
- **Замечание**: Инициализация `SettingsManager` и `WindowsManager` в `lib.rs` использует метод `.expect()`. Если файлы `settings.json` или `windows.json` на диске окажутся поврежденными (содержат пустую строку или невалидный JSON), десериализация serde завершится ошибкой, что приведет к немедленной панике и краху приложения на этапе старта.
- **Файл**: `src-tauri/src/lib.rs:59`
- **Решение**: Обеспечить отказоустойчивость: перехватывать ошибки десериализации, переименовывать битый файл во временный бэкап (например, `settings.json.bak`) и создавать чистый конфигурационный файл со значениями по умолчанию.

### [MINOR] Неуправляемый жизненный цикл низкоуровневого хука клавиатуры Windows
- **Замечание**: Поток хука в `initialize_soundpanel_hook` запускает бесконечный цикл `GetMessageW` без возможности корректного завершения и снятия хука. Это препятствует освобождению системных ресурсов при закрытии/перезагрузке компонентов.
- **Файл**: `src-tauri/src/soundpanel/hook.rs:73`
- **Решение**: Предусмотреть механизм отмены (например, отправкой `WM_QUIT` потоку хука) для контролируемого выхода из цикла `GetMessageW` и вызова `UnhookWindowsHookEx`.

## Предложения по оптимизации

### [OPTIMIZE] Избыточное создание SettingsManager и дисковое чтение при каждом синтезе речи (hot path)
- **Текущее**: В `speak_text_internal` (`commands/mod.rs:63`), `synthesize_and_export` (`tts_pipeline.rs:178`), а также при инициализации хоткеев создается новый экземпляр `SettingsManager` через `SettingsManager::new()`. Это приводит к принудительному дисковому чтению и парсингу `settings.json` при каждой отправке текста на TTS.
- **Предложение**: Извлекать `SettingsManager` из глобального состояния Tauri (`app_handle.state::<SettingsManager>()`), где он хранится как синглтон с RwLock-кэшем.
- **Выгода**: Полное исключение дискового ввода-вывода и парсинга JSON на горячем пути генерации речи. Снижение задержки (latency) перед озвучиванием.

### [OPTIMIZE] Ввести единый application settings service
- **Текущее**: Команды напрямую вызывают десятки методов `SettingsManager`, часть команд отдельно трогает AppState, часть сама эмитит события.
- **Предложение**: Сделать слой доменных операций (`settings service`) с правилами validate → persist → apply runtime → emit. Tauri-команды оставить тонкими адаптерами IPC.
- **Выгода**: Единая транзакционная семантика, меньше дублирования и проще тестировать последовательности обновления.

### [OPTIMIZE] Типизировать события и IPC-названия
- **Текущее**: Имена событий и payload передаются строками (`"settings-changed"`, `"tts-provider-changed"`), payload часто пустой.
- **Предложение**: Централизовать event names и определить payload с `section`, `revision` и, где нужно, DTO. Для команд использовать единый TypeScript registry или генерацию типов.
- **Выгода**: Меньше расхождений между backend и frontend, возможность верификации актуальности snapshot.

### [OPTIMIZE] Разделить persisted settings, runtime state и read-model в документации
- **Текущее**: `AppSettingsDto` объединяет файл конфигурации, `WindowsManager`, runtime interception/preprocessor и soundpanel storage (`src-tauri/src/commands/mod.rs:116-144`).
- **Предложение**: В архитектурной схеме явно обозначить три слоя: durable config, runtime state и UI read-model; для каждого поля указать owner, persistence и event.
- **Выгода**: Новые фичи не будут случайно добавлять ещё один источник истины.

### [OPTIMIZE] Добавить end-to-end contract tests для настроек
- **Текущее**: Существующие тесты хорошо покрывают serde/совместимость `SettingsManager`, но не проверяют путь command → event → frontend DTO.
- **Предложение**: Тестировать сохранение каждого раздела, наличие события, повторное чтение из unified DTO и поведение при concurrent updates.
- **Выгода**: Архитектурные регрессии будут обнаруживаться до ручной проверки UI.

## Замечания по информационной безопасности

### [SECURITY] Единый DTO расширяет поверхность передачи чувствительных настроек
- **Уязвимость**: `get_all_app_settings` собирает много доменов в один IPC-ответ. Любое добавление поля с секретом автоматически делает его доступным всем компонентам, имеющим доступ к общему read-model. Дополнительно типы frontend не должны скрывать это через `any`.
- **Файл**: `src-tauri/src/config/dto.rs:984`
- **Решение**: Разделить public settings DTO и secret/runtime DTO; API keys/tokens возвращать только как `has_value`/masked metadata, если полное значение не требуется конкретному экрану. Это ограничит их ненужную передачу по IPC.
