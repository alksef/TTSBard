## Некритичные замечания

### [MINOR] Событие `settings-changed` не является единым контрактом для всех настроек

- **Проблема**: `useAppSettings` перезагружает единый DTO только после `settings-changed`, но значительная часть команд, изменяющих конфигурацию, это событие не отправляет. Например, аудио-команды находятся в `src-tauri/src/commands/playback.rs:93-160`, AI-настройки — в `src-tauri/src/commands/ai.rs:15-116`, прокси — в `src-tauri/src/commands/proxy.rs:206-265`, тогда как событие явно отправляется лишь отдельными командами (`playback.rs:396`, `webview.rs:101`, `commands/mod.rs:192-261`).
- **Риск**: `AppSettingsDto` во Vue может оставаться устаревшим после успешного сохранения. Компоненты, читающие DTO, и компоненты, вызывающие индивидуальные `get_*`, начинают видеть разные значения.
- **Решение**: зафиксировать правило: любая команда, меняющая persisted settings, публикует одно типизированное событие с изменённым разделом/версией; либо перейти на единый command-layer, который автоматически публикует событие после успешной записи. Добавить матрицу «setter → event → consumer» и тест на каждый раздел.

### [MINOR] Два разных менеджера конфигурации используют разные гарантии согласованности

- **Проблема**: основной `SettingsManager` держит in-memory cache и общий lock для `settings.json` (`src-tauri/src/config/settings.rs:841-955`), а `WindowsManager` каждый раз читает/перезаписывает `windows.json` без cache и общей блокировки (`src-tauri/src/config/windows.rs:242-270`, setters `301-345`). При этом единый DTO собирается из обоих источников в `src-tauri/src/commands/mod.rs:105-155`.
- **Риск**: параллельные изменения окон могут потерять одно из обновлений; общая модель «загрузить → изменить → сохранить» работает по-разному в разных доменах. Нельзя гарантировать одинаковую семантику миграции, atomic write и concurrent update для всех настроек.
- **Решение**: вынести общий persistence primitive (cache, lock, atomic write, validate/migrate) и использовать его для обоих файлов; отдельно документировать, почему настройки разделены на `settings.json` и `windows.json`.

### [MINOR] Runtime TTS state и persisted settings имеют несколько источников истины

- **Проблема**: TTS runtime хранится в `AppState::tts_config` (`src-tauri/src/state.rs:83-85`), persisted значения — в `SettingsManager`, а команды используют их непоследовательно. Например, `set_openai_api_key` сначала меняет runtime и инициализирует TTS, затем сохраняет файл (`src-tauri/src/commands/ai.rs:574-589`); `set_fish_audio_api_key` делает то же (`ai.rs:668-683`), но другие getters читают непосредственно `SettingsManager` (`ai.rs:594-598`, `ai.rs:688-719`).
- **Риск**: при ошибке записи файл и runtime расходятся до перезапуска. Отдельные команды могут вернуть разные значения в зависимости от того, читают ли они cache manager или AppState.
- **Решение**: выбрать владельца конфигурации: сначала валидировать и атомарно сохранить новый snapshot, затем применять его к runtime, либо иметь транзакционный сервис `update_settings_and_runtime`. Запретить прямое изменение `tts_config` из команд, кроме этого сервиса.

### [MINOR] Unified DTO сосуществует с разрозненными IPC-чтениями и обходом типового контракта

- **Проблема**: приложение имеет `get_all_app_settings` и типизированный `AppSettingsDto`, но часть UI продолжает вызывать отдельные getters (`HotkeysPanel.vue:19`, `AudioPanel.vue:245`), а `useTelegramAuth.ts:280` использует `invoke<any>('get_all_app_settings')`.
- **Риск**: изменения DTO не проверяются TypeScript-компилятором во всех потребителях; разные компоненты получают конфигурацию в разные моменты и с разной обработкой ошибок.
- **Решение**: разделить IPC на явно выбранные read-models: либо один DTO для начальной загрузки и реактивного обновления, либо типизированные доменные composables. Убрать `any`, вынести тип `AppSettingsDto` в общий generated/contract слой или добавить contract tests Rust↔TypeScript.

### [MINOR] Перезагрузка настроек может потерять событие во время активной загрузки

- **Проблема**: `createAppSettings.load()` сразу выходит, если `isLoading.value` уже установлен (`src/composables/useAppSettings.ts:54-58`). Обработчик `settings-changed` вызывает `reload()` без очереди (`useAppSettings.ts:113-117`).
- **Риск**: событие изменения, пришедшее во время предыдущего `get_all_app_settings`, не приводит к повторной загрузке; UI может остаться со старым snapshot до следующего изменения.
- **Решение**: использовать coalescing-флаг `reloadRequested` или serial queue: если событие пришло во время загрузки, выполнить ровно один reload после завершения текущего запроса.

## Предложения по оптимизации

### [OPTIMIZE] Ввести единый application settings service

- **Текущее**: команды напрямую вызывают десятки методов `SettingsManager`, часть команд отдельно трогает AppState, часть сама эмитит события.
- **Предложение**: сделать слой доменных операций (`settings service`) с правилами validate → persist → apply runtime → emit. Tauri-команды оставить тонкими адаптерами IPC.
- **Выгода**: единая транзакционная семантика, меньше дублирования и проще тестировать последовательности обновления.

### [OPTIMIZE] Типизировать события и IPC-названия

- **Текущее**: имена событий и payload передаются строками (`"settings-changed"`, `"tts-provider-changed"`), payload часто пустой.
- **Предложение**: централизовать event names и определить payload с `section`, `revision` и, где нужно, DTO. Для команд использовать единый TypeScript registry или генерацию типов из Rust DTO.
- **Выгода**: меньше расхождений между backend и frontend, можно проверять порядок и актуальность snapshot.

### [OPTIMIZE] Разделить persisted settings, runtime state и read-model в документации

- **Текущее**: `AppSettingsDto` объединяет файл конфигурации, `WindowsManager`, runtime interception/preprocessor и soundpanel storage (`src-tauri/src/commands/mod.rs:116-144`).
- **Предложение**: в архитектурной схеме явно обозначить три слоя: durable config, runtime state и UI read-model; для каждого поля указать owner, persistence и event.
- **Выгода**: новые фичи не будут случайно добавлять ещё один источник истины.

### [OPTIMIZE] Добавить end-to-end contract tests для настроек

- **Текущее**: существующие тесты хорошо покрывают serde/совместимость `SettingsManager`, но не проверяют путь command → event → frontend DTO.
- **Предложение**: тестировать как минимум сохранение каждого раздела, наличие события, повторное чтение из unified DTO и поведение при concurrent updates для `settings.json` и `windows.json`.
- **Выгода**: архитектурные регрессии будут обнаруживаться до ручной проверки UI.

## Замечания по информационной безопасности

### [SECURITY] Единый DTO расширяет поверхность передачи чувствительных настроек

- **Уязвимость**: `get_all_app_settings` собирает много доменов в один IPC-ответ (`src-tauri/src/commands/mod.rs:105-155`). Сейчас это удобно, но любое добавление поля с секретом автоматически делает его доступным всем компонентам, имеющим доступ к общему read-model. Дополнительно типы frontend не должны скрывать это через `any`.
- **Файл**: `src-tauri/src/config/dto.rs:984-1010`, `src/types/settings.ts:324-338`.
- **Решение**: разделить public settings DTO и secret/runtime DTO; API keys/tokens возвращать только как `has_value`/masked metadata, если полное значение не требуется конкретному экрану. Это не отменяет принятого в проекте допущения о хранении токенов в открытом виде, а ограничивает их ненужную передачу по IPC.

## Оценка точечного использования DeepSeek

DeepSeek здесь может помочь именно в реализации небольших, изолированных изменений, но не должен принимать архитектурное решение за проект. По правилам `docs/deepseek/WORKFLOW.md` задачу нужно дробить по bounded file set и после каждого прогона независимо проверять diff и тесты.

Рекомендуемые отдельные задачи:

1. Единый helper/сервис для `settings-changed` и миграция setter-команд по одному домену, например audio.
2. Coalescing reload в `useAppSettings.ts` с тестом на событие во время загрузки.
3. Устранение `invoke<any>` и выравнивание типов одного frontend-домена.
4. Общий persistence primitive для `SettingsManager`/`WindowsManager` и тест concurrent update.

Не стоит отдавать DeepSeek одним заданием «унифицировать всю архитектуру конфигурации»: это нарушит правило декомпозиции из `WORKFLOW.md`, смешает IPC, UI, runtime и persistence и усложнит независимое ревью.
