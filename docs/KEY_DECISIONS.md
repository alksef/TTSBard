# Key Architecture Decisions

> **Почему мы сделали именно так?** Документирование ключевых архитектурных решений и их обоснование.

---

## 1. Windows-only Platform (WH_KEYBOARD_LL)

**Решение:** Использовать Windows Low-Level Keyboard Hook для перехвата клавиш.

**Почему:**
- ✅ Глобальный перехват клавиш в любом приложении
- ✅ Работает на user-mode уровне (не нужен драйвер ядра)
- ✅ Минимальное влияние на производительность
- ✅ Простота установки и удаления

**Альтернативы, которые были рассмотрены:**
- ❌ **Кроссплатформенные решения** (Electron globalShortcut) — не позволяют перехватывать произвольный текст
- ❌ **Kernel-mode драйвер** — слишком сложная разработка и сертификация
- ❌ **Сlipboard-based решения** — ненадёжные, требуют копирования в буфер

**Ограничения:**
- Только Windows 10/11
- Не перехватывает клавиши от приложений с правами Administrator
- Не может блокировать `Ctrl+Alt+Del` (Secure Desktop)

## 2. Tauri 2.0 вместо Electron

**Решение:** Использовать Tauri 2.0 вместо Electron.

**Почему:**
- ✅ **Меньший размер бинарника** — ~10 MB vs ~200 MB (Electron)
- ✅ **Меньшее потребление памяти** — Rust backend + WebView
- ✅ **Безопасность** — Rust memory safety + сниженная поверхность атаки
- ✅ **Нативная интеграция** — прямой доступ к Windows API

**Альтернативы:**
- ❌ **Electron** — слишком тяжелый для такой задачи
- ❌ **Pure Rust UI (egui, iced)** — недостаточно гибкий для сложного UI
- ❌ **Flutter Desktop** — избыточный для Desktop-приложения

**Компромисс:**
- Ограничение на Windows-only (из-за keyboard hook)
- Но это приемлемо, так как hook всё равно требует Windows API

## 3. Event-Driven Architecture (MPSC)

**Решение:** Использовать MPSC channels для асинхронной коммуникации между компонентами.

**Почему:**
- ✅ **Развязка компонентов** — hook не зависит от TTS сервиса
- ✅ **Асинхронность** — не блокирует UI thread
- ✅ **Расширяемость** — легко добавить новых подписчиков
- ✅ **Тестопригодность** — можно мокать события

**Паттерн:**
```rust
// Event definition
pub enum AppEvent {
    InterceptionChanged(bool),
    TextReady(String),
    // ...
}

// Channel setup
let (tx, rx) = mpsc::channel::<AppEvent>();

// Emit
tx.send(AppEvent::TextReady(text))?;

// Handle
while let Some(event) = rx.recv().await {
    match event { /* ... */ }
}
```

## 4. Command Pattern (Tauri Commands)

**Решение:** Использовать Tauri commands как мост между Frontend и Backend.

**Почему:**
- ✅ **Type Safety** — TypeScript типы可与 Rust типами
- ✅ **Автоматическая сериализация** — serde handles JSON
- ✅ **Безопасность** — команды изолированы от прямого доступа к состоянию
- ✅ **Асинхронность** — async/await из коробки

**Паттерн:**
```rust
#[tauri::command]
async fn command_name(
    state: State<'_, AppState>,
    tx: EventSender,
    param: String,
) -> Result<(), String> {
    // ...
}
```

## 5. Thread-Safe State (Arc<Mutex<T>>)

**Решение:** Использовать `Arc<Mutex<T>>` для shared state.

**Почему:**
- ✅ **Thread safety** — безопасный доступ из разных потоков
- ✅ **Async/await** — работает с Tokio runtime
- ✅ **Простота** — стандартный подход для Rust

**Альтернативы:**
- ❌ **RwLock** — не нужен, так как в основном пишем, а не читаем
- ❌ **Channels** — избыточно для простого state
- ❌ **Actor model** — слишком сложно для этого случая

## 6. Plugin Architecture (SoundPanel)

**Решение:** Выделить SoundPanel в отдельный модуль с собственной state/storage.

**Почему:**
- ✅ **Инкапсуляция** — SoundPanel логика изолирована
- ✅ **Тестируемость** — можно тестировать отдельно
- ✅ **Отключаемость** — легко убрать или заменить
- ✅ **Переиспользование** — можно использовать в других проектах

**Структура:**
```
soundpanel/
├── mod.rs       # Public interface
├── state.rs     # State management
├── bindings.rs  # Tauri commands
├── storage.rs   # Persistence
├── audio.rs     # Audio playback
└── hook.rs      # Keyboard hook
```

## 7. Multiple TTS Providers

**Решение:** Поддерживать несколько TTS провайдеров через абстракцию.

**Провайдеры:**
- **OpenAI TTS** — cloud-based, высокое качество
- **Silero Bot (Telegram)** — бесплатный альтернативный провайдер
- **TTSVoiceWizard (Local)** — локальный TTS для офлайн работы

**Почему:**
- ✅ **Резервирование** — если один не работает, используем другой
- ✅ **Гибкость** — пользователь выбирает провайдер
- ✅ **Стоимость** — можно переключаться на бесплатные варианты
- ✅ **Офлайн режим** — Local TTS работает без интернета

**Абстракция:**
```rust
pub trait TtsEngine {
    async fn speak(&self, text: &str) -> Result<(), String>;
}
```

## 8. Dual Audio Output

**Решение:** Поддерживать одновременный вывод на два устройства (динамики + виртуальный микрофон).

**Почему:**
- ✅ **Стриминг** — можно слышать звук и транслировать в Discord/Zoom
- ✅ **Мониторинг** — пользователь слышит, что отправляется в микрофон
- ✅ **Гибкость** — можно включить/выключить любое устройство

**Реализация:**
- Используем rodio для воспроизведения на выбранном устройстве
- Виртуальный микрофон (VB-Cable, VoiceMeeter) для трансляции

## 9. Text Preprocessor (Presets)

**Решение:** Добавить препроцессор для быстрой замены коротких кодов на полный текст.

**Синтаксис:**
- `\key` — замена текста по коду
- `%username` — замена имени пользователя

**Почему:**
- ✅ **Скорость ввода** — `\name ` быстрее, чем "Александр"
- ✅ **Часто используемые фразы** — быстрые пресеты для greetings и т.д.
- ✅ **Имена пользователей** — быстрое упоминание пользователей в чате
- ✅ **Мгновенная замена** — замена происходит после пробела

---

## 10. WebView Server for OBS

**Решение:** Добавить HTTP/WebSocket сервер для интеграции с OBS Studio.

**Почему:**
- ✅ **Стриминг** — можно показывать текст в OBS
- ✅ **Real-time** — WebSocket для мгновенных обновлений
- ✅ **Кастомизация** — можно настроить HTML/CSS под свой дизайн
- ✅ **Удалённый доступ** — можно использовать с другого компьютера

**Альтернативы:**
- ❌ **OBS Plugin** — сложнее в разработке и установке
- ❌ **Text file watching** — не real-time
- ❌ **External service** — зависит от интернета

---

## 11. Twitch Chat Integration

**Решение:** Добавить интеграцию с Twitch Chat для отправки TTS сообщений.

**Почему:**
- ✅ **Интерактивность** — зрители могут видеть сообщения в чате
- ✅ **Стримеры** — полезно для Twitch стримеров
- ✅ **Автоматическое переподключение** — устойчивость к разрывам
- ✅ **Start on boot** — автоподключение при запуске

**Реализация:**
- IRC клиент для подключения к Twitch
- OAuth токен для авторизации
- Автоматическое переподключение

---

## 12. System Tray Integration

**Решение:** Сворачивать приложение в системный трей вместо закрытия.

**Почему:**
- ✅ **Всегда доступно** — приложение работает в фоне
- ✅ **Не загромождает taskbar** — не мешает работе
- ✅ **Быстрый доступ** — двойной клик для разворачивания
- ✅ **Контекстное меню** — быстрый доступ к функциям

**Поведение:**
- Закрытие главного окна → сворачивание в трей
- Двойной клик → разворачивание
- Правый клик → контекстное меню

---

## 13. Multiple Floating Windows

**Решение:** Поддерживать несколько floating windows с разными режимами.

**Окна:**
- **Main Window** — настройки приложения
- **Floating Window** — текстовый ввод (перехват)
- **SoundPanel Window** — звуковая панель

**Почему:**
- ✅ **Взаимное исключение** — только одно активно одновременно
- ✅ **Разные режимы** — каждое окно для своей задачи
- ✅ **Независимые настройки** — opacity, color, clickthrough для каждого

**Ограничение:**
- Если активно Floating Window, SoundPanel не откроется, и наоборот
- Это предотвращает конфликты hotkeys

---

## 14. F6 Hotkey Mode

**Решение:** Добавить режим F6, где Enter не закрывает floating window.

**Почему:**
- ✅ **Последовательные сообщения** — можно отправить несколько сообщений без закрытия
- ✅ **Удобство** — не нужно постоянно открывать окно
- ✅ **Toggle** — можно включить/выключить по необходимости

**Поведение:**
- F6 выключен: Enter → отправить текст + закрыть окно
- F6 включен: Enter → отправить текст + окно остаётся открытым

## 15. File Storage Strategy

**Решение:** Использовать разные пути для разных типов данных.

**Пути:**
- `%USERPROFILE%\.config\tts-app\settings.json` — основные настройки
- `%APPDATA%\ttsbard\replacements.txt` — пресеты
- `%APPDATA%\ttsbard\usernames.txt` — имена пользователей
- `%APPDATA%\ttsbard\soundpanel_bindings.json` — привязки звуковой панели
- `%APPDATA%\ttsbard\soundpanel\` — аудиофайлы

**Почему:**
- ✅ **Стандартные пути** — соответствует Windows conventions
- ✅ **Изоляция** — разные типы данных в разных местах
- ✅ **Легкий бэкап** — можно скопировать только нужное
- ✅ **Переносимость** — легко перенести на другой компьютер

**Альтернативы:**
- ❌ **Program Files** — требует прав администратора
- ❌ **Реестр Windows** — сложнее для бэкапа
- ❌ **База данных** — избыточно для этого случая

## 16. Error Handling Strategy

**Решение:** Использовать `Result<T, String>` для всех операций.

**Почему:**
- ✅ **Простота** — String достаточно для пользовательских ошибок
- ✅ **Пользовательские сообщения** — понятные ошибки на русском/английском
- ✅ **Events для frontend** — отправка ошибок через события

**Паттерн:**
```rust
pub type AppResult<T> = Result<T, String>;

// Emit error event
tx.send(AppEvent::TtsError("API key invalid".to_string()))?;
```

## 17. Audio Playback Strategy

**Решение:** Использовать разные подходы для разных типов аудио.

**TTS Audio:**
- Системный плеер по умолчанию
- Позволяет использовать внешние плееры

**SoundPanel Audio:**
- rodio библиотека для внутрипроцессного воспроизведения
- Асинхронное воспроизведение (non-blocking)

**Почему:**
- ✅ **Гибкость** — TTS можно externally настроить
- ✅ **Производительность** — rodio эффективен для short sounds
- ✅ **Асинхронность** — не блокирует UI

## 18. Window Manager Pattern

**Решение:** Использовать Win32 API для создания floating windows с special styles.

**Стили:**
- `WS_EX_LAYERED` — прозрачность
- `WS_EX_TRANSPARENT` — click-through
- `WS_EX_NOACTIVATE` — без фокуса
- `WS_EX_TOOLWINDOW` — скрыть из taskbar

**Почему:**
- ✅ **Full control** — полный контроль над поведением окна
- ✅ **Always on top** — поверх всех окон
- ✅ **Non-intrusive** — не перехватывает фокус
- ✅ **Click-through** — можно кликать сквозь окно

**Альтернативы:**
- ❌ **Tauri window API** — недостаточно гибкий
- ❌ **Electron frameless window** — не поддерживает click-through корректно

---

## 19. Configuration Management

**Решение:** Выделить конфигурацию в отдельный модуль `config/`.

**Структура:**
- `mod.rs` — exports
- `settings.rs` — settings structs
- `validation.rs` — validation logic
- `windows.rs` — Windows-specific config

**Почему:**
- ✅ **Организация** — весь конфиг в одном месте
- ✅ **Валидация** — централизованная проверка настроек
- ✅ **Платформенная изоляция** — Windows-specific настройки отдельно

## 20. Testing Strategy

**Решение:** Использовать комбинацию unit и integration тестов.

**Unit Tests:**
- TTS client logic
- Sound panel state management
- Settings serialization
- Preprocessor logic

**Integration Tests:**
- Command handlers
- Event emission/handling
- File operations

**Manual Tests:**
- Hook behavior (требует Windows)
- Window interactions
- Audio playback quality

**Почему:**
- ✅ **Баланс** — автоматические тесты для logic, manual для UI
- ✅ **Реалистичность** — manual тесты для user-facing функций
- ✅ **Покрытие** — критические пути покрыты тестами

## Заключение

Эти архитектурные решения были приняты на основе:

1. **Требований проекта** — Windows-only, глобальный перехват клавиш
2. **Ограничений платформы** — Windows API, user-mode
3. **Опыта** — лучшие практики Rust/Tauri/TypeScript
4. **Пользовательских потребностей** — простота, скорость, надёжность

Многие решения являются компромиссами между сложностью, производительностью и удобством использования.
