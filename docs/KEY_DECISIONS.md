# Key Architecture Decisions

> **Почему мы сделали именно так?** Документирование ключевых архитектурных решений и их обоснование.

---

## 1. Windows-only Platform (WH_KEYBOARD_LL)

**Решение:** Использовать Windows Low-Level Keyboard Hook для перехвата клавиш.

**Почему:**
- ✅ Глобальный перехват клавиш в любом приложении
- ✅ Работает на user-mode уровне (не нужен драйвер ядра)
- ✅ Минимальное влияние на производительность
- ✅ Простота установки и удаления

**Альтернативы, которые были рассмотрены:**
- ❌ **Кроссплатформенные решения** (Electron globalShortcut) — не позволяют перехватывать произвольный текст
- ❌ **Kernel-mode драйвер** — слишком сложная разработка и сертификация
- ❌ **Сlipboard-based решения** — ненадёжные, требуют копирования в буфер

**Ограничения:**
- Только Windows 10/11
- Не перехватывает клавиши от приложений с правами Administrator
- Не может блокировать `Ctrl+Alt+Del` (Secure Desktop)

## 2. Tauri 2.0 вместо Electron

**Решение:** Использовать Tauri 2.0 вместо Electron.

**Почему:**
- ✅ **Меньший размер бинарника** — ~10 MB vs ~200 MB (Electron)
- ✅ **Меньшее потребление памяти** — Rust backend + WebView
- ✅ **Безопасность** — Rust memory safety + сниженная поверхность атаки
- ✅ **Нативная интеграция** — прямой доступ к Windows API

**Альтернативы:**
- ❌ **Electron** — слишком тяжелый для такой задачи
- ❌ **Pure Rust UI (egui, iced)** — недостаточно гибкий для сложного UI
- ❌ **Flutter Desktop** — избыточный для Desktop-приложения

**Компромисс:**
- Ограничение на Windows-only (из-за keyboard hook)
- Но это приемлемо, так как hook всё равно требует Windows API

## 3. Event-Driven Architecture (MPSC)

**Решение:** Использовать MPSC channels для асинхронной коммуникации между компонентами.

**Почему:**
- ✅ **Развязка компонентов** — hook не зависит от TTS сервиса
- ✅ **Асинхронность** — не блокирует UI thread
- ✅ **Расширяемость** — легко добавить новых подписчиков
- ✅ **Тестопригодность** — можно мокать события

**Паттерн:**
```rust
// Event definition
pub enum AppEvent {
    InterceptionChanged(bool),
    TextReady(String),
    // ...
}

// Channel setup
let (tx, rx) = mpsc::channel::<AppEvent>();

// Emit
tx.send(AppEvent::TextReady(text))?;

// Handle
while let Some(event) = rx.recv().await {
    match event { /* ... */ }
}
```

## 4. Command Pattern (Tauri Commands)

**Решение:** Использовать Tauri commands как мост между Frontend и Backend.

**Почему:**
- ✅ **Type Safety** — TypeScript типы可与 Rust типами
- ✅ **Автоматическая сериализация** — serde handles JSON
- ✅ **Безопасность** — команды изолированы от прямого доступа к состоянию
- ✅ **Асинхронность** — async/await из коробки

**Паттерн:**
```rust
#[tauri::command]
async fn command_name(
    state: State<'_, AppState>,
    tx: EventSender,
    param: String,
) -> Result<(), String> {
    // ...
}
```

## 5. Thread-Safe State (Arc<Mutex<T>> / Arc<RwLock<T>>)

**Решение:** Использовать `Arc<Mutex<T>>` и `Arc<RwLock<T>>` для shared state.

**Почему:**
- ✅ **Thread safety** — безопасный доступ из разных потоков
- ✅ **Async/await** — работает с Tokio runtime
- ✅ **Простота** — стандартный подход для Rust

**Выбор между Mutex и RwLock:**
- **Mutex** — для state с частой записью (текущее состояние interception)
- **RwLock** — для read-heavy конфигураций (настройки TTS, горячие клавиши)

**Альтернативы:**
- ❌ **Только Mutex** — неоптимально для read-heavy конфигураций
- ❌ **Channels** — избыточно для простого state
- ❌ **Actor model** — слишком сложно для этого случая

## 6. Plugin Architecture (SoundPanel)

**Решение:** Выделить SoundPanel в отдельный модуль с собственной state/storage.

**Почему:**
- ✅ **Инкапсуляция** — SoundPanel логика изолирована
- ✅ **Тестируемость** — можно тестировать отдельно
- ✅ **Отключаемость** — легко убрать или заменить
- ✅ **Переиспользование** — можно использовать в других проектах

**Структура:**
```
soundpanel/
├── mod.rs       # Public interface
├── state.rs     # State management
├── bindings.rs  # Tauri commands
├── storage.rs   # Persistence
├── audio.rs     # Audio playback
└── hook.rs      # Keyboard hook
```

- ✅ **Резервирование** — если один не работает, используем другой

- ✅ **Стоимость** — можно переключаться на бесплатные варианты
- ✅ **Офлайн режим** — Local TTS работает без интернета
- ✅ **Разнообразие голосов** — Fish Audio предоставляет marketplace голосовых моделей

## 8. Dual Audio Output

**Решение:** Поддерживать одновременный вывод на два устройства (динамики + виртуальный микрофон).

**Почему:**
- ✅ **Стриминг** — можно слышать звук и транслировать в Discord/Zoom
- ✅ **Мониторинг** — пользователь слышит, что отправляется в микрофон
- ✅ **Гибкость** — можно включить/выключить любое устройство

**Реализация:**

## 9. Text Preprocessor (Presets)

**Решение:** Добавить препроцессор для быстрой замены коротких кодов на полный текст.

**Синтаксис:**
- `\key` — замена текста по коду
- `%username` — замена имени пользователя

**Почему:**
- ✅ **Скорость ввода** — `\name ` быстрее, чем "Александр"
- ✅ **Часто используемые фразы** — быстрые пресеты для greetings и т.д.
- ✅ **Имена пользователей** — быстрое упоминание пользователей в чате
- ✅ **Мгновенная замена** — замена происходит после пробела

## 10. WebView Server for OBS

**Решение:** Добавить HTTP/Server-Sent Events сервер для интеграции с OBS Studio.

**Почему:**
- ✅ **Стриминг** — можно показывать текст в OBS
- ✅ **Real-time** — SSE для мгновенных обновлений
- ✅ **Кастомизация** — можно настроить HTML/CSS под свой дизайн
- ✅ **Удалённый доступ** — можно использовать с другого компьютера

**Технология:**
- **Server-Sent Events (SSE)** вместо WebSocket
- SSE проще для однонаправленной трансляции (server → client)
- Автоматический reconnect при разрыве
- Меньше overhead чем WebSocket

**Альтернативы:**
- ❌ **WebSocket** — избыточен для однонаправленного потока
- ❌ **OBS Plugin** — сложнее в разработке и установке
- ❌ **Text file watching** — не real-time
- ❌ **External service** — зависит от интернета

## 14. F6 Hotkey Mode

**Решение:** Добавить режим F6, где Enter не закрывает floating window.

**Почему:**
- ✅ **Последовательные сообщения** — можно отправить несколько сообщений без закрытия
- ✅ **Удобство** — не нужно постоянно открывать окно
- ✅ **Toggle** — можно включить/выключить по необходимости

**Поведение:**
- F6 выключен: Enter → отправить текст + закрыть окно
- F6 включен: Enter → отправить текст + окно остаётся открытым

## 15. File Storage Strategy

**Решение:** Использовать разные пути для разных типов данных.

**Пути:**
- `%USERPROFILE%\.config\tts-app\settings.json` — основные настройки
- `%APPDATA%\ttsbard\replacements.txt` — пресеты
- `%APPDATA%\ttsbard\usernames.txt` — имена пользователей
- `%APPDATA%\ttsbard\soundpanel_bindings.json` — привязки звуковой панели
- `%APPDATA%\ttsbard\soundpanel\` — аудиофайлы
- `%APPDATA%\ttsbard\webview\` — WebView HTML/CSS шаблоны

**Почему:**
- ✅ **Стандартные пути** — соответствует Windows conventions
- ✅ **Изоляция** — разные типы данных в разных местах
- ✅ **Легкий бэкап** — можно скопировать только нужное
- ✅ **Переносимость** — легко перенести на другой компьютер

**Альтернативы:**
- ❌ **Program Files** — требует прав администратора
- ❌ **Реестр Windows** — сложнее для бэкапа
- ❌ **База данных** — избыточно для этого случая

## 16. Error Handling Strategy

**Решение:** Использовать `Result<T, String>` для всех операций.

**Почему:**
- ✅ **Простота** — String достаточно для пользовательских ошибок
- ✅ **Пользовательские сообщения** — понятные ошибки на русском/английском
- ✅ **Events для frontend** — отправка ошибок через события

**Паттерн:**
```rust
pub type AppResult<T> = Result<T, String>;

// Emit error event
tx.send(AppEvent::TtsError("API key invalid".to_string()))?;
```

**Альтернативы:**

## 21. AI Text Correction (OpenAI + Z.ai)

**Решение:** Реализовать модуль AI текстовой коррекции с поддержкой нескольких провайдеров.

**Провайдеры:**
- **OpenAI (GPT-4o-mini)** — высокое качество, быстрая коррекция
- **Z.ai (GLM-4.5)** — альтернативный провайдер для резервирования

**Почему:**
- ✅ **Trait-based design** — `AiClient` trait позволяет легко добавлять новые провайдеры
- ✅ **Единый интерфейс** — все провайдеры реализуют `async fn correct(&self, text: &str, prompt: &str) -> Result<String, AiError>`
- ✅ **Гибкость** — пользователь может переключаться между провайдерами
- ✅ **Резервирование** — если один провайдер недоступен, можно использовать другой
- ✅ **Кастомные промпты** — пользователь может настроить поведение коррекции

**Альтернативы:**
- ❌ **Один провайдер** — зависимость от одного сервиса
- ❌ **Локальная модель** — требует больше ресурсов и медленнее
- ❌ **Встроенная грамматика** — недостаточно качественная коррекция

**Паттерн:**
```rust
#[async_trait]
pub trait AiClient: Send + Sync {
    async fn correct(&self, text: &str, prompt: &str) -> Result<String, AiError>;
}

pub enum AiProvider {
    OpenAi(OpenAiClient),
    ZAi(ZAiClient),
}
```

## 23. Customizable Hotkeys

**Решение:** Заменить hardcoded hotkeys на систему записи горячих клавиш через UI.

**Почему:**
- ✅ **Гибкость** — пользователь может назначить любые комбинации

- ✅ **Конфликты** — автоматическая проверка на конфликты между hotkeys
- ✅ **Persistence** — настройки сохраняются в config/hotkeys.rs

pub struct Hotkey {

pub struct HotkeySettings {

- ❌ **Ручной ввод текста** — неудобно и ошибкоопасно
- ❌ **Внешние конфигурационные файлы** — сложнее для обычных пользователей

## 24. Theme System (Dark/Light)

**Решение:** Реализовать систему тем с CSS переменными для dark/light режимов.

**Почему:**
- ✅ **CSS Variables** — единственный источник истины для цветов
- ✅ **RGB значения** — все цвета определены как `--rgb-*` для easy manipulation
- ✅ **Theme switching** — `[data-theme="light"]` атрибут для переключения
- ✅ **Semantic colors** — `--color-success`, `--color-warning`, `--color-danger`
- ✅ **Переиспользование** — переменные используются везде (buttons, cards, inputs)

**Структура:**
```css
:root {
  --rgb-accent: 29, 140, 255;
  --rgb-success: 74, 222, 128;
  --color-accent: rgb(var(--rgb-accent));
  --color-bg: rgb(var(--rgb-bg));
}

[data-theme="light"] {
  --rgb-bg: 250, 252, 255;
  --rgb-text: 15, 23, 42;
}
```

**Альтернативы:**
- ❌ **Hardcoded colors** — невозможно переключать темы
- ❌ **Separate CSS files** — дублирование кода
- ❌ **CSS-in-JS** — overhead для desktop приложения

## 25. Unified Proxy Settings

**Решение:** Единая настройка прокси для всех провайдеров (TTS, AI, Twitch, Telegram).

**Почему:**
- ✅ **Простота** — одна настройка вместо отдельных для каждого сервиса
- ✅ **Consistency** — всё работает через один прокси
- ✅ **Меньше конфигурации** — пользователь настраивает прокси один раз
- ✅ **Типы прокси** — поддержка SOCKS5, SOCKS4, HTTP
- ✅ **Per-provider toggle** — можно включить/выключить прокси для каждого провайдера

**Реализация:**
```rust
pub struct NetworkSettings {
    pub proxy_url: Option<String>,  // Единый прокси
    pub proxy_type: ProxyType,       // Socks5 | Socks4 | Http
    pub openai_use_proxy: bool,      // Тоггл для OpenAI
    pub fish_use_proxy: bool,        // Тоггл для Fish Audio
    // ...
}
```

**Альтернативы:**
- ❌ **Отдельные прокси для каждого** — сложно настраивать
- ❌ **Только системный прокси** — меньше контроля
- ❌ **Без прокси** — невозможно использовать из регионов с ограничениями

## 26. Editor Modes (Quick vs AI)

**Решение:** Реализовать два режима редактора для разных сценариев использования.

**Quick Editor Mode:**
- Enter → запускает TTS в фоне, не дожидаясь завершения
- Текст очищается сразу после запуска
- Окно закрывается мгновенно
- Идеально для быстрого ввода

**AI Editor Mode:**
- Enter → запускает AI коррекцию текста
- Кнопка для ручной коррекции (Sparkles icon)
- Ожидание результата коррекции
- Кнопка отправки после коррекции

**Почему:**
- ✅ **Сценарии использования** — разные режимы для разных задач
- ✅ **Quick mode** — для быстрого ввода без исправлений
- ✅ **AI mode** — для качественной коррекции перед отправкой
- ✅ **Гибкость** — можно переключаться между режимами
- ✅ **Settings** — оба режима настраиваются independently

**Альтернативы:**
- ❌ **Один режим** — неудобно для разных сценариев
- ❌ **Всегда с коррекцией** — медленно для быстрых сообщений
- ❌ **Никогда без коррекции** — качество страдает

## 27. Logging System

**Решение:** Реализовать систему логирования с настраиваемыми уровнями и per-module логами.

**Почему:**
- ✅ **Configurable levels** — TRACE, DEBUG, INFO, WARN, ERROR
- ✅ **Per-module** — можно настроить уровень логов для каждого модуля отдельно
- ✅ **File logging** — логи сохраняются в файл для диагностики
- ✅ **Development vs Production** — разные уровни для dev и prod
- ✅ **Structured logging** — tracing crate для structured logs

**Пример конфигурации:**
```toml
[logging]
level = "INFO"  # Глобальный уровень
modules = [
    { name = "tts::openai", level = "DEBUG" },
    { name = "ai", level = "TRACE" },
    { name = "twitch", level = "WARN" },
]
```

**Альтернативы:**
- ❌ **println!** — невозможно контролировать
- ❌ **Один уровень** — слишком много или мало логов
- ❌ **External logging service** — избыточно для desktop app
