# BUG-05: Вход в Telegram не завершается для аккаунтов с двухфакторной аутентификацией

- **Дата:** 2026-07-17
- **Компонент:** Telegram user authorization / Silero TTS
- **Статус:** исправлен
- **Приоритет:** высокий — блокирует подключение аккаунтов с включённым 2FA

## Симптом

Если у Telegram-аккаунта включён пароль двухфакторной аутентификации, после ввода корректного кода из Telegram приложение показывает ошибку:

> Требуется двухфакторная аутентификация. Эта функция пока не реализована.

Пользователю не предлагается поле для пароля 2FA, поэтому авторизация не переходит в состояние `connected`, сессия не сохраняется, а функции Silero TTS остаются недоступны.

## Шаги воспроизведения

1. Использовать Telegram-аккаунт с включённым дополнительным паролем 2FA.
2. Открыть окно авторизации Telegram.
3. Ввести API ID, API Hash и номер телефона.
4. Запросить код и ввести корректный код Telegram.
5. Наблюдать ошибку вместо запроса пароля 2FA.

## Ожидаемое поведение

После успешной проверки кода UI должен перейти в отдельное состояние `password_required` и запросить пароль 2FA. После ввода правильного пароля авторизация должна завершиться, пользователь должен увидеть состояние `connected`, а сессия — быть сохранена.

Для неправильного пароля нужно оставаться на шаге 2FA и показывать понятную ошибку без потери login-контекста. Для отмены или повторного входа должен существовать безопасный путь начать авторизацию заново.

## Фактическая причина

В локальной копии `D:\RustProjects\grammers` поддержка этого сценария уже заложена:

- `grammers-client/src/client/auth.rs:sign_in` преобразует RPC-ошибку `SESSION_PASSWORD_NEEDED` в `SignInError::PasswordRequired(PasswordToken)`;
- `grammers-client/src/client/auth.rs:check_password` получает параметры `account.getPassword`, вычисляет SRP-проверку и вызывает `auth.checkPassword`;
- при неверном пароле библиотека возвращает `SignInError::InvalidPassword`, не раскрывая пароль.

В приложении проблема находится в адаптере `src-tauri/src/telegram/client.rs`:

- `sign_in` вызывает `client.sign_in(&token, code)`;
- ветка `Err(SignInError::PasswordRequired(_))` возвращает строку `Требуется двухфакторная аутентификация. Эта функция пока не реализована.`;
- `login_token` возвращается в mutex, но отдельной команды для передачи пароля нет.

Frontend также не имеет соответствующего шага:

- `src/composables/useTelegramAuth.ts` после `telegram_sign_in` ожидает только успех или финальную ошибку;
- `src/components/TelegramAuthModal.vue` содержит поля API ID/API Hash/телефон/код, но не поле пароля 2FA и не состояние `password_required`;
- в `src-tauri/src/commands/telegram.rs` зарегистрированы `telegram_request_code` и `telegram_sign_in`, но отсутствует команда для 2FA-пароля.

Таким образом, сообщение об ошибке технически правдиво для текущего адаптера, но вводит в заблуждение: требуемая криптографическая реализация уже есть в используемой библиотеке.

## Результаты ресерча

Официальная документация Telegram описывает тот же переход: `auth.signIn` может вернуть `SESSION_PASSWORD_NEEDED`, если у аккаунта включён 2FA; затем клиент должен получить параметры через `account.getPassword`, сформировать `InputCheckPasswordSRP` и вызвать `auth.checkPassword`.

- [Telegram: User Authorization](https://core.telegram.org/api/auth)
- [Telegram: Two-factor authentication / SRP](https://core.telegram.org/api/srp)
- [Telegram: account.getPassword](https://core.telegram.org/method/account.getPassword)
- [Telegram: auth.checkPassword](https://core.telegram.org/method/auth.checkPassword)

Это не означает, что пароль должен отправляться в Telegram API в открытом виде: приложение передаёт его в `grammers`, а библиотека выполняет SRP/KDF-проверку локально.

## Область исправления

Нужно согласованно изменить backend и frontend:

1. Сохранить `PasswordToken` или эквивалентный pending-auth контекст после `PasswordRequired` и вернуть типизированный результат `password_required`, а не обычную строку ошибки.
2. Добавить Tauri-команду для отправки пароля 2FA, вызывающую `grammers` `check_password`.
3. Добавить в composable и модальное окно состояние/форму `password_required`.
4. Обработать `InvalidPassword`, истёкший/сброшенный auth-контекст, повторный ввод и отмену.
5. Не логировать код, пароль, `PasswordToken`, SRP-параметры и содержимое ошибок, если они могут содержать секреты.
6. Проверить обычный вход без 2FA, вход с 2FA, неверный пароль, повторный запуск и автосстановление сохранённой сессии.

## Критерии приёмки

- Аккаунт без 2FA продолжает входить по прежнему сценарию.
- Аккаунт с 2FA после корректного кода получает форму пароля, а не ошибку «не реализовано».
- Корректный пароль переводит UI в `connected` и сохраняет сессию.
- Неверный пароль не сбрасывает шаг 2FA и не раскрывает пароль в UI/логах.
- После отмены или истечения pending-auth контекста пользователь может начать вход заново.
- `cargo check` и `npx vue-tsc --noEmit` проходят.

## Реализованное исправление

Добавлен полноценный backend/frontend flow 2FA:

- `grammers` `PasswordToken` хранится только в backend;
- добавлены состояния `PasswordRequired` / `password_required`;
- добавлена команда `telegram_check_password` с использованием `check_password` и SRP;
- в UI добавлено поле пароля 2FA с повторной попыткой после неверного пароля;
- пароль очищается после попытки и закрытия модального окна;
- секреты не выводятся в логи.

Проверки: `cargo check`, `npx vue-tsc --noEmit`, `git diff --check` — успешно.
