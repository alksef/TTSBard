# ROADMAP-014 — Наборы звуков и быстрое редактирование

**Дата:** 2026-07-08
**Статус:** research / ✅ РЕШЕНО (на уровне stage; план — отдельный файл)
**Решение:** **гибрид** — (1) несколько **наборов (Sets)** звуков, каждый со своими привязками A-Z,
переключаются; (2) **быстрое редактирование прямо в окне вызова** (drag-drop файла, добавить
звук на лету). Основное окно (SoundPanelTab) остаётся для тщательной настройки; окно вызова
получает режим быстрого редактирования, синхронизированный с активным набором.
**Связано:** `SoundPanelApp.vue` (плавающее окно), `SoundPanelTab.vue` (основное окно),
`soundpanel/state.rs`, `soundpanel/storage.rs`, `soundpanel/bindings.rs`, `commands/soundpanel_window.rs`.

## Цель (запрос пользователя)
- Сейчас саундпанель — **один плоский набор из 26 привязок A-Z** (`HashMap<char, SoundBinding>`,
  `state.rs:37`). Для стримера этого мало: разные контексты (обычные звуки / мемы / реакции на игру)
  хотелось бы держать отдельными наборами и быстро переключать.
- Вторая идея: сейчас **редактирование только в основном окне** (`SoundPanelTab.vue`). Хочется
  править прямо в окне вызова — добавить звук на лету, не уходя в основное окно.
- Вопрос: «может отказаться от задания списка в основном окне и прямо в форме вызова добавить —
  оценить какой вариант лучше». → Решение: **не отказываться**, а сделать **гибрид** (оба места
  редактируют единое хранилище; основное окно — для большой настройки, окно вызова — для быстрых
  правок в контексте активного набора).

## Контекст кода (что уже есть)

### Бэкенд
- `SoundPanelState` (`state.rs:31-59`) — состояние: `bindings: Arc<Mutex<HashMap<char, SoundBinding>>>`,
  appearance (opacity/bg/clickthrough), intercept-настройки. **Один набор.**
- `SoundBinding` (`state.rs:18-28`): `{key: char, description, filename, original_path?}`.
- `soundpanel/storage.rs`:
  - `load_bindings`/`save_bindings` → `%APPDATA%\ttsbard\soundpanel_bindings.json` (голый `Vec<SoundBinding>`).
  - `copy_sound_file` → копирует аудио в `%APPDATA%\ttsbard\soundpanel\`, `delete_sound_file`.
  - Appearance хранится **в windows.json** через `WindowsManager` (storage.rs:166-190), НЕ в bindings.json.
- `soundpanel/bindings.rs` — команды: `sp_get_bindings`, `sp_add_binding`, `sp_remove_binding`,
  `sp_test_sound`, `sp_*_floating_*`, `sp_play_binding`.
- **⚠️ Двойной источник привязок:** `SoundPanelTab.vue:204` читает `appSettings.soundpanel_bindings`
  (unified config `settings.json`), а `SoundPanelApp.vue:47` (`sp_get_bindings`) читает
  `SoundPanelState` (`soundpanel_bindings.json`). Они синхронизируются через событие
  `soundpanel-bindings-changed`. Это существующий технический долг — наборам (Sets) надо один
  источник правды.

### Фронт — основное окно
- `SoundPanelTab.vue`: таблица привязок (key/description/file/remove), диалог добавления
  (key select A-Z, description, file browse, test), секция appearance (color/opacity/clickthrough).

### Фронт — окно вызова (плавающее)
- `SoundPanelApp.vue` (`src-soundpanel/`): overlay-окно. `onKeydown` (97-116) ловит A-Z →
  `sp_play_binding`. Показывает `bindings-grid` (kbd + description). Appearance (transparency/color)
  + clickthrough + close. **Только воспроизведение, без редактирования.**

### Наборы сейчас
**Нет.** Один набор. Концепта Sets не существует.

## Решение — гибрид (две части)

### Часть 1. Наборы (Sets) — несколько групп привязок

#### Модель данных (бэкенд)
Вместо одного `Vec<SoundBinding>` → **наборы**:
```jsonc
// soundpanel_bindings.json (новый формат)
{
  "active_set_id": "uuid",
  "sets": [
    {
      "id": "uuid",
      "name": "Обычные",
      "bindings": [ { "key":"A","description":"...","filename":"...","original_path":null }, ... ]
    },
    {
      "id": "uuid",
      "name": "Мемы",
      "bindings": [ ... ]
    }
  ]
}
```
- `#[serde(default)]` на полях + миграция: если файл — старый `Vec<SoundBinding>`, обернуть в
  один дефолтный Set «Основной» (backwards-compat при загрузке, как phrase_history read_or_default).
- `SoundPanelState`: заменить `bindings: HashMap<char,SoundBinding>` на `sets: Vec<SoundSet>` +
  `active_set_id`. Геттер `get_all_bindings()` → возвращает bindings **активного** набора (чтобы
  `sp_play_binding` и плавающее окно работали без изменений по интерфейсу).
- **Новый source-of-truth:** убрать дублирование с `appSettings.soundpanel_bindings` — Sets живут
  только в `soundpanel_bindings.json` (+ `SoundPanelState`). `SoundPanelTab` читает их через
  команды `sp_*`, а не через unified settings. (Устраняет существующий двойной источник.)

#### Команды (бэкенд)
Расширить `bindings.rs`:
- `sp_get_sets() -> SoundSets` — все наборы (для основного окна).
- `sp_get_active_set() -> SoundSet` — активный (для плавающего окна: что играть).
- `sp_set_active_set(id)` — переключить активный набор + emit `soundpanel-active-set-changed`.
- `sp_add_set(name) -> SoundSet`, `sp_rename_set(id, name)`, `sp_remove_set(id)`.
- Существующие `sp_add_binding`/`sp_remove_binding` — добавить параметр `set_id` (по умолчанию
  активный), чтобы работало и из основного окна (с указанием набора), и из окна вызова (активный).
- `sp_play_binding(key)` — без изменений (играет из активного набора).

#### Фронт — основное окно (`SoundPanelTab.vue`)
- Сверху таблицы — **строка наборов** (как EditorTabs): `[Обычные] [Мемы] [АПЛ] +`.
  Активный набор подсвечен; переключение → `sp_set_active_set`. `+` → создать набор (prompt имени).
- Таблица привязок показывает bindings **активного набора**.
- Drag для переименования/удаления набора (контекстное меню или кнопки).
- Удаление набора → confirm (как `removeBinding`).

#### Фронт — окно вызова (`SoundPanelApp.vue`)
- Селектор активного набора (компактный dropdown/стрелки ◀ ▶ в title-bar) — переключение на лету,
  без открытия основного окна. Стример во время трансляции меняет набор звуков в один клик.
- `bindings-grid` показывает bindings активного набора.

### Часть 2. Быстрое редактирование в окне вызова

#### Режим редактирования в плавающем окне
- Кнопка-тумблер `[✎]` в title-bar (рядом с clickthrough/transparency): режим просмотра ↔ режим
  редактирования.
- **В режиме редактирования** grid показывает на каждой привязке кнопку замены файла + пустые
  слоты «+ добавить» для свободных клавиш A-Z.
- **Drag-and-drop файла** прямо на слот/клавишу → копирует файл (`copy_sound_file`) + создаёт
  привязку в активном наборе. Tauri drop-event (`onDragDrop` / `tauri://drag-drop`) — нужно
  включить в webview окна саундпанели.
- **Быстрый выбор файла** (клик на пустой слот) → `open()` диалог (как `browseFile` в SoundPanelTab).
- Все правки идут в **активный набор** через те же `sp_add_binding`/`sp_remove_binding` (с `set_id`
  активного). Persistence и событие `soundpanel-bindings-changed` переиспользуются.
- После правки — мгновенный возврат в режим просмотра (звук готов к проигрыванию).

#### Почему не «убрать совсем редактирование из основного окна»
- Основное окно лучше для **первичной настройки** множества звуков (таблица, описание, обзор
  файлов, тест) — простор, не спешка.
- Окно вызова лучше для **быстрых правок в моменте** (добавил один звук по ходу стрима).
- Гибрид = оба кейса покрыты; единое хранилище → нет конфликта.

## Варианты (что рассматривали)

### Для наборов
- **A. Несколько Sets с переключением** ⭐ выбран. Структурирует большое кол-во звуков; знакомый
  UX (вкладки). Переключение на лету в окне вызова — ключевой кейс для стримера.
- **B. Один набор + категории/теги на каждом звуке + фильтр.** Отвергнуто: 26 клавиш — мало, фильтр
  не даст одновременно видеть «свой звук на A» в двух контекстах; переключение наборов проще.
- **C. Бесконечный список с paging.** Отвергнуто: теряется мгновенность (стримеру нужен 1 клик).

### Для редактирования
- **1. Гибрид (основное + окно вызова)** ⭐ выбран. Покрывает оба кейса.
- **2. Только окно вызова (убрать SoundPanelTab).** Отвергнуто: теряется простор для настройки.
- **3. Только основное окно (сейчас).** Отвергнуто: нет быстрых правок в моменте.

## Риски / нюансы (для будущего плана)
1. **Миграция старого `soundpanel_bindings.json`** (голый `Vec`) → при загрузке обернуть в один
   Set «Основной». Бесконтактно для существующих пользователей. Зафиксировать в плане явно.
2. **Двойной источник привязок (существующий долг)** — `appSettings.soundpanel_bindings` vs
   `SoundPanelState`/`soundpanel_bindings.json`. Sets должны жить **в одном месте**
   (`soundpanel_bindings.json` + `SoundPanelState`). `SoundPanelTab` перевести на команды `sp_*`,
   убрать чтение из `appSettings.soundpanel_bindings` (или оставить как deprecated-дубликат на
   время миграции). Это отдельный шаг в плане — не смешивать с UI Sets.
3. **Drag-and-drop в Tauri webview** — нужно разрешить drop в окне саундпанели (по умолчанию
   webview может блокировать). Проверить `tauri://drag-drop` событие / `onDragDropTask` в Tauri v2.
   Spike: убедиться, что путь к перетянутому файлу доступен для `copy_sound_file`.
4. **clickthrough-конфликт** — в режиме clickthrough плавающее окно пропускает клики; режим
  редактирования требует кликов. ⇒ **режим редактирования автоматически выключает clickthrough**
   (или блокирует его включение пока активен). Зафиксировать.
5. **События синхронизации** — правка в окне вызова должна отразиться в основном окне (если открыто)
   и наоборот. Переиспользовать существующее `soundpanel-bindings-changed` (+ новое
   `soundpanel-active-set-changed`). Оба окна слушают.
6. **Производительность switching sets** — наборы маленькие (≤26), переключение = swap ссылок в
   состоянии + emit. Мгновенно.
7. **Удаление набора с файлами** — удалять ли аудиофайлы из `%APPDATA%\ttsbard\soundpanel\` при
   удалении набора? Решение: удалять только если файл не используется в других наборах (проверка
   по filename). Иначе оставить (мусор, но безопаснее). Зафиксировать.
8. **Имя набора по умолчанию / локализация** — «Основной» / «Набор 2». Как EditorTabs «Текст N».
9. **`active_set_id` persistence** — сохранять в `soundpanel_bindings.json` (как active_id в tabs).
   При повреждении — первый набор.

## Оценка трудозатрат
| Часть | Объём |
|---|---|
| Бэкенд: модель Sets + миграция старого формата + команды + единый source-of-truth | средний |
| Бэкенд: устранение дубля с appSettings.soundpanel_bindings | малый-средний |
| Фронт основное окно: строка наборов + переключение + CRUD наборов | средний |
| Фронт окно вызова: селектор набора + режим редактирования + drag-drop | средний-крупный |
| Spike: drag-drop файла в Tauri webview | малый (блокирующий для части 2) |

Уровень — **крупный** (бэкенд-модель + два фронта). Разбить на **два плана**:
- **14-A:** Наборы (Sets) — модель + оба UI показывают/переключают наборы (без inline-редактирования).
- **14-B:** Быстрое редактирование в окне вызова (drag-drop + режим редактирования) — после 14-A.

Реализация — через DeepSeek; Claude — план/ревью.

## KEY_DECISIONS
- **Гибрид:** наборы (Sets) в основном окне для настройки + быстрое редактирование в окне вызова.
- **Модель:** `soundpanel_bindings.json` → `{active_set_id, sets: [{id,name,bindings}]}`, миграция
  старого `Vec` в один дефолтный Set.
- **Единый source-of-truth:** Sets живут в `soundpanel_bindings.json` + `SoundPanelState`;
  устранить дубль с `appSettings.soundpanel_bindings` (существующий долг).
- **Окно вызова:** селектор активного набора (1 клик) + режим редактирования (drag-drop файла,
  добавить на лету) с автовыключением clickthrough.
- **Разбить на планы 14-A (Sets) и 14-B (inline-редактирование)** — после spike drag-drop.
- **Тема через CSS-vars** (light/dark), как во всём приложении; плавающее окно — своя overlay-тема.
