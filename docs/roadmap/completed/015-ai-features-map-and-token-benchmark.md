# ROADMAP-015 — Карта AI-функций и оценка токенов

**Дата:** 2026-07-08
**Статус:** `completed` — исследование завершено, выбранные AI-функции реализованы
**Контекст:** запрос пользователя — «рассмотреть чем может реально помочь AI, нужен ли он,
насколько быстро тратятся токены для автодополнения». Также — переописание проекта
(см. ROADMAP-016) меняет
**целевую аудиторию**: теперь это (а) люди с ограничениями речи (печатают сами) и (б) стримеры
(озвучка чата). AI-ценность для них **разная**.
**Связано:** `commands/ai.rs` (`correct_text`, `get_ai_completion`), `ai/mod.rs`, `02`/`03`
(autocomplete), `07`/`08` (spellcheck), spike 80 (hunspell).

## Что уже есть (зачатки AI)
- **AI-коррекция текста** — `correct_text` (`commands/ai.rs:118-149`): весь текст → провайдер
  (OpenAI/Z.ai) → исправленный текст. Вызывается кнопкой «AI» в InputPanel.
- **AI-продолжение фразы** — `get_ai_completion` (`commands/ai.rs:205-238`): контекст (текст до
  курсора) → продолжение 1-5 слов. Подается в autocomplete-слой редактора (план 73). Включается
  настройкой `editor.ai_completion` + наличие ключа.
- **Провайдеры:** OpenAI (async-openai) и Z.ai (Anthropic-совместимый). Кэш клиента в `AppState`.
- **Spellcheck** — офлайн hunspell (`spellcheck.rs`, spike 80 пройден, suggest 4/4). Онлайн-проверки
  орфографии нет. **AI-орфографии как таковой нет** — есть AI-коррекция (весь текст одним заходом).

## Часть 1. Токен-бенчмарк автодополнения (главный вопрос «сколько стоит»)

### Модель запроса `get_ai_completion`
- **System prompt** (жёстко в `commands/ai.rs:228-230`): ~60 токенов (фиксированный русский текст
  про «помощник для завершения текста... максимум 5 слов»).
- **User context** (текст до курсора): переменный. Типичная фраза для ТТС — 1-3 предложения ≈
  **20-80 токенов**.
- **Completion**: «1-2 слова или короткая фраза, макс 5 слов» ≈ **5-15 токенов output**.
- Итого за запрос: **~80-140 input + ~5-15 output токенов**.

### Частота вызовов
Зависит от триггера (сейчас — из autocomplete-слоя плана 73, дебаунс):
- **Пессимистично (без debounce/caching):** на каждое нажатие/паузу набора. Активный набор
  предложения = ~20-60 пауз/мин при неторопливом наборе. → **до ~60 запросов/мин**.
- **Реалистично (debounce 300-500мс + только после паузы):** ~5-15 запросов/мин активного набора.
- **Час активного использования (стример 4ч стрим, печатающий пользователь 2ч):**
  ~300-900 запросов/час (реалистично), до ~3600 (пессимистично).

### Стоимость на DeepSeek (цены июль 2026, V4 Flash / deepseek-chat)

| Метрика | Cache miss | Cache hit |
|---|---|---|
| Input за 1М токенов | **$0.14** | **$0.0028** (−98%) |
| Output за 1М токенов | **$0.28** | — |

> System prompt одинаковый между запросами → **prompt caching**: после первого запроса system+
> префикс контекста кэшируется, платим cache-hit rate по input.

### Расчёт (на 1000 запросов автодополнения)
- Input: 1000 × 100 токенов = 100К токенов. Cache hit (≈90% при стабильном system prompt):
  90К × $0.0028/1М + 10К × $0.14/1М = **$0.000252 + $0.0014 ≈ $0.00165**.
- Output: 1000 × 10 токенов = 10К × $0.28/1М = **$0.0028**.
- **Итого ≈ $0.0044 за 1000 запросов.** → **~$0.004 за 1000 автодополнений.**

### Суточные/часные сценарии
| Сценарий | Запросов | Стоимость (DeepSeek) |
|---|---|---|
| 1 час активного набора (реалистично ~600 запросов) | 600 | **~$0.003** |
| День активного использования (~3000 запросов) | 3000 | **~$0.013** |
| Месяц тяжёлого использования (~50К запросов) | 50 000 | **~$0.22** |
| Пессимистичный месяц (~200К запросов, без debounce) | 200 000 | **~$0.88** |

**Вывод: автодополнение на DeepSeek практически бесплатное** — центы в месяц даже при пессимизме.
Главная стоимость — **не деньги, а задержка** (латентность API: 0.3-2с на запрос) и **надёжность
сети**. Деньги перестают быть аргументом против при DeepSeek-цене; аргумент — UX (не тормозит ли
ввод) и приватность (текст уходит на API).

### Сравнение с OpenAI/Z.ai (текущие провайдеры)
- OpenAI (gpt-4o-mini и ниже): дороже в **10-50×** ($1-3/1М input против $0.14). При тех же 50К
  запросов/мес — ~$2-5/мес. Всё ещё дёшево, но заметнее.
- **Рекомендация:** добавить DeepSeek как провайдер (`AiProviderType::DeepSeek`, OpenAI-совместимый
  API `https://api.deepseek.com`) — дешёвый дефолт для автодополнения; OpenAI/Z.ai оставить для
  коррекции (где качество важнее). Это также решает «зачатки AI» → полноценный дешёвый путь.

## Часть 2. Карта AI-фичей (что реально полезно)

Оценка по трём осям: **Ценность** (ускоряет ли набор / улучшает озвучку для целевой аудитории),
**Сложность**, **Токены/стоимость**. Аудитория: (А) ограничения речи — печатают сами; (Б) стримеры —
озвучка чата.

| AI-фича | Что делает | Ценность | Сложность | Токены | Статус |
|---|---|---|---|---|---|
| **AI-коррекция текста** | исправить опечатки/пунктуацию во всём тексте одним заходом | ⭐⭐⭐ А/Б | малая | средние (весь текст) | ✅ есть (`correct_text`) |
| **AI-продолжение фразы** (autocomplete) | дописать 1-5 слов по контексту | ⭐⭐⭐ А, ⭐⭐ Б | малая | **минимальные** (см. бенчмарк) | ✅ есть (`get_ai_completion`) |
| **AI-орфография (inline)** | подсветка ошибок + варианты, контекстно (не по словарю) | ⭐⭐ А/Б | средняя | высокие (на паузах) | ❌ нет — см. часть 3 |
| **Экспрессивная разметка** | AI добавляет SSML/паузы/ударения для выразительности голоса | ⭐⭐ Б, ⭐ А | средняя-крупная | средние | ❌ нет — см. ниже |
| **Перефразирование** | «скажи короче/вежливее/эмоциональнее» | ⭐⭐ А/Б | малая | средние | ❌ нет |
| **Контекстные сниппеты/макросы** | AI предлагает готовые фразы по контексту (приветствие, прощание) | ⭐ А/Б | средняя | средние | ❌ нет (есть препроцессор `\ключ`) |
| **Авто-озвучка входящего чата** | AI суммирует/озвучивает сообщения Twitch/донаты | ⭐⭐⭐ Б (для стримера) | крупная | высокие | ❌ нет (есть Twitch panel без AI) |

### Топ-3 к реализации (по соотношению ценность/сложность)
1. **AI-продолжение (уже есть) + DeepSeek-провайдер** — дёшево, готово, добить только провайдером.
2. **AI-орфография (inline)** — для аудитории А критично (печатают сами, ошибки часты). См. часть 3.
3. **Перефразирование** — дёшево (обёртка над `correct` с другим prompt), заметная польза.

### Что НЕ делать / отложить
- **Авто-озвучка чата** — крупная фича, отдельная линия (Twitch AI-сводка). Не «ускоряет набор»,
  это другой продукт-направление. Зафиксировать как idea, не сейчас.
- **Контекстные сниппеты** — дублирует существующий препроцессор (`\ключ`, `%username`). Сначала
  оценить, не хватает ли текущих макросов, прежде чем звать AI.

## Часть 3. AI-орфография vs hunspell (вместо онлайн LanguageTool)

Stage 07/08 зафиксировал: офлайн hunspell (✅ spike 80 пройден, `suggest` 4/4, 5µs/слово) + онлайн
LanguageTool/Yandex (❌ не реализован). Новый вопрос: **может ли AI заменить онлайн-орфографию?**

### Сравнение (по исследованию)

| Критерий | Hunspell (офлайн, есть) | AI-орфография (LLM) | Онлайн LanguageTool |
|---|---|---|---|
| **Контекст (омонимы: там/тут)** | ❌ только слово | ✅ понимает контекст | ✅ частично |
| **Сленг / новые слова / имена** | ❌ нужен ручной словарь | ✅ из тренинга | ⚠️ средне |
| **Скорость** | 5µs/слово, мгновенно | 0.3-2с/запрос (латентность) | сеть, медленно |
| **Стоимость** | $0 (офлайн) | токены на каждый запрос | квоты/бесплатно с лимитами |
| **Офлайн-работа** | ✅ да | ❌ нужен интернет | ❌ нужен интернет |
| **Приватность** | ✅ текст не уходит | ❌ уходит на API | ❌ уходит на API |
| **Грамматика (не только орфография)** | ❌ только слова | ✅ грамматика+стиль | ✅ грамматика |

### Вывод и рекомендация — **гибрид**, не «AI вместо всего**
- **Hunspell остаётся базой** (офлайн, мгновенно, бесплатно, приватно) — для постоянной подсветки
  опечаток при наборе. Это уже сделано (план 81).
- **AI-орфография — точечное усиление по запросу**, не непрерывная проверка:
  - Кейс: слово подчёркнуто hunspell, но это **имя/сленг/термин** (false positive) → пользователь
    просит AI «проверить грамматику/контекст выделенного» → AI даёт контекстные варианты.
  - Или: «исправить всё» (уже есть как `correct_text`) = AI-орфография + грамматика одним заходом.
- **Почему НЕ непрерывная AI-проверка:** латентность (0.3-2с) убивает мгновенность набора; токены на
  каждую паузу; приватность (каждое слово уходит на API). Hunspell лучше для «живой» подсветки.
- **LanguageTool (онлайн) — не нужен**, если есть AI для точечных случаев. Зачем держать отдельный
  онлайн-API, если AI-провайдер (DeepSeek, копейки) покрывает и грамматику, и контекст? Отказ от
  LanguageTool упрощает стек (нет отдельной интеграции/квот). Это **меняет** вывод stage 07.

### Архитектура AI-орфографии (если делать)
- **НЕ новая непрерывная проверка.** Переиспользовать `correct_text`, но с **prompt для грамматики**
  и по **выделенному фрагменту** (а не всему тексту): «Проверь орфографию и грамматику, верни
  исправленный текст без пояснений».
- UI: пункт в EditorMenu (stage 07) «Проверить грамматику (AI)» по выделению. Backend — новая команда
  `ai_check_grammar(selection)` или параметр `mode` в `correct_text`.
- Покрывает «омонимы/сленг/имена» (где hunspell бессилен) — точечно, дёшево, по запросу.

## Решения (предварительные, для планов)
1. **Добавить DeepSeek как провайдер** (OpenAI-совместимый API) — дешёвый дефолт для автодополнения
   и точечных AI-вызовов. Решает «нужен ли AI» — да, при DeepSeek-цене деньги не аргумент.
2. **AI-орфография = гибрид:** hunspell (офлайн, база) + AI по запросу (грамматика/контекст
   выделенного). НЕ непрерывная AI-проверка. **LanguageTool не внедрять** (упрощает стек).
3. **Приоритет AI-фичей:** (1) продолжение+DeepSeek, (2) AI-грамматика по выделению, (3) перефразирование.
4. **Не сейчас:** авто-озвучка чата (отдельное направление), контекстные сниппеты (сначала оценить
   препроцессор).

## Риски / нюансы
1. **Латентность автодополнения** — даже дешёвый API тормозит ввод, если вызывать синхронно.
   Текущий слой 73 уже асинхронный + debounced; убедиться, что UI не блокируется. Это UX-риск, не
   ценовой.
2. **Prompt caching** — DeepSeek поддерживает cache-hit rate (−98% input). Чтобы он сработал,
   system prompt должен быть **стабильным** между запросами (сейчас жёстко в коде — хорошо).
   Контекст пользователя меняется → кэшируется только префикс system prompt. Зафиксировать.
3. **Приватность** — для аудитории А (медицински-смежный контекст возможен) явноwarn, что текст
   уходит на API (как с AI-ключами сейчас). Настройка вкл/выкл AI-фичей.
4. **DeepSeek API совместимость** — endpoint `https://api.deepseek.com/chat/completions`, модель
   `deepseek-chat` (deprecated с 2026-07-24 → миграция на V4 Flash). Проверить актуальную модель
   при реализации.
5. **Тестовый ключ** — использовать локальный тестовый ключ только для бенчмарка/теста; НЕ коммитить в код.
   Хранить в `.env`/вне репо (как другие ключи в settings.json, но тестовый — локально).

## Оценка трудозатрат
| Часть | Объём |
|---|---|
| DeepSeek-провайдер (`AiProviderType::DeepSeek`, client, settings UI) | средний |
| AI-грамматика по выделению (`correct_text` с mode/selection + EditorMenu пункт) | малый-средний |
| Перефразирование (prompt-вариант `correct`) | малый |
| Токен-бенчмарк-скрипт (замер реальных запросов через DeepSeek, опционально) | малый |

Реализация — через DeepSeek; Claude — план/ревью.

## KEY_DECISIONS
- **Деньги — не аргумент против AI** при DeepSeek-цене ($0.004/1000 запросов, центы/мес).
  Главные риски — латентность и приватность.
- **Добавить DeepSeek-провайдер** как дешёвый дефолт (OpenAI-совместимый).
- **AI-орфография = гибрид:** hunspell база + AI по запросу (грамматика/контекст). LanguageTool НЕ нужен.
- **Карта фичей:** продолжение (есть) → AI-грамматика по выделению → перефразирование. Авто-озвучка
  чата — отдельное направление (не сейчас).
- **Prompt caching** держать system prompt стабильным (−98% input на DeepSeek).
- **Приватность:** явный warn + on/off для AI-фичей (важно для аудитории А).
