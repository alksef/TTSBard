# ROADMAP-030 — Инвариант sample rate после DeepFilterNet

**Дата:** 2026-07-13  
**Статус:** stage завершён / вариант A реализован и проверен  
**Связано:** `src-tauri/src/audio/effects.rs`, [ROADMAP-025](./025-playback-pcm-pipeline.md), [ROADMAP-029](./029-dsp-audio-postprocessing.md), [ROADMAP-031](../active/031-resemble-enhance-research.md)

## 1. Проблема

DeepFilterNet работает на фиксированной частоте дискретизации модели — сейчас это 48 kHz. В `apply_enhance()` входной PCM при необходимости ресемплируется из исходной частоты в частоту модели:

```text
input sample rate → 48 kHz → DeepFilterNet
```

Однако обработанный сигнал не ресемплируется обратно. Затем `apply_effects()` безусловно записывает в `AudioPcm` значение `sample_rate = 48000`.

Само по себе это не обязательно искажает длительность: при входе 24 kHz количество samples примерно удваивается, а маркировка результата как 48 kHz сохраняет длительность. Проблема в том, что переход частоты скрыт внутри функции и не оформлен как явный контракт. Для других частот возникают следующие риски:

1. **Нарушение ожидаемого sample-rate контракта.** Для входов 16/22,05/24/44,1 kHz playback получает уже 48 kHz, хотя до enhancement частота была другой. Это может влиять на последующий stretch, DSP, буферизацию, тесты длительности и устройства, ожидающие исходный формат.
2. **Реальная ошибка длительности при рассинхронизации samples и metadata.** Если в будущем `apply_enhance()` начнёт возвращать исходную частоту, либо другой caller будет считать samples исходными, но сохранит текущую маркировку 48 kHz, длительность станет неверной. Сейчас это предотвращается ручным `sample_rate = 48000`, но такой инвариант не проверяется типами и тестами для разных частот.

Текущий тест `test_deep_filter_audio_fixture` использует только 48 kHz, поэтому проблему не обнаруживает.

## 2. Требуемый инвариант

Нужно выбрать и явно зафиксировать один из двух контрактов.

### Вариант A — нормализовать enhancement-выход в 48 kHz

```text
decode
  → original PCM
  → resample to 48 kHz
  → DeepFilterNet
  → stretch at 48 kHz
  → DSP at 48 kHz
  → playback at 48 kHz
```

Плюсы:

- проще всего для DeepFilterNet и realtime playback;
- не требуется обратный ресемплинг;
- единый формат для всех голосовых источников.

Минусы:

- включённый enhancement меняет формат результата даже для 16/24/44,1 kHz;
- поведение должно быть одинаково документировано для playback и export;
- при выключенном enhancement сохраняется старый sample rate, что создаёт два режима.

### Вариант B — сохранять исходную частоту результата

```text
decode
  → original PCM
  → resample to 48 kHz
  → DeepFilterNet
  → resample back to original sample rate
  → stretch at original rate
  → DSP at original rate
  → playback at original rate
```

Плюсы:

- `AudioPcm` сохраняет исходный публичный контракт;
- включение DeepFilterNet не меняет sample rate downstream;
- проще сравнивать результат enhancement с исходным аудио.

Минусы:

- два ресемплинга вместо одного;
- часть высокочастотной информации, обработанной на 48 kHz, будет снова отброшена при возврате к 16/24 kHz;
- если исходный TTS ограничен 16/24 kHz, enhancement не сможет сохранить достроенные высокие частоты.

## 3. Рекомендуемое решение

Для текущего playback-пайплайна предпочтителен **вариант A**: считать 48 kHz каноническим форматом после включения DeepFilterNet и явно оформить это как контракт.

Причины:

- DeepFilterNet уже является 48 kHz-моделью;
- последующий Signalsmith Stretch и native DSP могут работать с фактическим `sample_rate`, который уже передаётся в функцию;
- для speech playback единый 48 kHz обычно практичнее, чем сохранение низкочастотного источника;
- обратный ресемплинг обесценивает возможное улучшение полосы и особенно не нужен для чистого realtime TTS.

При этом enhancement не должен неожиданно менять sample rate в режиме экспорта, если экспортный формат требует сохранения оригинала. Для export следует позже добавить отдельную политику: `preserve_source_rate` или `output_rate`.

## 4. Что нужно проверить при реализации

- `apply_enhance()` возвращает samples с фактической частотой результата, а не скрывает её внутри `Vec<f32>`.
- `AudioPcm::new()` получает именно ту частоту, на которой находятся samples.
- длительность фразы до и после enhancement отличается не более чем на допустимую погрешность ресемплинга;
- количество каналов и interleaved layout сохраняются;
- stretch и DSP используют фактический sample rate после enhancement;
- короткие фразы, включая длину меньше одного hop DeepFilterNet, не получают неожиданного растяжения или обрезания;
- fallback при ошибке DeepFilterNet возвращает исходные samples и исходный sample rate.

## 5. Минимальный набор тестов

Добавить один и тот же моно-сигнал длительностью, например, 0,2 секунды для частот:

```text
16 000 Hz
22 050 Hz
24 000 Hz
44 100 Hz
48 000 Hz
```

Для каждого случая проверить:

- output sample rate равен выбранному контракту;
- `duration_secs()` до и после обработки отличается в пределах небольшой погрешности;
- output содержит конечные значения `f32`;
- число каналов не меняется;
- при отключённом enhancement sample rate и длина остаются без изменений;
- при ошибке enhancement активируется исходный PCM fallback.

Отдельно нужен тест на короткую фразу и тест на stereo/interleaved PCM. Нельзя ограничиваться только проверкой `samples.len()`: одинаковая длина samples при неправильном sample rate всё равно даёт неверную длительность.

## 6. Критерий завершения stage

Stage считается завершённым, когда:

- выбран и документирован контракт sample rate после DeepFilterNet;
- `AudioPcm` всегда получает фактическую частоту соответствующего PCM;
- тесты проходят для всех пяти входных частот, короткого сигнала и stereo;
- downstream-порядок `DeepFilterNet → Stretch → DSP → playback` не использует скрытое предположение о частоте;
- поведение fallback явно подтверждено тестом.

До завершения этого stage результаты сравнений с Resemble Enhance, DeepFilterNet и DSP на 16/24/44,1 kHz нельзя считать достоверными: сравниваться могут не только алгоритмы, но и разные sample-rate контракты.
