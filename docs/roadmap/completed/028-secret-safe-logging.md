# ROADMAP-028 — Безопасное диагностическое логирование

**Статус:** `completed` — чувствительные значения удалены из диагностического логирования

## Цель

Сохранить полезность логов для диагностики, но исключить попадание API keys, access tokens, Twitch tokens, MTProxy secrets и других чувствительных значений в открытом виде.

Эта задача не меняет `get_all_app_settings` и не требует разделения startup DTO. Фокус — backend-логи, frontend-диагностика и файлы, которые пользователь может передать на анализ.

## Почему это отдельная задача

Единый DTO используется намеренно для быстрой начальной загрузки приложения. В текущей модели это не считается самостоятельной уязвимостью: frontend является частью доверенного desktop-приложения, а изменения контракта проходят review.

При этом логирование имеет другой жизненный цикл: лог может быть сохранён на диске, отправлен разработчику, приложен к issue или передан внешнему инструменту анализа. Поэтому секрет в логе может выйти за пределы локального приложения и пользователя.

## Область аудита

Проверить все места, где данные могут попасть в диагностические артефакты:

- `tracing`/`log` вызовы backend;
- форматирование структур через `Debug` (`?settings`, `{:?}`);
- сообщения ошибок и контексты `anyhow`/`Result`;
- параметры Tauri-команд и ответы IPC, если они явно логируются;
- события и debug-снимки состояния;
- frontend `console.*`, debug logger и экспорт логов;
- `ai-log-failed.txt` и другие файлы диагностики;
- сетевые ошибки, в которых URL может содержать `token` или `api_key`;
- URL WebView, clipboard и сообщения внешним WebView.

## Классы чувствительных данных

Минимальный список полей для запрета открытого логирования:

- `api_key`, `apiKey`;
- `access_token`, `accessToken`;
- `token`;
- `secret`;
- `password`;
- `authorization` и значение заголовка `Authorization`;
- приватные session keys и cookie;
- URL с credentials или query-параметрами `token`, `key`, `secret`.

Нужно учитывать не только конкретные имена полей, но и значения внутри структур, например `TwitchSettings`, `WebViewSettings`, `TtsSettings`, `AiSettings` и `MtProxySettings`.

## Предлагаемая политика

### 1. Секреты не логируются

Запрещены:

```rust
tracing::debug!(settings = ?settings, "Loaded settings");
tracing::error!(error = ?err, settings = ?settings, "Operation failed");
debug!("API key: {}", api_key);
```

Даже `Debug` для структуры с секретными полями нельзя считать безопасным по умолчанию.

### 2. В логах остаётся диагностическая метаинформация

Вместо значения логируются:

- имя провайдера;
- имя раздела;
- `has_value: true/false`;
- длина значения, только если она действительно нужна для диагностики;
- безопасный идентификатор операции;
- статус и тип ошибки;
- host/port без credentials и секретных query-параметров.

Пример:

```rust
info!(
    provider = ?settings.ai.provider,
    has_api_key = settings.ai.openai.api_key.is_some(),
    "AI settings loaded"
);
```

### 3. Маскирование применяется как защитный слой

Для диагностических строк, где значение необходимо отличить от другого значения, использовать единый helper маскирования.

Предпочтительный безопасный default:

```text
[REDACTED]
```

Для ручной диагностики можно рассмотреть ограниченный формат:

```text
sk-••••••••••••••••••••-a1b2
```

Но последние символы тоже являются частью секрета. Поэтому suffix/prefix разрешать только для явно определённых диагностических случаев и не использовать в общем сериализаторе.

### 4. URL нормализуются перед логированием

Нельзя логировать URL целиком, если в нём могут быть:

- `token`;
- `api_key`;
- `secret`;
- username/password в authority;
- credentials в query или fragment.

В логах должен оставаться безопасный URL без чувствительных компонентов.

### 5. Ошибки не должны раскрывать исходные параметры

Нужно проверять ошибки HTTP-клиентов и библиотек: они могут включать URL, заголовки или тело запроса. Внешняя ошибка должна быть преобразована в безопасное диагностическое сообщение до записи в лог.

## Варианты реализации для обсуждения

### Вариант A — точечный аудит и ручное исправление

- найти все потенциально опасные логи;
- заменить их на `has_value`, provider, host и статус;
- добавить helper для единичных мест, где нужно маскирование;
- добавить тесты на helper.

Плюсы: небольшой риск и небольшой diff.

Минусы: новые логи в будущем снова могут случайно вывести секрет.

### Вариант B — структурированный redaction helper

Ввести единый backend helper/trait для безопасного представления чувствительных значений и использовать его в логируемых структурах.

Плюсы: единая политика и проще review.

Минусы: нельзя автоматически гарантировать безопасность обычного `Debug`; потребуется дисциплина и покрытие всех структур.

### Вариант C — глобальный фильтр по тексту логов

Перед записью искать в сообщении шаблоны `api_key`, `token`, известные значения или похожие секреты и заменять их.

Не рекомендуется как основной механизм: текстовый фильтр может пропустить секрет, повредить обычное сообщение или создать ложное ощущение безопасности. Его можно рассматривать только как последний защитный слой.

## Предлагаемый вариант

Взять вариант A с небольшим элементом варианта B.
Согласованный формат ограниченной маски: первые 4 символа + `[MASK]` + последние 4 символа. Для общего лога helper применяется только в явно определённых местах; общий сериализатор структур не маскирует значения автоматически.

1. провести поиск всех backend/frontend диагностических логов;
2. составить таблицу `место → риск → безопасная форма`;
3. вручную убрать прямое логирование секретов и структур с `Debug`;
4. добавить небольшой единый helper для маскирования отдельных значений;
5. нормализовать URL и ошибки сетевых клиентов;
6. добавить regression-тесты, проверяющие, что helper не возвращает исходное значение;
7. отдельно проверить содержимое файлов, которые можно передать на анализ.

## Что не входит в stage

- разделение `AppSettingsDto` на public/secret DTO;
- шифрование `settings.json`;
- изменение IPC-модели;
- изменение startup-загрузки настроек;
- запрет пользователю вручную копировать WebView token;
- проектирование полноценного secret vault;
- автоматическая очистка уже созданных старых логов.

## Вопросы для согласования

Решения:
1. Использовать ограниченную маску с первыми и последними 4 символами.
2. Отдельная команда или режим выгрузки безопасного диагностического лога не нужна.
3. Старые log-файлы не изменять и автоматически не очищать.
4. Frontend `console` доступен только в debug-режиме; включить его в аудит только для debug-контура.

## Критерии готовности stage

- найден список всех мест, где диагностические данные могут содержать секреты;
- ни один штатный backend-лог не выводит секрет в открытом виде;
- URL и ошибки сетевых клиентов не раскрывают credentials и secret query-параметры;
- диагностические файлы можно передать на анализ без ручного редактирования секретов;
- есть тесты на ключевой helper и проверки наиболее рискованных мест;
- политика зафиксирована в code review правилах для новых логов.
