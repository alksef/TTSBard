## 1. Решённая концепция (согласовано с пользователем)

Редизайн разделяет две вещи, которые сегодня слиты в один механизм:

### 1.1. SoundPanel — обычное окно (а не режим глобального перехвата)

**Сегодня:** открыли панель хоткеем → `set_interception_enabled(true)` →
low-level hook ловит **A-Z глобально** → играет звук → **сам себя выключает** и
прячет окно. Окно показывается **без фокуса** (`show()` без `set_focus()`),
поэтому клавиши и ловит hook, а не само окно.

**Хотим:** SoundPanel ведёт себя как обычное окно —
*вызвали → окно в фокусе → нажали кнопку (если забинжено) → воспроизвелось →
окно закрылось*. Клавиши обрабатывает **само окно** через обычный DOM
`keydown` (как уже делает `HotkeysPanel.vue` при записи хоткея), а не
глобальный OS-hook.

### 1.2. Intercept — опциональная доп. фича «однокнопочные хоткеи»

**Концепция пользователя:** «перехват становится доп-фичей для условно
однокнопочных горячих клавиш. Ради скорости и удобства платим тем, что на
NumPad биндим какие-нибудь действия, и тогда забинженные NumPad-клавиши
**фильтруются и не отдаются в систему** как нажатие».

То есть: включил toggle «Intercept» → забинженные клавиши NumPad (и опционально
другие) **не доходят до системы** (`return LRESULT(1)`), а вызывают действие;
остальные клавиши проходят насквозь (`CallNextHookEx`). Перехват работает
**глобально, всё время, пока toggle включён** (в отличие от сегодняшнего
soundpanel-hook, который живёт секунду и гаснет).

**Что биндится в Intercept (решено пользователем):** пока **только вызов окон**
— те же действия, что и у хоткеев: открыть soundpanel / playback-control /
главное окно + команды воспроизведения (pause/stop/repeat).
Звуки/TTS-фразы как действия — **позже**, отдельным планом. Сейчас цель:
**NumPad-кнопка вместо хоткея** (одна кнопка вместо `Ctrl+Shift+F2`).

## 3. Выбор механизма Intercept (согласовано)

**Вариант:** **обобщить существующий `WH_KEYBOARD_LL` hook** (а не
`tauri-plugin-global-shortcut`).

**Почему:**
- Hook уже стоит и работает; plugin имеет известную проблему фокуса
  («не срабатывает, когда окно приложения активно»), что критично —
  Intercept должен работать **всегда**, в т.ч. когда главное окно в фокусе.
- Точечная блокировка (`return LRESULT(1)` только для забинженных клавиш)
- именно то, что просит пользователь («фильтруются и не отдаются в систему»).
- Windows-only приемлемо: проект и так Windows-first (hook уже Windows-only).

**Альтернатива (plugin) — отвергнута** с обоснованием; зафиксировано здесь,
чтобы не возвращаться.

## 4. Ключевые архитектурные развилки и риски

### 4.1. ⚠️ Фокус SoundPanel (Stage 1) — главный риск

Чтобы SoundPanel ловил клавиши сам (через `keydown`), окно **должно иметь
фокус**. Сегодня фокуса намеренно нет. **Tradeoff, который надо решить в
плане:**

- **Вариант 1A (фокус + auto-hide по blur):** `show()` + `set_focus()`;
  окно закрывается, когда теряет фокус (`onBlur`). Минус: floating
  always-on-top окно стащит фокус с активного приложения (игра/стрим/IDE) —
  плохо для TTS-инструмента «поверх всего».
- **Вариант 1B (фокус только по клику/явному действию):** окно открывается
  без фокуса, фокус ставится по первому клику или спец-клавише; тогда
  `keydown` работает только после клика. Менее удобно.
- **Вариант 1C (веб-уровневый перехват без фокуса):** оставить hook, но
  обрабатывать клавиши **на стороне Vue** через Tauri-event из hook (hook
  ловит → шлёт событие → Vue-окно играет звук). По сути сохранить hook как
  транспорт, убрать «гашение и auto-hide». Это ближе к «работает как окно»
  без проблем фокуса.

**План #91 должен зафиксировать выбор** (рекомендация Claude — 1C как
наименее рискованный для floating-overlay, но финал за пользователем).

### 4.2. Persisted vs runtime-флаг Intercept

Сегодня `interception_enabled` — **runtime** (живёт в `SoundPanelState`,
включается на время показа панели). Для Intercept нужен **persisted toggle**
(вкл/выкл сохраняется между запусками) **+ persisted bindings**
(NumPad→action). Это **новое** persisted-состояние, не переиспользует
существующий runtime-флаг напрямую — нужна отдельная persisted-структура
(напр. `windows.json` или новая секция `settings.json`), а runtime-флаг
остаётся для soundpanel-режима или переименовывается.

### 4.3. Два контекста одного hook-процесса

Один `WH_KEYBOARD_LL` hook будет обслуживать и (возможно) остаточный
soundpanel-путь, и новый Intercept-режим. Нужны **чёткие правила
приоритета/режима** в proc-функции, чтобы они не конфликтовали. План должен
это прописать.

### 4.4. Набор actions Intercept = существующие хоткей-хендлеры

Действия, которые можно повесить на NumPad (переиспользуем готовое из
`hotkeys.rs`, **не дублируем**):
- `show_main_window`, `show_soundpanel_window`, `show_playback_control_window`
- `playback_pause`, `playback_stop`, `playback_repeat`

→ Извлечь хендлеры в переиспользуемые функции, вызывать и из хоткея, и из
Intercept.

## 5. Что входит в план #91 (объём реализации)

1. **Обобщение hook** (`hook.rs` + `state.rs`): NumPad-диапазон VK, убрать
   авто-гашение, toggle-гейт, фильтр только забинженных клавиш → action,
   остальные `CallNextHookEx`. Решить §4.3 (режимы).
2. **SoundPanel как окно**: решить §4.1 (вариант фокуса), перенести A-Z→звук
   из hook в обработчик окна (Vue/через событие), auto-close после звука.
3. **Intercept state + storage**: persisted toggle + persisted bindings
   (§4.2), DTO, команды `get_intercept_settings` / `set_intercept_*`,
   события `InterceptionChanged`.
4. **Actions-слой**: вынести хоткей-хендлеры в reusable-функции; Intercept
   вызывает их по биндингу.
5. **UI**: новая панель «Перехват» в `Sidebar.vue` (ниже «Горячие клавиши»),
   по образцу `HotkeysPanel.vue`: toggle вкл/выкл + запись NumPad-биндинга +
   выбор действия (вызов окна / команда воспроизведения).
6. **Не входит (позже):** биндинг на звук/TTS-фразу как действие Intercept.
