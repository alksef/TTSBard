# ROADMAP-046 — Реорганизация документации и AI-workflow

**Дата:** 2026-07-22

**Статус:** `in_progress` — этапы 1–5 завершены, этап 6 готов к проверке

**Область:** только документация, инструкции агентов и правила хранения рабочих
артефактов. Исходный код приложения этим stage не изменяется.

## Цель

Разделить постоянную документацию проекта, дорожную карту, задачи реализации и
временные артефакты работы AI-агентов.

После миграции:

- `docs/` содержит только поддерживаемые документы, полезные разработчику или
  пользователю вне конкретной AI-сессии;
- `docs/roadmap/` показывает путь развития проекта и историю принятых решений;
- `docs/tasks/` содержит только отобранные долговечные задачи реализации;
- мелкие задания DeepSeek, промежуточные ревью и логи находятся в локальном
  gitignored-каталоге `.work/ai/`;
- правила процесса имеют один источник истины;
- старые `stage`-материалы сначала классифицируются, а служебные DeepSeek
  plan/task/review-артефакты не переносятся в новый архив и удаляются обычным
  коммитом.

## Принципы миграции

1. Миграция выполняется небольшими этапами, без одновременного массового
   перемещения и переписывания содержимого.
2. Перед каждым этапом фиксируется текущий `git status`; существующие
   пользовательские изменения не включаются в documentation-коммиты.
3. История Git не переписывается. Удалённые планы и отчёты остаются доступными
   в старых коммитах.
4. Файл переносится в новую структуру только после проверки актуальности.
5. Нельзя создавать `archive/`, в который без разбора складывается весь старый
   `docs/`: архив без lifecycle лишь переносит существующую свалку.
6. Сначала создаются новые каталоги и индексы, затем переносятся документы,
   после этого обновляются ссылки и только в конце удаляются старые каталоги.

## Целевая структура

```text
docs/
├── README.md
├── roadmap/
│   ├── README.md
│   ├── active/
│   ├── completed/
│   └── rejected/
├── tasks/
├── decisions/
├── user/
├── development/
├── integrations/
├── reference/
└── research/

.work/
└── ai/
    └── <work-id>/
        ├── context.md
        ├── tasks/
        ├── reviews/
        ├── logs/
        └── screenshots/
```

### Назначение каталогов

- `docs/roadmap/` — направления развития продукта: что исследуется, принято в
  работу, завершено или осознанно отклонено.
- `docs/tasks/` — конкретные долговечные задачи реализации. Это не хранилище
  каждого prompt и не архив завершённых работ.
- `docs/decisions/` — короткие устойчивые решения: контекст, выбранный вариант,
  последствия и условия пересмотра.
- `docs/user/` — пользовательские инструкции и описания функций.
- `docs/development/` — сборка, отладка, релиз, тестирование и процесс работы.
- `docs/integrations/` — OBS/WebView, Telegram, Twitch, VTube Studio и другие
  внешние интеграции.
- `docs/reference/` — поддерживаемая техническая справка, которую нельзя
  надёжнее получить прямо из исходного кода.
- `docs/research/` — только отобранные исследования с источниками, выводом и
  дальнейшим решением.
- `.work/ai/` — локальные задания, итерации, вывод команд и временные отчёты.
  Каталог целиком игнорируется Git.

## Roadmap вместо `stage`

Текущий `docs/stage/` переименовывается по смыслу в `docs/roadmap/`. Единицей
становится **roadmap item**, а не stage или initiative.

### Состояния roadmap item

- `exploring` — исследование и сравнение вариантов;
- `planned` — направление принято, но реализация не начата;
- `in_progress` — реализация выполняется;
- `completed` — ожидаемый результат достигнут;
- `deferred` — решение отложено, но не отклонено;
- `rejected` — принято решение не реализовывать;
- `superseded` — заменено другим roadmap item или решением.

`exploring`, `planned`, `in_progress` и `deferred` находятся в `active/`.
Терминальные состояния физически отделяются в `completed/` и `rejected/`.
`superseded` размещается рядом с итоговым состоянием и обязательно содержит
ссылку на заменивший документ.

### Содержимое roadmap item

Каждый файл содержит:

```yaml
---
id: ROADMAP-046
status: planned
created: 2026-07-22
updated: 2026-07-22
related_tasks: []
---
```

Обязательные смысловые разделы:

- проблема и пользовательская ценность;
- исследованные варианты;
- принятое направление или причина отказа;
- крупные этапы без мелких implementation tasks;
- открытые вопросы и зависимости;
- outcome для завершённого item;
- `Reconsider when` для отклонённого item, если решение может измениться.

При переносе сохраняются существующие номера `01`–`46`, но имена приводятся к
трёхзначному виду: `046-documentation-structure-migration.md`. Массовая
перенумерация старых items не выполняется.

## Правила для `docs/tasks/`

`docs/plans/` переименовывается в `docs/tasks/` и превращается в
curated-каталог, а не журнал всех реализаций.

Task коммитится, если выполняется хотя бы одно условие:

- изменение проходит через несколько подсистем или этапов;
- до реализации нужно согласовать архитектуру или контракт;
- задача содержит важные ограничения, риски или порядок миграции;
- работа рассчитана на несколько независимых запусков;
- документ должен пережить отдельную AI-сессию и остаться исполнимым позже.

Не коммитятся:

- одноразовые task-prompts;
- мелкие исправления после ревью;
- копии общего roadmap item;
- task-файлы, состоящие только из очевидного списка файлов;
- отчёты о том, что агент выполнил команды.

Для новых задач используется имя `YYYY-MM-DD-short-description.md`. Глобальный
числовой counter больше не используется. Существующие номера сохраняются у
немногих отобранных задач, если на них уже есть устойчивые ссылки.

Статусы задач: `planned`, `in_progress`, `blocked`, `deferred`. После выполнения
task удаляется, а outcome переносится в roadmap, decision или профильную
документацию.

## Политика AI-артефактов

### Коммитить

- единый нормативный AI-workflow;
- шаблон качественной implementation task;
- roadmap items;
- архитектурные решения;
- отобранные implementation tasks;
- итоговую пользовательскую и разработческую документацию.

### Не коммитить

- `.work/ai/**/tasks/` — задания DeepSeek и corrective iterations;
- `.work/ai/**/reviews/` — промежуточные verdict и review notes;
- `.work/ai/**/logs/` — stdout, stderr, `.log`, `.err`, JSONL;
- `.work/ai/**/screenshots/` — временные доказательства и диагностику;
- сгенерированные diff/patch-файлы, когда канонический результат уже в Git.

Исключение: если локальный task оказался самостоятельной полезной технической
спецификацией, он очищается от инструкций конкретной модели и переносится в
`docs/tasks/` или `docs/decisions/`. Сам исходный prompt не коммитится.

## Карта переноса существующих каталогов

### `docs/stage/`

Перенести после индивидуальной классификации:

- актуальные и отложенные items → `docs/roadmap/active/`;
- завершённые items с полезным outcome → `docs/roadmap/completed/`;
- осознанно отклонённые варианты → `docs/roadmap/rejected/`;
- файлы, полностью дублирующие актуальную документацию или код → удалить после
  переноса единственного полезного вывода.

После переноса всех файлов удалить `docs/stage/`.

### `docs/plans/` → `docs/tasks/`

Проверить каждый план по curated-критериям:

- активные и долговечно полезные задачи оставить;
- дубли пар `design`/`implementation` объединить, если разделение больше не
  помогает;
- завершённые планы без полезного rationale удалить;
- устаревшие draft/in-progress планы сверить с кодом и либо пометить
  `abandoned`/`superseded`, либо удалить;
- удалить `docs/plans/counter.txt` после перехода на date-based naming;
- переименовать отобранный каталог в `docs/tasks/`.

Локальные задания DeepSeek остаются в `.work/ai/` и не смешиваются с
долговечными проектными tasks.

### `docs/deepseek/`

- `docs/deepseek/WORKFLOW.md` объединить с процессными правилами в
  `docs/development/ai-workflow.md`;
- `docs/deepseek/plan/` удалить целиком: эти файлы являются служебными
  инструкциями для реализации и не мигрируют в `docs/tasks/`;
- если для незавершённого запуска ещё нужен контекст из DeepSeek plan, временно
  перенести его в `.work/ai/<work-id>/context.md`, не добавляя в Git;
- `docs/deepseek/tasks/` удалить из текущего дерева после завершения активных
  незакоммиченных работ;
- `docs/deepseek/reviews/` удалить после извлечения действительно долговечных
  findings в roadmap, task, decision или audit;
- raw OpenCode logs и `.log.err` удалить локально либо перенести в
  `.work/ai/<work-id>/logs/`;
- после извлечения полезной информации удалить `docs/deepseek/`.

Новые DeepSeek tasks с момента включения нового workflow создаются только в
`.work/ai/`.

### `docs/reviews/`

- промежуточные отчёты не возвращать в tracked-документацию;
- reusable review prompt перенести в шаблоны workflow, если он остаётся нужен;
- большой аудит сохранять только после переработки в самостоятельный документ
  с долгосрочным backlog, например `docs/development/audits/...`;
- после классификации удалить старый каталог.

### `docs/research/`, `docs/works/`, `docs/ideas/`, `docs/depth-analysis/`

- `docs/research/` оставить, но проверить наличие источников, вывода и связи с
  roadmap item или decision;
- полезные документы из `works`, `ideas` и `depth-analysis` переработать в
  roadmap/research/decision;
- черновики, сгенерированные изображения и промежуточный анализ перенести в
  `.work/ai/` либо удалить;
- каталоги `works`, `ideas` и `depth-analysis` после классификации удалить;
- не коммитить целиком текущий untracked `docs/depth-analysis/` без ревью
  каждого файла.

### Корневые файлы `docs/`

Предварительная целевая классификация:

- `QUICK_START.md`, текущий пользовательский `README.md`, презентация и
  пользовательские guides → `docs/user/`;
- `architecture.md` → `docs/development/architecture.md` либо краткий
  `docs/reference/architecture.md` после проверки актуальности;
- `KEY_DECISIONS.md` → разбить на отдельные `docs/decisions/*.md`; исходный
  монолит удалить после переноса решений;
- `github/build.md`, `dev/*`, build/debug/release инструкции →
  `docs/development/`;
- `sse-integration.md`, `webview.md` и документы внешних сервисов →
  `docs/integrations/`;
- `rust-modules.md`, `vue-components.md`, `cheatsheet.md` → проверить на
  устаревание; оставить в `reference/` только то, что не дублирует структуру
  исходного кода и реально поддерживается;
- `project-overview.md` → объединить с новым `docs/README.md` или
  `docs/development/architecture.md`, не сохранять третий обзор проекта;
- `note-last-claude.txt` удалить;
- `.context-rules.md` удалить после переноса актуальных context-правил в
  единый workflow/agent instructions;
- старые bug reports из `docs/bugs/` либо превратить в roadmap/rejected/
  completed records, либо удалить после подтверждения исправления.

Точная раскладка пользовательских audio/DSP-документов выполняется отдельным
этапом после проверки их аудитории и актуальности; механически переносить их
только по имени нельзя.

## Единый workflow

Нормативный процесс хранится в одном файле:

```text
docs/development/ai-workflow.md
```

`AGENTS.md`, `CLAUDE.md` и другие tool-specific instruction files содержат
только короткую роль агента, обязательные ограничения и ссылку на этот файл.
Они не копируют полное описание процесса.

Workflow должен явно определить:

1. порог между прямой мелкой правкой и передачей реализации DeepSeek;
2. когда требуется roadmap item, implementation plan или decision;
3. создание task workspace в `.work/ai/<work-id>/`;
4. фиксацию baseline SHA, исходного dirty state и разрешённого scope;
5. запуск OpenCode через проверенную PowerShell-команду без абсолютного пути к
   конкретному checkout;
6. task-scoped review вместо проверки всего `git diff HEAD`;
7. risk-based validation matrix для Rust, frontend, UI и packaging;
8. максимум corrective iterations и условие эскалации;
9. завершение работы: обновить durable docs, удалить/оставить локальные
   artifacts, подготовить чистый scoped commit.

Устаревшие `.claude/rules/planning.md`, context exclusions и ссылки на
несуществующие `.Codex/rules` обновляются только после появления нового
canonical workflow.

## Этапы выполнения

### Этап 1 — каркас без переносов

1. Создать целевые каталоги с короткими `README.md`.
2. Создать `.work/ai/` и добавить `.work/` в `.gitignore`.
3. Создать `docs/roadmap/README.md` с текущими разделами Active, Planned,
   Completed и Rejected.
4. Не перемещать существующие документы на этом этапе.

**Приёмка:** новое дерево и lifecycle описаны; существующие ссылки работают;
ни один старый файл не удалён.

### Этап 2 — миграция roadmap

1. Классифицировать все файлы `docs/stage/` по фактическому состоянию кода.
2. Переносить небольшими тематическими группами.
3. Добавлять metadata и outcome/rejection rationale.
4. Обновлять `docs/roadmap/README.md` после каждой группы.
5. Удалить пустой `docs/stage/` только после проверки ссылок.

**Приёмка:** каждый старый stage имеет однозначный исход: active, completed,
rejected, merged или deleted-with-rationale.

### Этап 3 — пользовательская и инженерная документация

1. Разделить корневые документы по аудитории.
2. Объединить дублирующие обзоры.
3. Проверить команды, пути, версии и названия модулей по текущему коду.
4. Обновить `docs/README.md` как короткую карту, а не ещё один полный guide.

**Приёмка:** в корне `docs/` остаются только README и целевые подкаталоги;
нет двух документов, претендующих на один источник истины.

### Этап 4 — DeepSeek artifacts

1. Удалить `docs/deepseek/plan/` без переноса служебных prompts.
2. Удалить task/review execution trail из tracked tree.
3. Переключить новые запуски на `.work/ai/`.
4. Перенести полезные правила старого workflow в `docs/development/`.

**Приёмка:** `docs/deepseek/` больше не нужен; локальные execution artifacts не
появляются в Git.

### Этап 5 — implementation tasks

1. Проверить каждый файл старого `docs/plans/` по фактическому состоянию кода.
2. Оставить только незавершённые долговечные задачи.
3. Переименовать curated-каталог в `docs/tasks/`.
4. Удалить counter и ссылки на завершённые execution plans.

**Приёмка:** `docs/tasks/` содержит только самостоятельно исполнимые текущие
задачи; завершённая история доступна через roadmap и Git history.

### Этап 6 — решения и исследования

1. Разбить актуальные пункты `KEY_DECISIONS.md` на короткие decisions.
2. Для отменённых решений указать `superseded` и ссылку на замену.
3. Связать curated research с roadmap/decision.
4. Удалить неактуальные research/scratch-документы.

**Приёмка:** важное «почему» сохранено отдельно от временной истории работы.

### Этап 7 — унификация инструкций

1. Создать `docs/development/ai-workflow.md`.
2. Обновить `AGENTS.md`, `CLAUDE.md`, `.ignore`, `.claudeignore` и связанные
   rules так, чтобы они не противоречили друг другу.
3. Удалить устаревшие counters, hardcoded Linux-команды и несуществующие пути.
4. Добавить проверенную Windows-команду или PowerShell-wrapper запуска.

**Приёмка:** для каждого правила существует один нормативный источник.

### Этап 8 — финальная очистка и автоматическая проверка

1. Проверить все относительные Markdown-ссылки.
2. Найти ссылки на удалённые `docs/stage`, `docs/deepseek`, старые plans и
   reviews.
3. Проверить отсутствие tracked `.log`, `.err`, scratch screenshots и пустых
   файлов.
4. Добавить лёгкую CI/local-проверку Markdown links и обязательных status для
   roadmap/tasks/decisions.
5. Обновить корневой README проекта ссылками на новую документацию.

**Приёмка:** структура объясняется из `docs/README.md`, ссылки проходят
автоматическую проверку, временные AI-артефакты не появляются в `git status`.

## Что не делаем

- не переписываем Git history ради косметической очистки;
- не коммитим весь текущий untracked DeepSeek execution trail;
- не переносим старые файлы в `archive/` без классификации;
- не смешиваем долговечные tasks с локальными prompts в `.work/ai/`;
- не вводим отдельный issue tracker внутри Markdown;
- не обновляем исходный код приложения в рамках этой миграции;
- не выполняем всю миграцию одним огромным коммитом.

## Итоговая приёмка stage

Stage завершён, когда:

1. `docs/stage/` заменён на управляемый `docs/roadmap/`.
2. `docs/tasks/` содержит только отобранные implementation tasks.
3. `docs/deepseek/`, старые review/scratch-каталоги удалены после извлечения
   полезной информации.
4. Новые task/review/log artifacts создаются только в `.work/ai/` и не
   отслеживаются Git.
5. Пользовательские, инженерные, integration и reference docs разделены по
   аудитории.
6. Архитектурные решения имеют собственные короткие records.
7. `AGENTS.md`, `CLAUDE.md` и workflow не противоречат друг другу.
8. Все ссылки и status metadata проходят автоматическую проверку.
