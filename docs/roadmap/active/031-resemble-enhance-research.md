# ROADMAP-031 — Исследование Resemble Enhance для TTSBard

**Дата:** 2026-07-13  
**Статус:** `deferred` — исследование завершено, реализация отложена до отдельного proof-of-concept
**Источник:** [официальный репозиторий Resemble Enhance](https://github.com/resemble-ai/resemble-enhance)

## 1. Вывод для обсуждения

Resemble Enhance интересен как отдельный offline-enhancer, но не как следующий шаг в текущем realtime PCM-потоке. Его denoiser потенциально можно вынести в ONNX, а полный enhancer следует рассматривать только после отдельного proof-of-concept и замеров на целевой машине.

## 2. Что именно делает проект

Официальное описание разделяет pipeline на два модуля:

1. **Denoiser** — отделяет речь от шума.
2. **Enhancer** — повышает воспринимаемое качество, восстанавливает искажения и расширяет полосу.

Модели обучались на речи 44.1 kHz. Это не означает, что из 16/24 kHz можно безошибочно восстановить оригинальные высокие частоты: enhancer генерирует правдоподобное продолжение спектра и может менять тембр/артефакты. Для TTS это особенно важно — «улучшение» может изменить дикцию или индивидуальность голоса.

Актуальные факты на дату документа:

- официальный репозиторий в основном Python/PyTorch;
- опубликованный релиз — `v0.0.1` от 14 декабря 2023;
- лицензия репозитория — MIT;
- официальная CLI рассчитана на обработку файлов/каталогов, а не на встроенный Rust streaming API;
- параметры enhancer включают число function evaluations (`nfe`), solver, `lambd` и `tau`, поэтому полный enhancer не является дешёвым фиксированным фильтром.

## 3. Совместимость с текущим потоком

Текущий поток уже даёт нужную точку интеграции:

```text
TTS bytes → decode → AudioPcm → optional enhancement → playback
```

Но есть ограничения:

- `AudioPcm` обычно содержит короткие фразы, тогда как модель должна корректно обрабатывать padding, границы и длительность;
- полноценный enhancer, вероятно, потребует resample к собственной частоте и затем контроля длины/каналов;
- запуск Python subprocess на каждую фразу создаст задержку и отдельную точку отказа;
- встраивание PyTorch в Tauri неприемлемо по размеру и сложности поставки;
- ONNX-путь для полного enhancer не подтверждён upstream. В issue об ONNX упоминается сторонний экспорт denoiser, при этом enhancer описан как трудный для экспорта и слишком медленный для ряда применений.

## 4. Реалистичные варианты

### Вариант A — не интегрировать сейчас

Сначала реализовать DSP-stage. Он дешёвый, детерминированный и хорошо соответствует чистому TTS, где нет реального фонового шума. DeepFilterNet уже закрывает отдельный сценарий очистки.

### Вариант B — offline «улучшить файл»

Добавить отдельную команду экспорта/обработки WAV, которая вызывает изолированный Python environment или внешний helper. Это не должно влиять на realtime playback. Нужны явные индикатор прогресса, отмена, проверка версии модели и сохранение оригинала.

### Вариант C — только denoiser через ONNX proof-of-concept

Исследовать сторонний экспорт отдельно от upstream: вход mono/48 kHz или требуемый model rate, фиксированные chunk/overlap, состояние между чанками, CPU latency и лицензии всех компонентов. До подтверждения качества не включать его в пользовательский pipeline.

### Вариант D — полный enhancer in-process

Не планировать как обычную Rust-задачу. Для него сначала нужны воспроизводимый экспорт всех частей модели, runtime, модельные веса, контроль генеративной случайности, benchmark CPU/GPU и тесты на сохранение дикции. Только после этого можно выбирать ONNX Runtime, другой runtime или helper process.

## 5. Предлагаемый эксперимент

Подготовить 20–30 одинаковых TTS-фраз от используемых провайдеров: 16/24/44.1 kHz, MP3 и WAV, разные длины и согласные. Сравнить:

```text
original
DSP
DeepFilterNet
Resemble denoiser (если доступен воспроизводимый экспорт)
Resemble full enhancer offline
```

Фиксировать wall-clock time, peak/LUFS, длительность, sample rate, CPU/RAM, размер моделей, субъективные артефакты и сохранение текста/дикции. Отдельно проверить, не появляется ли «пластиковый» тембр или ложные высокочастотные детали.

## 6. Критерии решения

Resemble Enhance имеет смысл включать только если одновременно выполнены условия:

- результат стабильно лучше DSP + DeepFilterNet на целевых TTS-образцах;
- обработка не ломает границы коротких фраз;
- offline-вариант явно обозначен пользователю, либо realtime latency укладывается в согласованный бюджет;
- модель и все зависимости можно легально и воспроизводимо поставлять;
- есть fallback без потери воспроизведения;
- качество проверено не только спектрограммой, но и прослушиванием.

До этого статус — «кандидат на offline restoration», не «готовый эффект». Для текущего продукта приоритет: сначала DSP A/B и измерения, затем отдельный Resemble POC.
