# ROADMAP-013 — Остаёмся на JSON вместо SQLite

**Дата:** 2026-07-08
**Статус:** `rejected` — переход на SQLite отклонён, остаёмся на JSON
**Решение:** **остаться на JSON-файлах + in-memory Trie** (не переходить на SQLite). Документировать
порог, при котором миграция на БД стала бы оправданной, и что она будет означать.
**Связано:** `02-local-history-autocomplete.md`, `05-phrase-history.md`, stage `12` (новый tabs.json —
тоже JSON), `history.rs`.

## Цель (запрос пользователя)
Оценить текущее хранилище истории (`phrase_history.json` + `input_history.json` + `ngrams.json` +
скоро `tabs.json`) и решить — переходить ли на embedded БД (SQLite).

## Что сейчас хранится (объёмная оценка)

| Файл | Структура | Лимит | Реальный объём |
|---|---|---|---|
| `phrase_history.json` | `Vec<PhraseEntry{id,text,count,last_used}>` | 200 (`PHRASE_HISTORY_SIZE`, `history.rs:9`) | ~десятки КБ — единицы МБ |
| `input_history.json` | `Vec<HistoryEntry{word,count,last_used}>` (Trie-источник) | без жёсткого, на практике ~тысячи слов | сотни КБ |
| `ngrams.json` | `NgramData{bigrams, trigrams}` HashMap<HashMap> | растёт с историей | сотни КБ — единицы МБ |
| `tabs.json` (новый, stage 12) | `Vec<EditorTab{id,title,text}>` | ~50 | десятки КБ |

**Итого все пользовательские данные:** единицы МБ в худшем случае. Загружаются в память целиком
при старте (`history.rs:90-116`), поиск/предложения — по in-memory Trie/HashMap (микросекунды,
см. spike 80: 5µs/слово).

## Почему SQLite здесь = оверкилл

1. **Объём не требует БД.** SQLite даёт выигрыш при: тысячах-миллионах записей, полнотекстовом
   поиске (FTS), индексах, сложных JOIN-запросах, конкурентных писателях. Здесь — сотни фраз,
   простые `filter(by substring)` + `sort by count/last_used`, один писатель (UI-поток через Tauri).
2. **Цена внедрения:**
   - Новая зависимость `rusqlite` (или `sqlx`) + bundled SQLite = **+вес инсталлятора** (~1-2 МБ
     статической линковки) — критично для проекта, где следят за размером бандла (см. spike 80 про
     вес словарей).
   - Схема + **миграции** (каждое изменение структуры — migration-скрипт), тогда как JSON
     меняется добавлением `#[serde(default)]`-поля (механизм уже отлажен, `PhraseEntry` `history.rs:18-31`).
   - Сериализация Trie/ngrams в реляционную схему — нетривиально (HashMap<HashMap> не ложится в
     таблицы естественно; либо JSON-колонка, либо потеря преимуществ БД).
   - In-memory Trie **всё равно нужен** для быстрого prefix/substring-поиска — значит, из БД всё
     равно грузим всё в память при старте → выигрыш БД в поиске не используется.
3. **Существующая архитектура(file-based) — принципиальное решение проекта** (`KEY_DECISIONS` #15,
   зафиксировано в `05` при выборе A vs C). JSON + Trie — согласованное, простое в бэкапе
   (скопировать папку `%APPDATA%\ttsbard\`), без внешних зависимостей.

## Когда SQLite БЫЛ бы оправдан (порог пересмотра)

Зафиксировать как **roadmap-триггеры** (реализовать ПОЗЖЕ, не сейчас), если появится ANY из:
- **Объём:** история фраз растёт > 2000–5000 (текущий лимит 200) ИЛИ пользователь явно просит
  «хранить всю историю навсегда». Тогда кольцевой буфер 200 перестаёт устраивать.
- **Полнотекстовый поиск:** нужен поиск фраз не по подстроке, а по смыслу/ранжированный
  (FTS5, токенизация) — Trie-подстрока уже не хватает.
- **Метаданные/теги:** к фразам/звукам добавляются теги, категории, избранное, атрибуты →
  реляционная модель естественнее (сейчас `PhraseEntry` — плоский).
- **Конкурентность:** несколько окон/процессов пишут историю одновременно → нужны транзакции
  (сейчас один писатель).
- **Аналитика/журналирование:** детальная история по времени (каждая отправка как отдельная запись
  с timestamp), статистика использования → растёт без лимита → БД.

> **Roadmap (если порог достигнут):** единый `ttsbard.db` (SQLite), миграция JSON→DB один раз при
> старте, FTS5 для поиска фраз, таблицы `phrases`/`words`/`ngrams`/`tabs`/`sounds`. План — отдельным
> stage, когда триггер сработает.

## Решение
**Сейчас — остаёмся на JSON.** Новый `tabs.json` (stage 12) — той же модели. Roadmap на SQLite
зафиксирован, но не активируется без триггеров выше. Это **осознанный отказ** от преждевременной
оптимизации.

## KEY_DECISIONS
- **Не переходить на SQLite** при текущих объёмах (≤200 фраз, слова/ngrams — единицы МБ).
- **JSON + in-memory Trie остаётся** — простота, малый бандл, лёгкий бэкап, отлаженные `#[serde(default)]`-миграции.
- **Порог пересмотра** зафиксирован (>2000 фраз / FTS-поиск / теги / конкурентность / аналитика).
- **Trie всё равно в памяти** → выигрыш БД в поиске не реализуется → аргумент против БД усиливается.
