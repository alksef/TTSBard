# Research: Актуализация проблемы и идеи улучшения работы с TTS

**Дата:** 2026-06-28
**Автор:** Claude (research)
**Статус:** research note
**Связано:** `PROBLEMS.md`, `docs/KEY_DECISIONS.md`, `docs/stage/01..03`, `docs/plans/71..73`

---

## 1. Цель документа

Переоценить исходную проблему (`PROBLEMS.md`), проверить — **достаточно ли текущего решения
после доработок (планы 71–73)**, и собрать **новые идеи** улучшения работы с TTS на основе
текущих best practices и конкурентов (AAC-приложения, стримерские TTS-тулы, realtime-voice).

## 2. Что уже есть в проекте (сильные стороны)

По `README.md` / `docs/KEY_DECISIONS.md` (27 архитектурных решений) проект уже покрывает:

| Категория | Реализовано |
|---|---|
| **Скорость ввода** | Глобальный keyboard hook (`WH_KEYBOARD_LL`), пресеты (`\key`, `%username`), конвертация чисел, F6-режим (несколько сообщений без закрытия окна) |
| **Интеграция** | OBS WebView (SSE), Twitch Chat, dual audio output (динамики + виртуальный микрофон) |
| **TTS-провайдеры** | OpenAI TTS, Silero (Telegram), Local TTS (TTSVoiceWizard-совместимый), Fish Audio (кастомные голоса) |
| **AI** | Текстовая коррекция (OpenAI + Z.ai) через `AiClient` trait, quick/AI режимы редактора |
| **UX/Окна** | System tray, плавающие окна (click-through, no-activate, layered), защита от захвата экрана |
| **Звук** | SoundPanel (26 привязок A-Z), audio effects (fade-out trim) |
| **Прочее** | Кастомные hotkeys, тема dark/light, единый proxy, per-module логирование |

**Вывод:** проект **очень зрелый** для своего ниши (TTS для общения с минимальным отвлечением,
Windows-only, с упором на стримеров/мессенджеры).

## 4. Актуализация: достаточно ли решения после доработок 71–73?

### Краткий ответ
Для **основного сценария** (общение в мессенджерах/стримах, Windows) — **да, после 71–73
решение становится сильным и завершённым по核心-функционалу**.

### Что точно закрыто после 71–73
- ✅ Скорость ввода (пресеты + автокомплит слов + предсказание фраз).
- ✅ Минимальное отвлечения (горячие клавиши, tray, защита от захвата).
- ✅ Интеграции (OBS/Twitch/dual-audio).
- ✅ Качество речи (несколько провайдеров + AI-коррекция).

### Где остаются пробелы (см. раздел 5)
- **Управление воспроизведением** речи (очередь, прерывание, пауза) — слабо.
- **Качество произношения специфичных слов** (имена, никнеймы, термины) — нет лексикона.
- **Контекст фраз** между сессиями (история отправленного) — только пресеты.
- **Низкая латентность** (стриминг аудио) — зависит от провайдера, нет собственной очереди.

**Вывод:** ядро достаточно; но есть ~5–7 идей, которые могут **существенно** поднять
пользовательскую ценность при относительно умеренной сложности. См. ниже.

## 5. Идеи улучшения (по приоритету ценность/сложность)

### 🟢 Высокая ценность, низкая/средняя сложность

**A. Очередь и управление воспроизведением (Speech Queue + Interrupt)**
- Сейчас отправка нескольких фраз подряд может конфликтовать (зависит от провайдера).
- **Идея:** единая очередь воспроизведения: пауза/resume, **отмена текущей фразы** (горячая
  клавиша / Esc), «очистить очередь», индикатор «говорит/в очереди N».
- Конкуренты (TTS Monster, TTSLabs, TTSVoiceWizard) имеют moderation/speech queue.
- *Связь:* ложится на существующую аудио-архитектуру (rodio + event-driven MPSC).

**B. Лексикон произношения (Pronunciation Dictionary)**
- Имена, никнеймы, термины, аббревиатуры часто звучат криво у TTS.
- **Идея:** пользовательский словарь `слово → фонетика/произношение` (или `слово → замена`),
  применяемый перед отправкой в TTS (как препроцессор, но для звучания).
- Конкуренты: «custom word dictionaries (lexicon)» у TTSLabs; SSML-произношение у Inworld.
- *Связь:* естественно расширяет существующий Text Preprocessor (`docs/KEY_DECISIONS.md` #9).

**C. История отправленных фраз (Persistent Phrase History)**
- Сейчас история — только в сессии (`PROBLEMS.md:86`) и пресеты вручную.
- **Идея:** persistent журнал последних N отправленных фраз с быстрым повтором
  (один клик/горячая клавиша → повторно озвучить).
- AAC-приложения (Speech Assistant, Spoken) имеют «history for quick access».
- *Связь:* переиспользует хранилище плана 72 (`input_history.json` → `phrase_history.json`).

**D. Авто-ducking / авто-приглушение фона при речи**
- Когда TTS говорит, фоновая музыка/звук могут перекрывать речь.
- **Идея:** автоматически приглушать системный звук или SoundPanel-воспроизведение на время
  речи (или хотя бы не накладывать SoundPanel-звук поверх активной фразы).
- *Связь:* SoundPanel + аудио-движок; частично «non-blocking producer-consumer» уже есть.

### 🟡 Средняя ценность / средняя сложность

**E. Низколатентный стриминг аудио (sentence-level streaming)**
- Современные realtime-TTS (Cartesia 40–90ms, Inworld <130ms, Simplismart <400ms) отдают
  аудио чанками до генерации всей фразы.
- **Идея:** если провайдер поддерживает streaming API (OpenAI/UDP-стриминг), начинать
  воспроизведение с первого чанка — радикально снижает воспринимаемую задержку.
- *Сложность:* зависит от провайдера; OpenAI уже поддерживает streaming TTS.

**F. Voice cloning / персональный голос**
- Пользователи (особенно AAC, потерявшие голос) хотят «свой» голос.
- **Идея:** интеграция cloning-провайдера (ElevenLabs / Fish Audio custom model уже частично
  есть) как ещё одного провайдера; запись образца → клонирование.
- *Связь:* архитектура `TtsEngine` trait (#7) + Fish Audio (#22) уже близки.

**G. Multi-voice форматирование в одном сообщении**
- Конкуренты: «Name: Message» → разный голос для частей сообщения.
- **Идея:** синтаксис (например уже частично задумано `[<code>]` — смена голоса, см. README
  «Запланированные функции») для переключения голоса внутри одной фразы.
- *Связь:* достать из бэклога README и реализовать через провайдер-абстракцию.

### 🔵 Низкий приоритет / «nice to have»

**H. Фильтр мата / цензуры (Profanity filter)**
- Для стримеров: замена/скрытие плохих слов перед TTS.
- *Связь:* ещё один шаг препроцессора.

**I. Whisper STT (обратный канал: голос → текст)**
- TTSVoiceWizard использует Whisper для голосовых команд.
- *Идея:* опционально — голосовые команды/триггеры. Выходит за текущий скоуп, но архитектурно
  возможно.

**J. Снапшоты/профили настроек**
- Несколько профилей (стрим / личное общение / рабочий звонок) с разными голосами,
  провайдерами, hotkeys. Быстрое переключение.

## 6. Рекомендация по роадмапу

1. **Сначала завершить 71–73** (редактор + автокомплит) — это база и закрытие «скорости ввода».
2. **Затем «быстрые победы» (A, B, C)** — очередь/прерывание, лексикон произношения,
   persistent-история фраз. Высокое соотношение ценность/сложность, ложатся на существующую
   архитектуру.
3. **Потом E (стриминг)** и **G (multi-voice)** — если latency/гибкость голоса станут
   критичны.
4. **F (voice cloning)** — отдельная большая тема, по запросу аудитории.
5. H/I/J — по мере возникновения потребности.

---

## 7. Итог

- **Проблема актуальна**, но проект её **в значительной мере уже решает**.
- После доработок **71–73** ядро решения становится **достаточным** для основного сценария.
- **Дополнительного ценностного потолка не достигнуто** — есть ~7 идей (A–J), из которых
  **A, B, C** — наиболее перспективные следующие шаги. Они не «доделывают» проект, а
  поднимают его к уровню лучших AAC/стримерских TTS-тулов.
