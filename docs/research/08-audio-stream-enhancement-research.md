# Research: архитектура улучшения аудиопотока TTS

**Дата:** 2026-07-13  
**Статус:** research / без немедленной реализации  
**Связано:** [ROADMAP-029](../roadmap/completed/029-dsp-audio-postprocessing.md), [ROADMAP-030](../roadmap/completed/030-deepfilternet-resampling-invariant.md), [ROADMAP-031](../roadmap/active/031-resemble-enhance-research.md), `src-tauri/src/audio/player.rs`

## 1. Вывод

Для текущего TTS-потока самый большой практический эффект ожидается не от super-resolution, а от контроля всего сигнального пути:

```text
provider format
  → decode
  → explicit sample-rate contract
  → boundary cleanup
  → optional stretch
  → EQ / gentle compression / true-peak protection
  → loudness policy
  → PCM playback
```

Resemble Enhance и другие генеративные restoration-модели следует оставить отдельным offline-режимом. Для чистого TTS они могут добавить яркость, но также способны изменить тембр, согласные и индивидуальность голоса.

## 2. Приоритет P0 — исправить основу потока

### 2.1 Явный объект PCM с metadata

Нельзя передавать результат ресемплинга как один `Vec<f32>` с частотой, назначенной позднее. Любой шаг должен возвращать пару или структуру:

```text
AudioPcm { samples, sample_rate, channels }
```

`AudioPcm::duration_secs()` должен использовать именно metadata результата. Это особенно важно для DeepFilterNet: он работает на 48 kHz, а входы провайдеров могут быть 16/22,05/24/44,1 kHz.

### 2.2 Качественный ресемплинг

Встроенный ресемплер DeepFilterNet достаточен для его внутреннего пути, но не обязательно должен стать общим ресемплером приложения. Для общего потокового ресемплинга стоит сравнить `rubato` с текущей реализацией: crate поддерживает sinc-ресемплеры, FFT-ресемплеры, chunked processing и real-time режим без аллокаций внутри критического вызова. [Документация rubato](https://docs.rs/rubato/latest/rubato/)

Рекомендация:

- realtime: заранее созданный sinc-resampler с умеренным качеством;
- offline/export: более длинный sinc-фильтр;
- не использовать простой linear/polynomial interpolation для финального downsample, если важны верхние согласные.

## 3. Приоритет P1 — улучшения, которые слышны чаще всего

### 3.1 Управление громкостью по loudness, а не только peak

Текущие peak-компрессор и limiter защищают от клиппинга, но не гарантируют одинаковую воспринимаемую громкость разных провайдеров и фраз. Для TTS полезнее отдельная политика:

- измерять short-term/relative loudness;
- ограниченно поднимать тихие фразы;
- не усиливать длинные паузы и шумовой пол;
- оставлять safety limiter последним;
- не применять агрессивный auto-gain к каждой короткой фразе.

Для анализа можно использовать EBU R128-совместимый `ebur128`; для потокового измерения существует отдельный zero-allocation `ebur128-stream`. [ebur128](https://docs.rs/ebur128/latest/ebur128/) и [ebur128-stream](https://docs.rs/ebur128-stream/latest/ebur128_stream/)

Для коротких TTS-фраз интегральный LUFS ненадёжен: лучше использовать short-term/моментальный показатель и ограничить correction gain, например ±3 dB. Это даст более стабильный поток, но не должно превращаться в «войну за громкость».

### 3.2 Обработка границ фраз

В текущем playback используется `rodio::buffer::SamplesBuffer`, но в pipeline не обнаружены fade-in/fade-out или zero-crossing correction. На стыках коротких TTS-фраз возможны clicks, особенно после stretch, MP3 decode или обрыва хвоста.

Нужен небольшой boundary-stage:

- убрать DC offset или проверить его перед обработкой;
- делать короткий fade-in на начале фразы только при необходимости;
- делать fade-out на конце только если хвост обрывается неестественно;
- при склейке двух фраз использовать короткий equal-power crossfade, а не механическое наложение;
- не обрезать естественную тишину без VAD/энергетического порога.

Ориентир для fade/crossfade — несколько миллисекунд, с A/B-проверкой на шипящих и взрывных согласных. Это часто полезнее нейросетевого enhancer для realtime-потока.

### 3.3 True-peak protection

Обычный sample-peak limiter не всегда видит inter-sample peaks после EQ и ресемплинга. Для playback достаточно оставить текущий safety limiter, но в export-режиме стоит добавить oversampled peak check, например 2x–4x, и ceiling около -1 dBTP.

Не следует постоянно лимитировать голос на несколько dB: limiter должен быть защитой, а не основным способом сделать TTS громче.

## 4. Приоритет P1/P2 — спектральная обработка

### 4.1 EQ

Уже запланированный native EQ — наиболее безопасный способ подстроить голос:

- low-cut только против DC/низкочастотного мусора;
- небольшой presence boost для разборчивости;
- high-shelf только если исходная частота действительно содержит нужную полосу;
- пресеты `Natural`, `Clear`, `Warm` должны быть мягкими и обратимыми.

High-shelf не восстанавливает отсутствующие частоты. Для 16 kHz-потока boost выше примерно 7–8 kHz преимущественно поднимает край полосы и шум/артефакты.

### 4.2 Динамическая обработка

Мягкий компрессор 1.5:1–2:1 может выровнять фразы, но TTS часто уже имеет компрессированную динамику. Агрессивный compressor делает паузы неестественными и усиливает артефакты MP3. Поэтому compressor должен быть bypass по умолчанию или иметь очень мягкий preset.

Expander/gate добавлять не рекомендуется: он способен обрезать окончания слов и дыхательные элементы.

## 5. Приоритет P2 — нейросетевые варианты

### 5.1 DeepFilterNet

Оставить как opt-in denoise для действительно шумных или пережатых источников. DeepFilterNet официально ориентирован на шумоподавление и 48 kHz; Rust-бинарь и библиотека подходят для realtime, но модель не является bandwidth extension. [Репозиторий DeepFilterNet](https://github.com/Rikorose/DeepFilterNet)

На чистом TTS обязательный запуск DeepFilterNet не оправдан: возможна потеря части согласных.

### 5.2 Resemble Enhance

Оставить offline restoration для WAV/экспорта и специальных сравнений. Официальный pipeline разделяет denoiser и enhancer; enhancer восстанавливает и расширяет полосу генеративным способом, поэтому результат может быть правдоподобным, но не идентичным исходному голосу. [Репозиторий Resemble Enhance](https://github.com/resemble-ai/resemble-enhance)

### 5.3 Real-time RE-USE

Это более свежий исследовательский кандидат: модель заявляет поддержку нескольких sample rate, bandwidth extension, codec artifacts и разных latency-конфигураций. Однако опубликованный checkpoint распространяется по NVIDIA One-Way Noncommercial License и ориентирован на NVIDIA GPU. Для коммерческого desktop-продукта это пока не универсальная замена DeepFilterNet. [Карточка Real-time RE-USE](https://huggingface.co/nvidia/Real-time_RE-USE)

### 5.4 RNNoise/DTLN и лёгкие ONNX-модели

Они могут быть полезны как дешёвый denoise для микрофонного входа, но для чистого TTS не решают главную проблему и могут портить фрикативные согласные. Их не следует добавлять только ради слова «AI enhancement».

## 6. Что даст максимальный эффект именно TTSBard

Предварительный рейтинг:

| Направление | Эффект | Риск | Realtime |
|---|---:|---:|---:|
| Корректный sample-rate contract | высокий | низкий | да |
| Boundary fade/crossfade/de-click | высокий на коротких фразах | низкий | да |
| Loudness policy | высокий для разных провайдеров | средний | да, ограниченно |
| Качественный resampling | средний | низкий | да |
| Мягкий EQ | средний | низкий | да |
| Soft compressor + limiter | средний | средний | да |
| DeepFilterNet | высокий только на шумном входе | средний | да |
| Resemble full enhancer | средний/непредсказуемый | высокий | скорее нет |
| Bandwidth extension другой нейросетью | средний/непредсказуемый | высокий | редко |

## 7. Предлагаемый порядок дальнейших stages

1. Завершить Stage 31: явный sample-rate contract и тесты всех входных частот.
2. Добавить boundary processing и тесты clicks на начале/конце коротких фраз.
3. Завершить DSP stage с мягкими preset-ами.
4. Добавить измерение loudness и ограниченную нормализацию между провайдерами.
5. Сравнить текущий ресемплер с `rubato` на sweep, voice consonants и длительности.
6. Только после этого сделать offline A/B набор для Resemble Enhance и других restoration-моделей.

## 8. Критерий оценки

Любое улучшение принимать только после сравнения `original / DSP / DeepFilterNet / neural restoration` на одинаковых PCM и metadata. Нужны:

- субъективное прослушивание на русской речи;
- отдельные тесты шипящих, взрывных согласных и окончаний слов;
- измерения peak, true peak, loudness, duration и CPU time;
- проверки коротких фраз и MP3-источников;
- контроль сохранения текста, дикции и индивидуального тембра.

Если обработка делает звук «ярче», но слушатель не предпочитает её исходнику вслепую, считать её улучшением нельзя.
