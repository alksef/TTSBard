# Research: интеграция Stream Deck с TTSBard

**Дата:** 2026-07-18
**Статус:** research note
**Связано:** `src-tauri/src/commands/mod.rs`, `src-tauri/src/commands/playback.rs`, `src/components/InputPanel.vue`, `src/components/InterceptPanel.vue`

---

## Вывод

Интеграция со Stream Deck оправдана как опциональная функция для стримеров и пользователей, которым удобнее физические программируемые кнопки. Она не должна быть условием использования приложения: для первого релиза достаточно полноценных глобальных горячих клавиш и пресетов фраз. Нативный плагин следует делать только после проверки спроса на расширенное управление и отображение состояния на кнопках.

## Что уже есть в приложении

TTSBard уже имеет основу для управления без фокуса на основном окне:

- глобальные горячие клавиши через `tauri-plugin-global-shortcut`;
- команда озвучивания `speak_text` и ввод текста в `InputPanel`;
- команды управления воспроизведением: пауза, продолжение и остановка;
- soundboard, выбор устройства и регулирование громкости;
- интеграции/панели для Twitch, Telegram и перехвата текста.

Следовательно, Stream Deck не требует отдельного пути к TTS-движку: ему нужен удобный слой команд над уже существующими действиями.

## Пользовательские сценарии

| Приоритет | Действие на Stream Deck | Польза |
| --- | --- | --- |
| P0 | Озвучить заранее заданную фразу | Быстрые ответы без переключения в окно приложения. |
| P0 | Пауза / продолжение / остановка | Немедленный контроль текущей реплики в эфире. |
| P0 | Открыть окно ввода | Ввод новой фразы без поиска окна в панели задач. |
| P1 | Переключить голос, язык или профиль | Разные роли и сцены стрима. |
| P1 | Запустить звук из soundboard | Единое физическое управление речью и эффектами. |
| P1 | Переключить выходное устройство или mute | Быстрая защита от случайного вывода звука в эфир. |
| P2 | Показать на кнопке «говорит», «пауза», ошибку | Контроль состояния без взгляда на приложение. |
| P2 | Отправить произвольный текст из поля действия | Гибкие макросы и персональные шаблоны. |

## Варианты реализации

### Вариант A: только горячие клавиши — рекомендуемый MVP

Пользователь назначает действия TTSBard на глобальные хоткеи, а в приложении Stream Deck связывает кнопку с соответствующей комбинацией клавиш.

Преимущества:

- минимальная стоимость разработки и поддержки;
- не нужен отдельный плагин, SDK и процесс установки;
- работает с обычными Stream Deck-профилями и может быть документирован в справке;
- покрывает P0-сценарии.

Ограничения: нет динамического статуса на кнопках, передачи произвольного текста и понятной настройки пресетов непосредственно в Stream Deck.

### Вариант B: официальный плагин Stream Deck — следующий этап

Отдельный плагин предоставляет действия TTSBard и property inspector для настройки текста, голоса и режима воспроизведения. Он связывается с запущенным приложением через локальный API.

Преимущества: хорошие настройки, состояния и иконки на кнопках; не нужно расходовать глобальные сочетания клавиш; можно передавать данные действия напрямую.

Стоимость: требуется сопровождать JavaScript-плагин, совместимость с Stream Deck SDK, установку/обновления и ситуацию, когда TTSBard не запущен.

### Вариант C: общий локальный API без плагина

Приложение публикует loopback HTTP/WebSocket API с аутентификацией. Его могут использовать Stream Deck, OBS-скрипты и другие локальные автоматизации.

Это полезный архитектурный фундамент, но само по себе не является хорошим пользовательским интерфейсом. Внешние пользователи будут зависеть от непубличного API, поэтому вариант разумно делать вместе с вариантом B, а не вместо него.

## Рекомендуемая последовательность

1. Проверить и оформить стабильные глобальные действия: озвучить фразу, пауза, продолжение, остановка, открыть окно.
2. Добавить в справку готовый профиль/таблицу назначения кнопок Stream Deck.
3. Собрать обратную связь: нужны ли статусы, пресеты и ввод текста прямо с устройства.
4. При подтверждённом спросе спроектировать versioned loopback API и выпустить нативный плагин.

## Эскиз API для нативного плагина

Транспорт: HTTP для команд и WebSocket для событий; слушать исключительно `127.0.0.1`, использовать случайный токен с локальным хранением. Не открывать порт в локальную сеть.

Пример команд:

```text
POST /v1/speech       { text, voice_id?, interrupt? }
POST /v1/playback/pause
POST /v1/playback/resume
POST /v1/playback/stop
POST /v1/soundboard/play { sound_id }
GET  /v1/status
```

События: `speech_started`, `speech_finished`, `playback_paused`, `voice_changed`, `error`. Они позволят обновлять title и icon кнопки без опроса API.

## Риски и критерии качества

- Не заменять существующие горячие клавиши: Stream Deck должен быть дополнительным способом управления.
- Команда остановки обязана быть идемпотентной и срабатывать быстро, даже при ошибке синтеза.
- Плагин должен ясно показывать состояние «TTSBard не запущен», а не молча терять действие.
- Локальный API не должен принимать соединения извне и не должен содержать неизменяемый токен в плагине.
- Не включать интеграцию в критический путь запуска приложения: отсутствие Stream Deck не должно влиять на TTS.

## Решение

Сейчас: подготовить качественный сценарий через глобальные хоткеи и пресеты фраз.
Позже: выпускать нативный плагин только при подтверждённой потребности в динамическом статусе, прямом вводе текста или управлении профилями.
