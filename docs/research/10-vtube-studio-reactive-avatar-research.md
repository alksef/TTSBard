# Research: VTube Studio как реактивный аватар для TTSBard

**Дата:** 2026-07-17  
**Статус:** research note / развитие интеграции  
**Связано:** `docs/research/02-vtuber-studio-integration-research.md`, `src-tauri/src/events.rs`, `src-tauri/src/playback.rs`, `src/components/InputPanel.vue`

---

## 1. Цель

Исследование `02-vtuber-studio-integration-research.md` описывает прямой lip-sync:
TTSBard анализирует амплитуду собственного TTS-аудио и передаёт `MouthOpen` в
VTube Studio (VTS) через локальный WebSocket API.

Эту интеграцию можно расширить до реактивного поведения аватара:

- показывать, что пользователь набирает текст;
- различать набор, обработку текста, ожидание в очереди и воспроизведение;
- запускать выражения и анимации по событиям TTSBard;
- связывать отдельные звуки и типы фраз с реакциями;
- при необходимости показывать предметы, цветовые эффекты и VFX.

Главная продуктовая цель — убрать паузу, в которой молчащий пользователь печатает,
а аватар выглядит неактивным. Зритель должен понимать состояние коммуникации до
начала озвучивания фразы.

---

## 2. Нужен ли отдельный плагин

### Решение

Отдельный DLL-модуль или устанавливаемое расширение внутри VTube Studio не нужно.
TTSBard сам является внешним VTS API-плагином:

1. VTube Studio включает локальный WebSocket API.
2. TTSBard подключается к `ws://localhost:8001` или выбранному порту.
3. При первом подключении VTS показывает запрос разрешения.
4. Полученный токен используется при последующих подключениях.
5. TTSBard отображается в списке разрешённых/активных плагинов VTS.

Пользователю достаточно установить TTSBard, включить Plugin API в VTube Studio и
один раз подтвердить доступ. Отдельный установщик интеграции только усложнит
дистрибуцию.

API может работать на другом порту, особенно при запуске нескольких экземпляров
VTube Studio. Помимо ручного ввода порта, VTS поддерживает обнаружение экземпляров
через UDP broadcast на порту `47779`. Автообнаружение полезно как последующее
улучшение, но ручной порт `8001` достаточен для MVP.

## 3. Какие механизмы предоставляет VTube Studio

### 3.1. Инъекция параметров

TTSBard может передавать стандартные и созданные плагином параметры:

- `MouthOpen`;
- `TTSBardTyping`;
- `TTSBardSpeaking`;
- `TTSBardThinking`;
- `TTSBardQueue`;
- `TTSBardEnergy`;
- `TTSBardError`.

Пользователь связывает custom parameter с нужным Live2D output-параметром в
настройках модели VTube Studio. Для входа можно настраивать диапазон и smoothing.

Параметры подходят для непрерывных или долгоживущих состояний. Например,
`MouthOpen` меняется 30–50 раз в секунду, а `TTSBardTyping` остаётся равным `1`
до завершения набора.

Ограничения:

- эффект custom parameter требует поддержки со стороны рига модели или ручной
  привязки к существующему Live2D-параметру;
- один параметр в режиме `set` может контролироваться только одним API-плагином;
- если прекратить отправлять параметр более чем на секунду, VTS возвращает
  управление обычному tracking source;
- необязательный `weight` позволяет смешивать значение плагина с face tracking;
- режим `add` можно использовать для кратковременной добавочной реакции, не
  перехватывая параметр полностью.

### 3.2. Запуск VTS hotkey

API позволяет получить список hotkey текущей модели и запустить hotkey по ID или
имени. Через заранее настроенные в VTS hotkey можно:

- включать и выключать expressions;
- проигрывать `.motion3.json`;
- менять idle animation;
- перемещать модель;
- включать костюмы и аксессуары;
- выполнять действия Live2D Items.

Hotkey удобны для дискретных реакций: «удивиться», «засмеяться», «подпрыгнуть».
Они не подходят для lip-sync и других параметров, которые нужно обновлять
непрерывно.

Нужно учитывать, что expression hotkey часто работает как toggle. Повторная
отправка может выключить выражение, а изменение состояния пользователем или
другим плагином способно нарушить предположение TTSBard. Для парных действий
`start/finish` следует проверять состояние expression либо предоставлять
раздельные пользовательские привязки «включить» и «сбросить».

Hotkey следует хранить по уникальному ID, а имя использовать для отображения и
восстановления привязки. Имена могут повторяться, а набор hotkey зависит от
загруженной модели.

### 3.3. Items

VTS API умеет загружать, перемещать и удалять PNG, JPG, GIF, animation folder и
Live2D Items. Это даёт реакции, не требующие специального параметра модели:

- пузырь `…` во время набора;
- клавиатура или телефон;
- микрофон во время речи;
- знак ошибки;
- сердечки, вопросительный и восклицательный знаки.

Items — наиболее универсальный fallback для моделей без специального рига.
Недостатки: нужны подготовленные ассеты, корректная позиция относительно каждой
модели и аккуратное управление уже загруженными предметами.

### 3.4. Tint и post-processing

API позволяет:

- менять tint всей модели или выбранных ArtMesh;
- включать post-processing presets;
- настраивать bloom, chromatic aberration, weather и другие VFX.

Это подходит для редких сильных реакций, но не для постоянного обновления.
Post-processing не рассчитан на изменения каждый кадр; для плавных переходов
нужно пользоваться встроенным fade time. Эффекты также могут конфликтовать с
визуальным оформлением пользователя, поэтому они должны быть выключены по
умолчанию.

## 4. Предлагаемая модель состояний TTSBard

### 4.1. Typing

`Typing` должен отражать реальный набор текста, а не просто наличие текста в
редакторе.

Предлагаемое поведение:

1. При первом изменении текста отправить `typing = true`.
2. На последующих изменениях только перезапускать локальный таймер.
3. После `700–1000` мс без изменений отправить `typing = false`.
4. При отправке, очистке редактора, потере контекста окна или закрытии приложения
   немедленно сбросить `typing = false`.
5. Не отправлять WebSocket-запрос на каждую клавишу.

Событие возникает во Vue-редакторе, но долговременное WebSocket-соединение должно
оставаться на Rust-бэкенде. Frontend должен передавать бэкенду только изменение
логического состояния.

Варианты визуализации:

- custom parameter включает typing motion;
- hotkey включает заранее подготовленную expression/motion;
- Item показывает `…` или анимированную клавиатуру.

Для первой версии предпочтителен custom parameter, а Item можно добавить как
универсальный профиль позже.

### 4.2. Thinking

Состояние активно во время:

- `correct_text`;
- `get_ai_completion`;
- проверки грамматики;
- потенциальных будущих AI-операций над текстом.

В текущем `InputPanel.vue` уже существуют локальные флаги `isCorrecting`,
`isCompleting` и `isCheckingGrammar`. Их можно свести в одно состояние
`thinking = true/false`.

Визуально это может быть отдельная задумчивая expression, взгляд вверх или Item.
`Thinking` не следует автоматически считать `Typing`.

### 4.3. Speaking и lip-sync

`Speaking` включается по фактическому `PlaybackStarted`, а не по отправке текста
в TTS. Выключается по:

- `PlaybackFinished`;
- `PlaybackStopped`;
- ошибке воспроизведения.

При `PlaybackPaused` рот должен закрываться, но отдельная speaking expression
может либо остаться активной, либо перейти в пользовательское состояние Pause.

Внутри `Speaking` параметр `MouthOpen` управляется реальной амплитудой PCM:

```text
PCM playback
  -> короткое окно RMS/envelope
  -> noise gate
  -> gain/curve
  -> attack/release smoothing
  -> MouthOpen 0..1
```

На завершении нужно явно отправить `MouthOpen = 0`, затем прекратить его обновлять,
чтобы VTS вернул управление обычному face tracking.

Дополнительный `TTSBardEnergy` можно вычислять из более медленно сглаженной
амплитуды. Он пригодится для покачивания тела, свечения аксессуара или интенсивности
анимации, не дублируя быстрые движения рта.

### 4.4. Queue и ожидание

В TTSBard уже есть `QueueChanged`. В интеграцию можно передавать:

- `queue_empty: bool`;
- `queue_size: integer`;
- нормализованный `TTSBardQueue` в диапазоне `0..1`.

Однако визуальная ценность размера очереди ниже, чем у Typing и Speaking.
Для первой версии достаточно реакции на переходы:

- очередь стала непустой;
- очередь снова пуста.

### 4.5. Error

Короткая реакция может запускаться при:

- ошибке синтеза;
- ошибке AI-коррекции;
- ошибке playback;
- потере подключения к VTS.

Ошибка подключения к самому VTS не должна пытаться вызвать VTS-реакцию. Она
отображается только в UI TTSBard.

Реакции на рабочие ошибки должны иметь cooldown, чтобы серия повторных ошибок не
заполняла очередь hotkey.

## 5. Автоматические эмоции и реакции на текст

### 5.1. Надёжный первый уровень

Начинать следует с детерминированных правил:

- `?` в конце — вопрос/наклон головы;
- `!` или несколько `!` — удивление/энергичная motion;
- `ахаха`, `лол`, `😂` — смех;
- `❤️`, `спасибо` — положительная реакция;
- пользовательские слова и регулярные выражения;
- ручной выбор reaction preset перед отправкой.

Правило выбирает реакцию, но запускается только при `PlaybackStarted`, чтобы
expression совпала с реальным началом речи.

### 5.2. Ручные пресеты

Рядом с действием «Озвучить» можно выбрать:

- Обычная;
- Радость;
- Грусть;
- Злость;
- Удивление;
- Смех;
- Шёпот;
- Крик;
- Без реакции.

Каждый пресет может содержать:

- start hotkey;
- finish/reset hotkey;
- коэффициент `MouthOpen`;
- значение `TTSBardEnergy`;
- необязательный Item;
- необязательный tint/VFX preset.

Ручной выбор предсказуем и должен иметь приоритет над автоматическими правилами.

### 5.3. AI-классификация

AI может классифицировать готовую фразу по небольшому закрытому набору эмоций.
Это следует рассматривать как позднюю опцию:

- возможны неуместные реакции;
- появляется дополнительная задержка;
- результат сложнее объяснить пользователю;
- нельзя блокировать отправку срочной реплики ожиданием классификации.

Если функция появится, она должна иметь короткий timeout, fallback `Обычная` и
возможность полного отключения.

## 6. Универсальная система event-to-action

Вместо жёсткого набора реакций полезно заложить таблицу правил:

| Событие TTSBard | Возможное действие VTS |
|---|---|
| `TypingStarted` | параметр `TTSBardTyping = 1` |
| `TypingStopped` | параметр `TTSBardTyping = 0` |
| `ThinkingStarted` | thinking hotkey или parameter |
| `PlaybackStarted` | speaking parameter + reaction hotkey |
| PCM envelope | `MouthOpen`, `TTSBardEnergy` |
| `PlaybackPaused` | закрыть рот, pause hotkey |
| `PlaybackFinished` | reset hotkey, speaking `0` |
| очередь стала непустой | waiting parameter/hotkey |
| TTS error | error hotkey с cooldown |
| soundboard binding | назначенный hotkey/Item |

Поддерживаемые action-типы:

1. `SetParameter`;
2. `PulseParameter`;
3. `TriggerHotkey`;
4. `LoadItem`;
5. `UnloadItem`;
6. `SetTint`;
7. `SetPostProcessingPreset`.

В MVP UI не обязан показывать универсальный редактор правил. Внутренняя модель
может быть общей, а интерфейс — состоять из понятных полей «При начале речи»,
«При ошибке», «Во время набора».

## 7. Профили моделей

Реакции нельзя считать одинаковыми для всех Live2D-моделей:

- отличаются hotkey ID;
- отличаются доступные expressions и motions;
- custom parameters могут быть привязаны к разным output;
- Item требует другой позиции и масштаба;
- часть моделей не имеет параметров для рук или typing animation.

Настройки следует хранить по `modelID`:

```json
{
  "vtube_studio": {
    "enabled": true,
    "port": 8001,
    "lip_sync": {
      "enabled": true,
      "gain": 1.5,
      "gate": 0.02,
      "attack_ms": 35,
      "release_ms": 90,
      "tracking_weight": 1.0
    },
    "models": {
      "<model-id>": {
        "typing_mode": "parameter",
        "typing_parameter": "TTSBardTyping",
        "thinking_hotkey_id": null,
        "speech_start_hotkey_id": null,
        "speech_finish_hotkey_id": null,
        "error_hotkey_id": null
      }
    }
  }
}
```

Это ориентировочная схема, а не обязательный контракт реализации.

При смене модели VTS-интеграция должна:

1. определить новый `modelID`;
2. загрузить соответствующий профиль;
3. проверить существование сохранённых hotkey;
4. пересоздать принадлежащие TTSBard custom parameters;
5. показать в UI отсутствующие привязки, не запуская случайные fallback-hotkey.

## 8. Архитектура в TTSBard

### Backend

Один `VtsManager` на Rust-бэкенде отвечает за:

- подключение и авторизацию;
- reconnect с backoff;
- определение текущей модели;
- очередь команд с приоритетами;
- высокочастотную отправку lip-sync;
- низкочастотные параметры состояния;
- выполнение hotkey/Item/VFX actions;
- сброс управляемых состояний при stop/disconnect;
- диагностику соединения.

WebSocket не следует держать во Vue: главное окно может быть скрыто, а playback и
реакции должны продолжать работать.

Высокочастотный lip-sync желательно отделить от обычной очереди действий, чтобы
медленный ответ на Item или hotkey не создавал ступенчатое движение рта.

### Источники событий

Текущие backend-события уже покрывают большую часть жизненного цикла:

- `PlaybackStarted`;
- `PlaybackFinished`;
- `PlaybackPaused`;
- `PlaybackResumed`;
- `PlaybackStopped`;
- `QueueChanged`;
- `TtsError`.

Нужно добавить или провести до backend логические состояния:

- `TypingChanged(bool)`;
- `ThinkingChanged(bool)`;
- выбранный reaction preset для конкретной фразы;
- при необходимости событие soundboard с ID binding.

При добавлении подписчика нельзя создавать второго конкурирующего consumer для
существующего `mpsc`-канала событий. VTS должен получать события через
централизованный event loop, broadcast или отдельный явный вызов менеджера.

### Приоритеты команд

1. `Stop/reset` и закрытие рта;
2. lip-sync parameter update;
3. playback start/finish actions;
4. typing/thinking state;
5. декоративные Item/tint/VFX.

Декоративные реакции можно пропустить при перегрузке. Сброс рта и состояний
пропускать нельзя.

## 9. UX настроек

Раздел `Настройки -> Интеграции -> VTube Studio`:

### Подключение

- переключатель «Включить интеграцию»;
- статус: выключено / подключение / ожидается разрешение / подключено / ошибка;
- найденный экземпляр и текущая модель;
- порт;
- кнопки «Подключить», «Повторить авторизацию», «Проверить».

### Lip-sync

- включено;
- чувствительность/gain;
- noise gate;
- attack/release;
- смешивание с tracking;
- индикатор текущего уровня;
- кнопка теста движения рта.

### Состояния

- набор текста;
- обработка текста;
- начало/конец речи;
- пауза;
- ошибка;
- очередь.

Для каждого состояния пользователь выбирает:

- ничего;
- custom parameter;
- VTS hotkey;
- позднее — Item.

Список hotkey загружается из текущей модели, поэтому пользователю не нужно вручную
копировать ID.

### Диагностика

Нужно различать:

- VTube Studio не запущен;
- Plugin API выключен;
- неверный порт;
- разрешение отклонено или отозвано;
- модель не загружена;
- hotkey больше не существует;
- параметр занят другим API-плагином;
- действие временно запрещено открытым окном VTS.

## 10. Приоритеты внедрения

### P0 — базовое соединение

- авторизация и сохранение токена;
- reconnect;
- статус подключения;
- определение текущей модели;
- получение списка hotkey;
- тестовая команда.

### P1 — основная пользовательская ценность

- lip-sync по амплитуде PCM;
- `Speaking`;
- `Typing` с debounce;
- `Thinking`;
- настраиваемые start/finish/error hotkey;
- профиль по `modelID`;
- гарантированный reset при stop/disconnect.

### P2 — реакции

- ручные reaction presets;
- правила по пунктуации, словам и emoji;
- привязка soundboard-кнопок к hotkey;
- queue/waiting reaction;
- cooldown и защита от повторов.

### P3 — визуальные расширения

- Items для `…`, клавиатуры, телефона и ошибок;
- tint;
- post-processing presets;
- генерация speech bubble;
- опциональная AI-классификация эмоций.

## 11. Рекомендуемый MVP

Первый релиз интеграции должен включать:

1. подключение TTSBard как VTS API-плагина без отдельной установки;
2. lip-sync по реальному воспроизводимому PCM;
3. параметры `TTSBardTyping`, `TTSBardSpeaking`, `TTSBardThinking`;
4. debounce-индикацию набора текста;
5. выбор hotkey для начала речи, завершения речи и ошибки;
6. настройки чувствительности и сглаживания рта;
7. отдельный профиль текущей модели;
8. кнопки тестирования каждого действия;
9. понятную диагностику подключения и отсутствующих привязок.

Items, VFX и AI-эмоции не нужны для доказательства ценности первой версии.

---

## 12. Критерии успешности

- VTS требует разрешение только при первом подключении или после его отзыва.
- После перезапуска VTS/TTSBard соединение восстанавливается автоматически.
- Во время набора реакция появляется не более чем через `150` мс и исчезает после
  заданного периода бездействия.
- Набор не создаёт WebSocket-запрос на каждую клавишу.
- Рот следует паузам и громкости фактического TTS-аудио.
- Pause, Stop, Finish, ошибка и disconnect всегда закрывают рот.
- Обычный face tracking возвращает управление после окончания TTS.
- Смена модели не запускает hotkey от старого профиля.
- Отсутствующая expression/hotkey не ломает playback.
- При выключенной интеграции TTS и playback работают без изменения поведения.

## 13. Риски

1. **Конфликт параметров.** Другой плагин может уже управлять `MouthOpen`.
2. **Toggle-семантика expressions.** Повторный hotkey способен выключить
   выражение вместо включения.
3. **Различия моделей.** Универсальной typing-анимации не существует.
4. **Залипшее состояние.** При аварийном завершении нужно сбрасывать параметры и
   не оставлять expression активной.
5. **Перегрузка запросами.** Typing и декоративные эффекты нельзя обновлять с
   частотой lip-sync.
6. **Ложные эмоции.** Автоматические правила и особенно AI могут выбрать
   неуместную реакцию.
7. **Конфликт с пользователем.** Tint/VFX/Items способны менять вручную
   настроенную сцену, поэтому должны быть opt-in.
8. **Несколько экземпляров VTS.** Фиксированный `8001` может подключить не тот
   экземпляр; позднее желательно UDP discovery и выбор instance.

---

## 14. Вывод

Интеграцию стоит строить не как одиночную функцию lip-sync, а как небольшой
event-to-action слой между TTSBard и VTube Studio.

Самая сильная комбинация для первой версии:

```text
Typing -> Thinking -> Speaking + amplitude lip-sync -> Idle
```

Она закрывает весь период коммуникации: зритель видит, что пользователь печатает,
понимает, когда текст обрабатывается, затем получает синхронную речь и возврат
аватара в обычное состояние.

Реакции по hotkey дают высокую ценность при небольшой сложности, а Items, tint,
VFX и AI-классификацию разумно развивать после стабилизации соединения,
профилей моделей и жизненного цикла состояний.
