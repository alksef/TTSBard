# Предложения по улучшению аудиопотока TTSBard

**Дата:** 2026-07-13  
**Статус:** research / предложение к обсуждению  
**Основано на:** [08-audio-stream-enhancement-research.md](08-audio-stream-enhancement-research.md), `src-tauri/src/audio/effects.rs`, `src-tauri/src/audio/player.rs`

## 1. Моя рекомендация

Не строить основной realtime-поток вокруг нейросетевого enhancement. Для TTS источник обычно уже является чистой речью, поэтому восстановление и генерация новых спектральных деталей дают меньшую пользу, чем правильная работа с громкостью, границами фраз и sample rate.

Предлагаемый основной поток:

```text
TTS bytes
  → decode to AudioPcm
  → canonical sample-rate policy
  → boundary cleanup
  → optional DeepFilterNet only for noisy sources
  → Signalsmith stretch
  → gentle EQ
  → optional mild compressor
  → true-peak safety limiter
  → playback
```

Loudness measurement следует использовать для выбора gain, но не вставлять тяжёлую глобальную нормализацию без ограничения: короткие фразы слишком нестабильны для полноценного integrated LUFS.

## 2. Предложение A — три профиля вместо одного «Enhance»

Текущая семантика `enhance` смешивает несколько разных задач. Лучше разделить пользовательские профили:

### Natural

Для чистого TTS по умолчанию:

- DeepFilterNet выключен;
- EQ выключен или очень мягкий;
- compressor выключен;
- safety limiter включён с большим запасом;
- применяются только format/boundary invariants.

Это должен быть эталонный режим для A/B-сравнения.

### Clear

Для слегка глухого голоса:

- мягкий low-cut против низкочастотного мусора;
- небольшой presence boost в области разборчивости;
- compressor около 1.5:1–2:1;
- ограниченный makeup gain;
- финальный limiter.

Профиль не должен пытаться «дорисовывать» частоты, которых нет в исходном сигнале.

### Cleanup

Для MP3/OGG, шумных или явно деградировавших источников:

- DeepFilterNet opt-in;
- мягкое подавление, чтобы не съедать согласные;
- тот же DSP после denoise;
- видимый индикатор, что включена потенциально изменяющая голос обработка.

Resemble Enhance не включать в эти realtime-профили. Для него нужен отдельный offline-профиль `Restoration`.

## 3. Предложение B — сначала стабилизировать формат

`AudioPcm` должен быть единственным носителем samples, `sample_rate` и `channels` на всём backend-пути. Каждый преобразователь должен принимать и возвращать `AudioPcm`, а не отдельные `Vec<f32>` и числовые параметры.

Нужно выбрать канонический режим:

- для playback с DeepFilterNet — 48 kHz;
- для playback без DeepFilterNet — либо исходный rate, либо единый 48 kHz, но одинаково для всех провайдеров;
- для export — отдельная политика `preserve_source_rate`/`output_rate`.

Моя рекомендация — единый 48 kHz для playback после decode, независимо от того, включён ли DeepFilterNet. Тогда `Stretch`, DSP и `SamplesBuffer` всегда работают с одинаковым контрактом. Для экспорта исходная частота может сохраняться отдельно.

Переход к 48 kHz должен выполняться одним явно выбранным качественным ресемплером. Для streaming-кандидата стоит сравнить текущий resampler с [`rubato`](https://docs.rs/rubato/latest/rubato/), особенно на 16→48, 22,05→48, 24→48 и 44,1→48 kHz.

## 4. Предложение C — обработать границы фраз

Это, вероятно, самое недооценённое улучшение для текущего приложения. TTS-фразы короткие и последовательно добавляются в очередь. Даже хороший DSP не исправит click, если фраза начинается или заканчивается на ненулевом скачке.

Нужен отдельный boundary processor:

1. оценить DC offset;
2. найти безопасную точку около начала и конца;
3. применить короткий fade-in/fade-out только при наличии скачка;
4. при явной склейке двух PCM-буферов использовать equal-power crossfade;
5. не удалять естественные паузы только по фиксированной длине.

Обработка должна быть условной. Безусловный fade на каждой фразе может съедать взрывные согласные и первые/последние фонемы.

Минимальные тесты:

- фраза с ненулевым первым sample;
- фраза с обрывом на пике;
- короткая фраза меньше 100 ms;
- две фразы с разными уровнями;
- проверка, что normal phrase без discontinuity остаётся практически неизменной.

## 5. Предложение D — loudness policy вместо агрессивного auto-gain

Разные провайдеры могут выдавать заметно разный уровень. Пользователь воспринимает это как нестабильность качества, хотя спектр может быть нормальным.

Предлагаю:

- измерять peak и short-term loudness после DSP;
- вычислять correction gain только для фразы выше минимальной длительности;
- ограничить correction диапазоном примерно ±3 dB;
- не поднимать gain, если фраза состоит почти полностью из тишины;
- применять limiter после correction;
- сохранять в debug-метриках input peak, output peak, gain и limiter reduction.

`ebur128` можно использовать как reference analyzer, но для коротких фраз нельзя считать integrated LUFS единственной истиной. Нужен A/B-тест на реальных фразах разных провайдеров.

## 6. Предложение E — отделить realtime и offline

Один и тот же эффект не обязан работать одинаково в двух режимах.

### Realtime playback

- отсутствие subprocess/Python;
- bounded latency;
- отсутствие генеративного восстановления;
- предсказуемый fallback;
- малый DSP budget на фразу.

### Offline export

- можно запускать Resemble Enhance или другую restoration-модель;
- можно использовать более длинный sinc-resampler;
- можно измерять integrated loudness и true peak по всей дорожке;
- можно показать пользователю отдельный preview и сохранить original.

Такой split позволяет экспериментировать с Resemble, не ухудшая обычное чтение текста.

## 7. Что я бы не добавлял сейчас

- обязательный Resemble Enhance после каждого TTS;
- RNNoise на чистом голосе;
- gate/expander, который может отрезать хвосты слов;
- сильный high-shelf для 16 kHz-записей;
- постоянную компрессию с gain reduction больше 3–4 dB;
- динамическое изменение sample rate между соседними фразами;
- несколько последовательно применённых denoiser/enhancer-моделей.

## 8. Приоритет реализации

### P0

1. Зафиксировать единый playback sample-rate contract.
2. Пропустить все преобразователи через `AudioPcm` с metadata.
3. Добавить тесты длительности, rate и channel layout.

### P1

1. Boundary processor и защита от clicks.
2. Мягкие DSP-профили `Natural` и `Clear`.
3. Диагностические метрики peak/limiter reduction.

### P2

1. Ограниченная loudness policy.
2. Сравнение resamplers.
3. Offline `Restoration` с Resemble Enhance.

### P3

1. Сравнение современных neural speech-enhancement моделей.
2. ONNX/in-process интеграция только при наличии измеримого выигрыша.

## 9. Критерий, что улучшение действительно полезно

Новый этап принимать только если он одновременно:

- улучшает blind A/B предпочтение на русских TTS-фразах;
- не ухудшает согласные и окончания слов;
- не меняет длину без явной причины;
- не поднимает latency выше установленного бюджета;
- сохраняет fallback без потери воспроизведения;
- не требует тяжёлой runtime-зависимости для обычного playback.

Главная метрика — не более яркий спектр и не более высокий peak, а предпочтение слушателя при сохранении дикции и тембра.
