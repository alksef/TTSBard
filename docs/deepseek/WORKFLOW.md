# Workflow: Claude ↔ DeepSeek (реализационный цикл)

> **Кому читать:** любому агенту (Claude или другому), работающему над этим репо.
> По `CLAUDE.md`: **Claude = план/ревью, DeepSeek = пишет код.** Ниже — как цикл
> запускается автоматически, без ручных кликов пользователя.

**Дата фиксации:** 2026-06-29
**Проверено:** OpenCode 1.2.26, провайдер DeepSeek настроен.

---

## Суть

Цикл «Claude ставит задачу → DeepSeek пишет код → Claude ревьюит дифф → при
замечаниях новая задача». DeepSeek запускается **напрямую из Claude через
Bash** (через `opencode run`), поэтому итерации автоматические.

## Каталоги

- `docs/deepseek/plan/`    — крупные планы фич (один файл на фичу, напр. `74-...`).
- `docs/deepseek/tasks/`   — **итерационные task-файлы** (точечные фиксы по ревью),
                              формат `NN-roundM-iterK.md` (напр. `74-round3-01.md`).
- `docs/deepseek/reviews/` — ревью (Round 1/2/3...) + verdict-отчёты по итерациям.

## Запуск DeepSeek (подтверждено рабочим)

```bash
opencode run \
  --model deepseek/deepseek-v4-pro \
  --dir /d/RustProjects/app-tts-v2 \
  --log-level ERROR \
  "$(cat docs/deepseek/tasks/74-round3-01.md)"
```

- Полностью **неинтерактивный**: авто-апрув чтения/правки файлов, выходит по
  завершении, exit 0 на успехе. **НЕ виснет** на запросах разрешений.
- Читает и **реально правит** файлы репо (Edit-инструмент внутри OpenCode).
- Модели: `deepseek/deepseek-chat` (быстро/дёшево), `deepseek/deepseek-v4-pro`
  (сильный кодинг), `deepseek/deepseek-reasoner` (пошаговое мышление).
  Список моделей: `opencode models deepseek`.
- Агент: `build` (primary, полные разрешения).
- Промпт = содержимое task-файла (`"$(cat ...)"`).

> **Почему не MCP / не ACP:** OpenCode умеет быть MCP-*клиентом*, но не
> MCP-сервером; `opencode serve`/`acp` — это собственный HTTP/ACP-сервер OpenCode,
> к которому у Claude нет клиента. Единственный путь — `opencode run` через Bash.

## Итерация цикла (один прогон)

1. **Claude пишет task-файл** `tasks/NN-roundM-iterK.md`: точные фиксы с
   файлами/строками/ожидаемым поведением. Без воды, без готового кода (DeepSeek
   пишет сам), но с конкретными условиями и критериями.
2. **Claude зовёт** `opencode run` с task-файлом как промптом → DeepSeek правит код.
3. **Claude проверяет:** `cargo check` + `npx vue-tsc --noEmit` (0/0),
   затем **код-ревью свежего диффа vs task-файл**.
4. **Verdict:** `APPROVED` → отчёт в `reviews/`; либо `CHANGES REQUESTED` →
   новый task-файл `...-iter(K+1).md` и повтор с шага 2.
5. **Лимит:** обычно ≤5 итераций на раунд; если не сходится — эскалация к человеку.

## Главное правило ревью

**НЕ доверять чек-боксам DeepSeek.** В плане 74 DeepSeek отмечал пункты `[x]`
выполненными, а реальная проверка находила регрессии каждый раз (см.
`reviews/review-plan-74-round2-...`). `[x]` = «добавил код», ≠ «поведение
работает end-to-end». Всегда: сборка + чтение диффа + трассировка путей кода.

## Что НЕ делает DeepSeek (делает Claude)

- Планы фич (`plan/`), архитектурные решения.
- Ревью, вердикты, новые task-файлы по итогам.
- Запуск сборки/проверок.
- Коммиты — только по явной просьбе пользователя.
