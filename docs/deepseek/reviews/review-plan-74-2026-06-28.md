## Критичные (обязательно)

### C1. ПОТЕРЯ DUAL OUTPUT — главная регрессия
`commands/mod.rs:215-243` всё ещё вычисляет `speaker_config` и `virtual_mic_config`, но
`commands/mod.rs:250` вызывает `pb.enqueue(phrase_id, text, audio_data)` **без них**.
`playback.rs:98` открывает только `host.default_output_device()` через один `OutputStream`.
→ **Виртуальный микрофон (VB-Cable/VoiceMeeter) перестанет получать звук.**
**Фикс:** PlaybackManager должен принимать конфиг вывода (оба устройства) и клонировать
`Decoder`-источник в оба sink'а — либо **обернуть существующий `AudioPlayer::play_mp3_async_dual`**
вместо собственного `OutputStream`. Предпочтительно второе (переиспользование проверенного кода).

### C2. Pause/Resume на «мёртвом» Sink + рассинхрон статуса
`playback.rs:114-182`:
- После `playback-finished` (`sink=None`) команда `Pause` молча ничего не делает, но
  `PlaybackManager::pause()` уже выставил `status=Paused` в общем состоянии → рассогласование
  (статус Paused, по факту ничего не играет, resume ничего не сделает).
- `sink.empty()` (строка ~176) проверяется каждые 100мс без проверки `!is_paused()` → у коротких
  фраз при паузе буфер может быть пустым → ложный `playback-finished` во время паузы.
**Фикс:** (a) не менять статус на Paused, если `sink=None`; (b) перед детекцией завершения
проверять `!s.is_paused()`.

### C3. Очередь НЕ движется автоматически (зависает после первой фразы)
`playback.rs:179` эмитит **Tauri-event** `app.emit("playback-finished", ())` (для фронта).
А `event_loop.rs:98` слушает **`AppEvent::PlaybackFinished`** (внутренний sender-канал) и только
там вызывает `pb.on_playback_finished()` (выбор следующей фразы).
→ Эти два канала не связаны → `on_playback_finished()` **никогда не вызывается** → очередь
зависает на первой фразе.
**Фикс:** эмитить `AppEvent::PlaybackFinished` во внутренний sender (чтобы event_loop подтянул
следующую фразу) — Tauri-event для фронта можно оставить отдельно, либо дублировать.
**Проверено лично** (`grep`): `playback.rs:179` → emit Tauri-event; `event_loop.rs:98` →
`AppEvent::PlaybackFinished`. Каналы не соединены.

### C4. Инициализация окна `playback-control` отсутствует
В `setup.rs` нет ни показа, ни применения темы, ни `set_window_exclude_from_capture` для
`playback-control`. Окно декларировано в `tauri.conf.json` (`visible:true`), но не обрабатывается
по образцу `soundpanel_window.rs`.
- **exclude-from-capture НЕ применён** — регрессия (alwaysOnTop-окно должно исключаться из захвата).
- **Тема dark/light не подключена** к этому окну (`data-theme` не выставляется, событие темы не
  эмитится на него) → светлая тема мёртвая.
**Фикс:** добавить инициализацию по образцу soundpanel (show + theme + exclude_from_capture),
применять тему при переключении (`commands/mod.rs::update_theme`).

### C5. `.unwrap()` в `enqueue` — паника в команде (нарушение правила проекта)
`playback.rs:220`: `self.state.read().current.as_ref().unwrap()`. Между освобождением write-лока
(после установки `current`) и этим read другой поток мог выполнить `stop()`/`on_playback_finished()`
и обнулить `current` → `unwrap()` паникует в Tauri-команде.
**Фикс:** вернуть `current` из блока записи (а не ре-чтением), либо `match`/`?` вместо `unwrap`.

### C6. Хоткей паузы переключает Pause даже из Idle
`hotkeys.rs::handle_playback_pause`: `PlaybackStatus::Idle` попадает в ветку `pause()` → статус
становится `Paused` без играющего звука, фронт покажет некорректное состояние.
**Фикс:** из `Idle` — ничего не делать (resume из Idle невозможен).

### C7. Команды playback возвращают `()` вместо `Result<(), String>`
`commands/playback.rs` — все команды игнорируют ошибки (нет обработки «менеджер не
инициализирован» / поток упал). Нарушение конвенции проекта и чек-листа.
**Фикс:** `Result<T, String>`, возвращать ошибку при недоступности менеджера/потока.

### C8. `cargo check` warning: `state.rs:121 field cached_devices is never read`
`AppState.cached_devices` теперь не читается (раньше передавался в `play_mp3_async_dual`).
Если dual output возвращается через C1 — поле снова понадобится; иначе убрать field/документировать.
**Связано с C1** — реши после C1.

## Средние (рекомендуется)

### M1. Кеш `audio_cache` раздувается при replay
`replay_from_cache` (playback.rs) достаёт аудио и вызывает `enqueue("replay_{id}", ...)` →
кладёт в кеш под новым id `replay_{orig}`. Повторный replay старого id больше не сработает,
кеш растёт, `id` для Vue `:key` ломается (`replay_replay_...`).
**Фикс:** не плодить новые id при replay — переиспользовать исходный id/кеш.

### M2. `try_recv()` + `sleep(100ms)` = latency и возможная потеря порядка команд
`playback.rs` — неблокирующий опрос в цикле. Две команды в одном 100мс-окне обрабатываются с
задержкой. **Фикс:** блокирующий `recv()` + `recv_timeout()` для проверки `sink.empty()`.

### M3. Ранний выход потока без уведомления
`playback.rs:100-112` — если нет устройства/`OutputStream` не создан, поток выходит, `cmd_tx`
остаётся живым → команды уходят в никуда, менеджер считает себя рабочим.
**Фикс:** выставлять флаг ошибки в состоянии, проверять в командах.

### M4. Завершение при Quit не реализовано
Нет хука на `AppEvent::Quit` для drop `cmd_tx`/join потока. Среднее (OS приберёт), но при
перезапуске dev-сервера могут быть артефакты.

### M5. `VecDeque` очереди без bound
`Shared::queue` растёт без лимита. Если синтез быстрее воспроизведения — копится бесконечно.
**Фикс:** ограничить (или документировать поведение при переполнении).

### M6. `try_seek(Duration::ZERO)` для mp3 может быть ненадёжен
На некоторых контейнерах seek в 0 через Symphonia глючит. **Фикс:** протестировать; запасной
вариант — перевыгрузка из кеша.
