# План: повторять Telegram auth.sendCode после AUTH_RESTART

## Цель

Убрать необходимость повторного нажатия пользователем при transient-ошибке Telegram `500 AUTH_RESTART` во время первого запроса кода.

## Scope

- `src-tauri/src/telegram/client.rs`

Не изменять пользовательские несвязанные правки, UI, Cargo.toml и другие auth-сценарии.

## Требования

1. В `TelegramClient::request_code` вызывать `client.request_login_code` максимум три раза: исходная попытка плюс два retry.
2. Повторять только структурированную ошибку `InvocationError::Rpc`, где `code == 500` и `name == "AUTH_RESTART"`.
3. Между попытками использовать небольшую возрастающую задержку, например 250 мс и 500 мс.
4. Для успеха сохранить полученный `LoginToken` как сейчас и вернуть `AuthState::CodeRequired`.
5. Для любой другой ошибки, таймаута или исчерпания retry вернуть существующее безопасное сообщение об ошибке.
6. Не логировать номер телефона, API hash, токен или полную ошибку с потенциальными чувствительными данными. Можно залогировать только номер попытки и безопасную категорию `AUTH_RESTART`.
7. Добавить unit-тест только если это возможно без реального Telegram; не добавлять сетевые/флейковые тесты.

## Проверка

- `cargo check` из `src-tauri`.
- `npx vue-tsc --noEmit` не требуется к изменению, но не ломать frontend.
- `git diff --check`.
