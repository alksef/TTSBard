# План 101: Оценка — нужна ли кнопка «Пропускать нажатия (click-through)» саундпанели

- **Дата:** 2026-07-08
- **Тип:** evaluation / UX-решение (не баг, вопрос дизайна)
- **Вопрос (от пользователя):** «оценка — а нужна ли теперь кнопка "Пропускать нажатия (click-through)"
  саундпанели?»
- **Контекст цикла:** см. `docs/deepseek/WORKFLOW.md`

---

## Контекст (что есть сейчас)

- **Click-through** (`sp_set_floating_clickthrough`) — плавающее окно саундпанели становится
  «прозрачным для кликов»: мышь проходит сквозь него в приложение под ним.
- **Где управляется:**
  - Плавающее окно (`SoundPanelApp.vue:122 toggleClickthrough`, кнопка 👆/🖱️ в title-bar).
  - Основное окно (`SoundPanelTab.vue:279 saveClickthrough`, чекбокс «Пропускать нажатия
    (click-through)» + hint «Окно не будет перехватывать клики мыши»).
  - Persist в `windows.json` (`WindowsManager.set_soundpanel_clickthrough`).
- **A-Z играют через DOM keydown** в окне саундпанели (план 97) — для этого окно **должно быть
  в фокусе** (или хотя бы получать keydown).
- Окно имеет кнопки: cycle-set (◀▶), click-through, transparency, close — для клика мышью.

## Анализ — нужен ли click-through?

### Когда click-through ПОЛЕЗЕН
Стример играет в игру на весь экран. Саундпанель — оверлей поверх игры. Нажимая буквы A-Z, он
озвучивает звуки, **не переключая фокус** с игры (чтобы игра продолжала получать мышь/клавиши).
Click-through позволяет окну «висеть» поверх, не перехватывая клики мыши по игре. Это валидный
use-case для стримеров (одна из двух аудиторий проекта).

### Противоречие с текущей архитектурой (window-mode + DOM keydown)
После плана 91 (саундпанель = окно + DOM keydown) окно **нуждается в фокусе**, чтобы получать
`keydown` A-Z. Но click-through-окно обычно **теряет/не имеет** фокус (мышь проходит сквозь).
Возможен конфликт: click-through ON → окно без фокуса → A-Z через DOM keydown не работают.
→ Если так, click-through стал **функционально бесполезным** после перехода на window-mode.

### Однако
NumPad/F-keys Intercept (план 91-B) работает через **low-level hook** (`hook.rs`), который
**не зависит от фокуса окна**. Т.е. звуки, привязанные к Intercept-клавишам (не A-Z), работают
и при click-through. Только A-Z (DOM) не будут.

## Варианты решения (выбрать с пользователем)

### Вариант A — Убрать click-through (если A-Z — основной путь)
Если A-Z через DOM — основной способ играть звуки в саундпанели, click-through мешает (окно без
фокуса → A-Z мертвы). Тогда кнопка бесполезна → убрать UI + настройку + backend. Минус: теряем
стримерский «hover-оверлей без кликов». Но раз A-Z не работают в click-through, пользы и так нет.

### Вариант B — Оставить click-through, перевести A-Z на low-level hook (как Intercept)
Сделать A-Z саундпанели через `WH_KEYBOARD_LL` (как NumPad/F-keys в `hook.rs`), чтобы они
работали **без фокуса окна** — тогда click-through становится осмысленным (окно висит,
A-Z играют глобально). Это **большая** работа (почти план 91 наоборот для A-Z) + раньше A-Z
убрали из хука намеренно (почему? — проверить git-историю плана 91).

### Вариант C — Оставить как есть, но документировать ограничение
Не трогать. В hint/settings указать: «при click-through буквы A-Z не работают (нужен фокус);
используйте Intercept (NumPad/F-keys) для глобального запуска». Минимальные правки.

### Вариант D — Убрать только из основного окна, оставить в плавающем
Кнопка click-through в плавающем окне (title-bar) — быстро переключить. Дублирующий чекбокс в
основном окне (SoundPanelTab) — лишний, т.к. есть внешний вид. Убрать из основного окна.

## Рекомендация (предварительная)
**Вариант A или D** — если подтвердить, что A-Z в click-through не работают. Сначала проверить
это (runtime), потом решить. **Скорее всего click-through устарел** после window-mode (план 91),
когда A-Z переехали на DOM keydown. Но решение за пользователем — это дизайн, не баг.
