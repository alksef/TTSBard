# Task 96-round1-01: Саундпанель пуста при старте — починить загрузку + find_active_index

План: `docs/deepseek/plan/96-2026-07-08-soundpanel-empty-on-startup.md` (читать обязательно).

## Симптом
При запуске звуковая панель пуста. После добавления новой привязки появляются **старые** записи.

## Проверено Claude (факты — НЕ перепроверять)
- Файл `%APPDATA%\ttsbard\soundpanel_bindings.json` — корректный новый формат `SoundSets`,
  активный набор валиден, содержит bindings. Backend `load_bindings` (setup.rs:158 → storage.rs)
  грузит корректно, `replace_sets` вызывается ДО показа окон.
- ⇒ Баг во фронте: первичная загрузка пуста, перезагрузка по событию подтягивает данные.

## Два фикса

### Фикс 1 (обязательный, точный баг) — `find_active_index` всегда возвращает 0
`src-tauri/src/soundpanel/state.rs:62-73`:
```rust
pub fn find_active_index(&self) -> usize {
    if !self.active_set_id.is_empty() {
        if let Some(idx) = self.sets.iter().position(|s| s.id == self.active_set_id) {
            return idx;
        }
    }
    if self.sets.is_empty() { 0 } else { 0 }   // ← ОБЕ ветки = 0! баг
}
```
Эта функция используется в `add_binding` (state.rs:160) и `remove_binding` (state.rs:173).
Если активный набор НЕ первый (index > 0), `add_binding`/`remove_binding` пишут в **первый**
набор, а не в активный. Это объясняет «добавил → появились СТАРЫЕ записи»: новая привязка
попадала не туда, а UI после `emit` перечитывал активный набор (где старые).

**Фикс:** вернуть реальный индекс. Правильно:
```rust
pub fn find_active_index(&self) -> usize {
    if let Some(idx) = self.sets.iter().position(|s| s.id == self.active_set_id) {
        idx
    } else {
        0
    }
}
```
(Если active_set_id невалиден/пуст — fallback на 0, как find_active fallback на first.)

**Тест** (добавить в `state.rs` `#[cfg(test)] mod tests`):
```rust
#[test]
fn test_find_active_index_not_first() {
    let sets = SoundSets {
        active_set_id: "set2".into(),
        sets: vec![
            SoundSet { id: "set1".into(), name: "First".into(), bindings: vec![] },
            SoundSet { id: "set2".into(), name: "Second".into(), bindings: vec![] },
        ],
    };
    assert_eq!(sets.find_active_index(), 1);   // сейчас упадёт (вернёт 0)
}

#[test]
fn test_find_active_index_invalid_falls_back_to_zero() {
    let sets = SoundSets {
        active_set_id: "bogus".into(),
        sets: vec![SoundSet { id: "set1".into(), name: "First".into(), bindings: vec![] }],
    };
    assert_eq!(sets.find_active_index(), 0);
}
```

### Фикс 2 (вероятный, race при показе окна) — перезагружать bindings при показе
Проверить `src-tauri/src/soundpanel_window.rs` (`show_soundpanel_window`): как создаётся окно?
Если плавающее окно (`src-soundpanel/SoundPanelApp.vue`) монтируется один раз при старте
(hidden), а `onMounted` отрабатывает тогда — к первому показу данные могут быть устаревшими,
перерисовка только по событию.

**Фикс (если применимо):** при показе плавающего окна форсить перезагрузку bindings. Варианты:
- В `show_soundpanel_window` после `window.show()` — `app_handle.emit("soundpanel-bindings-changed",
  ())` (фронт перечитает). Простой, надёжный.
- ИЛИ во фронте `SoundPanelApp.vue` слушать `tauri://focus`/`window-shown` → `loadSets()` +
  `loadBindings()`.
Выбрать менее инвазивный (emit при show — предпочтительнее, единая точка).
