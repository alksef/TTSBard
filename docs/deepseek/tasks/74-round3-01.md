## R-1 (КРИТИЧНО) — Repeat во время паузы оставляет sink в паузе → тишина

**Файл:** `src-tauri/src/playback.rs`, ветка `Ok(Cmd::Repeat)` (примерно строки 261-278).

**Баг:** если фраза на паузе (`Cmd::Pause` уже вызвал `sink.pause()`,
`sink.is_paused()==true`, `status==Paused`) и пользователь жмёт Repeat (🔁),
то `try_seek(Duration::ZERO)` на paused-sink срабатывает → `seek_ok==true` →
fallback НЕ идёт. Но `sink.is_paused()` остаётся `true`, `sink.play()` не
вызывается → позиция сброшена в 0, а звука нет, `status` остаётся `Paused`.

**Ожидаемое поведение:** Repeat ВСЕГДА должен дать звучание с начала, в том
числе из состояния паузы.

**Фикс:** в ветке `Cmd::Repeat`:
1. Запомни, был ли хотя бы один sink на паузе ДО seek.
2. После **успешного** seek обоих sink'ов: вызови `sink.play()` для КАЖДОГО
   sink (spk и mic), чтобы гарантировать звучание. Если до seek был пауз-статус —
   выстави `state.write().status = PlaybackStatus::Playing` и отправь
   `internal_ev.send(AppEvent::PlaybackResumed)` + `app.emit("playback-resumed", ())`.
3. Поставь локальные флаги потока `playing = true; stopped = false;` (иначе блок
   детекции завершения `if playing && !stopped` на строке ~283 не запустится и
   естественный конец фразы не детектируется → очередь не двинется).
4. Если sink'ов нет (`sink_spk.is_none() && sink_mic.is_none()`) — это no-op,
   ничего не делай (повторять нечего).
5. Fallback при ошибке seek (`!seek_ok`) — оставь как есть (Stop + Enqueue из
   `state.read().current`), он корректен. Но учти: в этом fallback тоже надо
   гарантировать `playing=true` после повторного Enqueue — это уже произойдёт в
   ветке Enqueue (ставит `playing=true`), так что дополнительно ничего не надо.

**Критерий:** после Pause → Repeat звук идёт с начала, `status==Playing`,
фронт получает `playback-resumed`. Repeat из Playing — перемотка в начало,
звучание продолжается. Repeat из Idle/Stopped (нет sink) — no-op.

## R-2 — `repeat()` публичный метод / команда `playback_repeat` молча no-op

**Файл:** `src-tauri/src/playback.rs`, метод `repeat()` (~строка 377).

**Сейчас:** `repeat()` всегда `Ok(())`, даже когда повторять нечего (sink'ов нет).

возвращает `Ok(())` при no-op — UI не понимает, сработало ли.

**Фикс (минимальный):** ничего не меняй в сигнатурах, ОСТАВЬ `Result<(), String>`.
Достаточно: в ветке `Cmd::Repeat` при отсутствии sink'ов (пункт R-1.4)
логируй `warn!("Repeat: nothing playing")` — чтобы no-op был видим в логах, а
не немым. Больше менять не нужно.

## M-1 (СРЕДНЕЕ) — дубликат в истории фраз при replay

**Файл:** `src-tauri/src/playback.rs`.

**Баг:** `replay_from_cache(id)` (~строка 381) зовёт `enqueue(orig_id, text, ...)`,
а `enqueue` (~строка 312) ВСЕГДА вызывает `add_history` (строка ~345). Повторный
replay одной фразы плодит одинаковые записи в `phrase_history` (один id, тот же
текст). Фронт `src-playback/PlaybackControlApp.vue` использует `:key="entry.id"`
(строка 132) → дублирующиеся элементы.

**Фикс:** в `add_history(id, text)` (~строка 350) сделай дедуп по `id`:
если запись с таким `id` уже есть в `phrase_history` — удали её, затем добавь
новую (с актуальным timestamp) в конец. Так recent будет содержать уникальные id,
а повтор replay одной фразы будет двигать её наверх списка, а не дублировать.

**Критерий:** 3× replay одной фразы → в `recent` ОДНА запись с этим id (последняя
по времени), `:key` не дублируется.
