# Task 124 — безопасная нормализация путей в логах

## Цель

Убрать раскрытие имени пользователя и локальной структуры каталогов из диагностических логов. Сохранить полезную информацию о назначении файла и относительном хвосте пути.

## Область изменений

- `src-tauri/src/secret_log.rs`
- `src-tauri/src/telegram/client.rs`
- `src-tauri/src/setup.rs`

Не менять формат хранения файлов, рабочую логику, конфигурацию или поведение авторизации.

## Требования

1. Добавить в `secret_log` helper для безопасного пути, например `safe_path_for_log(path: &Path) -> String`.
2. Нормализатор должен заменять известные корни на маркеры:
   - `%APPDATA%` → `[APP_DATA]`;
   - `%LOCALAPPDATA%` → `[LOCAL_APP_DATA]`;
   - `%TEMP%`/`%TMP%` → `[TEMP]`;
   - текущую рабочую директорию → `[WORKDIR]`.
3. Сопоставление корней должно работать на Windows независимо от регистра пути. Если путь находится внутри корня, сохранить относительный хвост, например `[APP_DATA]/ttsbard/telegram.session`.
4. Любой неизвестный абсолютный путь не логировать как есть: возвращать `[ABSOLUTE_PATH]` (или эквивалентный явно безопасный маркер).
5. Относительные пути можно оставить, так как они не раскрывают имя пользователя.
6. Заменить логирование `session_path = ?session_path` в `telegram/client.rs` на безопасное значение helper-а.
7. Заменить логирование `aff_path` и `dic_path` в `setup.rs` на безопасное значение helper-а.
8. Проверить `Debug`-реализацию `TelegramClient`: она не должна раскрывать абсолютный `session_path` при `{:?}`. Если она может попасть в лог, использовать безопасное представление или не включать значение пути.
9. Добавить unit-тесты для helper-а минимум для:
   - `%APPDATA%/ttsbard/telegram.session`;
   - пути внутри `%USERPROFILE%`, если он используется как fallback для защиты имени пользователя;
   - неизвестного абсолютного пути;
   - относительного пути;
   - регистра корня на Windows.
10. Не изменять старые лог-файлы — маскировка применяется только к новым сообщениям.

## Политика приватности

Имя пользователя Windows не должно появляться в новых логах. Не логировать полный `USERPROFILE`, `APPDATA`, `LOCALAPPDATA`, `TEMP` или абсолютные пути проекта. Для диагностики достаточно маркера корня и относительного хвоста.

## Приёмка

- В исходниках нет новых логов абсолютных путей из перечисленных мест.
- Новый лог session-файла имеет вид `[APP_DATA]/ttsbard/telegram.session`.
- Новый лог spellcheck имеет вид `[APP_DATA]/ttsbard/.../ru.aff` либо другой безопасный маркер с относительным хвостом.
- `cargo test` для crate и `cargo check` проходят.
- Изменения ограничены задачей и не затрагивают старые логи.
