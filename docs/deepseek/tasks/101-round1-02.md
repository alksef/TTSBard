# Task 101-round1-02 — Deduplicate persistent-cache replay in playback recent list

Read `docs/stage/34-silero-voice-and-playback-recent-dedup.md` and `docs/deepseek/plan/101-silero-voice-playback-recent.md`.

The current `replay_phrase_from_cache` creates a fresh UUID on every click, while `PlaybackManager.get_state()` exposes `audio_cache` as the playback window's `recent` list. As a result, repeatedly replaying the same history cache entry creates repeated recent records.

## Scope

- `src-tauri/src/commands/history.rs`
- `src-tauri/src/playback.rs`
- focused backend tests if practical

## Requirements

1. Give persistent-cache replay a stable identity derived from the history entry/cache key, so repeated replay of the same history entry does not create multiple recent-cache records.
2. Preserve normal newly synthesized phrases as distinct playback entries.
3. Avoid adding duplicate queue entries for repeated clicks on the same persistent history entry while it is already current or queued, where this can be done without changing normal queue semantics.
4. Keep existing memory-cache replay (`replay_phrase`) behavior intact.
5. Ensure IDs remain safe and do not collide unexpectedly with unrelated normal playback IDs.
6. Do not change the persistent audio-cache format or history schema.

Run focused playback/history tests and `cargo check --manifest-path src-tauri/Cargo.toml`.
