# Task 102-round2-01: Playback — диагностические логи (локализация regression)

План: `docs/deepseek/plan/102-2026-07-08-playback-window-not-showing.md`.
Round 1 (DeepSeek) тайм-аутнулся на расследовании. Этот раунд — **только добавить
диагностические логи**, чтобы пользователь увидел в `%APPDATA%\ttsbard\logs` где обрывается
цепочка. Без переписывания логики.

## Симптом
Окно управления не показывает воспроизведение + нет recent-истории. Регрессия.

## Что Claude нашёл в коде (факты)
- `playback.rs:194` — `PlaybackStarted` emit **срабатывает ТОЛЬКО если** `sink_spk.is_some() ||
  sink_mic.is_some()` (устройство вывода открылось). Если speaker+mic оба None (device resolve /
  open_sink failed — warn на строках 169/173/184/189) → playing не стартует, emit НЕ идёт, state
  остаётся Idle. ⇒ «не показывает воспроизведение».
- `playback.rs:152` — `if playing { continue; }` — пока playing=true, новые Enqueue игнорируются
  (но фразы копятся в queue). playing сбрасывается в `false` на stop (243) или когда оба sink
  `.empty()` (298).
- Emit `app.emit("playback-started", ...)` — broadcast (правильно), фронт `listen` глобальный
  (правильно). Канал совпадает — НЕ причина.
- `enqueue` (320) push-ит в audio_cache → recent должен пополняться, ЕСЛИ enqueue вызывается.

⇒ Подозреваемые: (a) TTS не отдаёт audio / ошибка до enqueue; (b) аудиоустройство не открывается
  (строка 194 не выполняется); (c) playing застревает true.

## Задача — добавить `info!`/`debug!` логи в ключевых точках (МИНИМАЛЬНО, без правки логики)

В `src-tauri/src/playback.rs` (playback loop, ~150-310):
1. При получении `Cmd::Enqueue` (151): `info!(target:"playback", text=%phrase.text, "Enqueue received")`.
2. На строке 194 — ДО if: `info!(target:"playback", has_spk=sink_spk.is_some(), has_mic=sink_mic.is_some(), "Playback start check")`. И **в ветке else** (когда оба None — сейчас её нет явно, код просто не заходит в if) — добавить `warn!(target:"playback", "No output sink — playback NOT started (speaker+mic both failed)")`. Это **главный диагностический лог** — если он появляется, значит аудиоустройство не открывается.
3. При `PlaybackStarted` emit (197-202): `info!(target:"playback", "PlaybackStarted emitted")`.
4. При `PlaybackFinished` (304): `info!(target:"playback", "PlaybackFinished, playing reset")`.
5. При `playing = false` (298, drains): `debug!(target:"playback", "Sinks drained, playing=false")`.

В `src-tauri/src/commands/mod.rs` `speak_text_internal` (~254-262):
6. Перед enqueue (258): `info!(target:"playback", "Enqueueing phrase to PlaybackManager")`.
7. После `pb.enqueue(...)` (259) — залогировать результат (bool): `info!(target:"playback", enqueued=result, "enqueue result")`.

В `src-tauri/src/commands/playback.rs` `get_playback_state` (55):
8. `debug!(target:"playback", status=?dto.status, current=dto.current.is_some(), recent=dto.recent.len(), "get_playback_state")`.

## Верификация
- `cargo check` 0/0.
- `cargo test --lib` зелёные (логи не ломают тесты).
- Логи только `info!`/`debug!`/`warn!` — НИКАКОЙ правки логики воспроизведения.

## Что это даст
Пользователь запускает, вводит текст → смотрит `%APPDATA%\ttsbard\logs` (или консоль с
RUST_LOG=info). Цепочка должна показать:
- «Enqueueing phrase» → «Enqueue received» → «Playback start check» → если has_spk/has_mic
  false → **«No output sink»** (причина = аудиоустройство) → если true → «PlaybackStarted emitted».
- Если «PlaybackStarted emitted» есть, но UI пустой → баг UI (отдельный фикс).
- Если enqueue не вызывается → баг в speak_text_internal до строки 258 (TTS-провайдер).

По логам локализуем точную причину → следующий раунд — точечный фикс.

## Не делать
- НЕ править логику воспроизведения (enqueue, playing-флаг, emit). Только логи.
- НЕ менять TTS-провайдеры.
