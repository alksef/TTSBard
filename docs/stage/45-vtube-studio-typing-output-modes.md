# Stage 45 — VTube Studio: универсальные выходы для состояния набора

## Проблема

Сейчас интеграция всегда создаёт custom parameter `TTSBardTyping` и отправляет
ему `1` при начале набора и `0` при завершении. Это подходит только моделям,
риг которых уже умеет реагировать на такой параметр. У большинства готовых
моделей такого параметра нет, но в них уже можно создать Hotkey для выражения,
анимации или предмета прямо в VTube Studio.

Нужны два способа отдать одно и то же логическое событие `typing`:

1. произвольный custom parameter для моделей с ригом;
2. две пользовательские команды Hotkey: одна для начала набора, другая для
   его завершения.

## Цель и границы

Добавить настройку способа доставки typing-state в панели VTube Studio. В UI
эти способы называются **«Событие»** и **«Горячие клавиши»**: первое означает
внешний VTube Studio custom parameter, второе — API-запуск двух Hotkey.
Изменение не затрагивает источник состояния: `useTypingBurst` и команда
`set_vtube_studio_typing(bool)` сохраняют существующую семантику одного
перехода `false → true` и одного перехода `true → false` на burst.

В этом stage не добавляются новые Live2D-параметры, не редактируется модель и
не имитируются физические нажатия клавиатуры. Hotkey запускаются штатным
VTube Studio API `HotkeyTriggerRequest`.

## Модель настроек

Заменить существующую typing-конфигурацию в `VTubeStudioSettings` новой
обязательной схемой:

```rust
pub enum VTubeStudioTypingOutputMode {
    Parameter,
    Hotkeys,
}

pub struct VTubeStudioTypingSettings {
    pub output_mode: VTubeStudioTypingOutputMode,
    pub parameter_name: String,
    pub typing_started_hotkey_id: String,
    pub typing_stopped_hotkey_id: String,
}
```

- `Default` для чистой установки: `Parameter`, `TTSBardTyping`, пустые
  идентификаторы Hotkey. Миграция старого `settings.json`,
  `#[serde(default)]` для заменяемых полей и сохранение прежнего поведения не
  требуются. Во время разработки старый конфиг можно удалить/пересоздать.
- `parameter_name` — именно ID custom parameter, а не отображаемое имя
  «события». Он применяется в `ParameterCreationRequest` и
  `InjectParameterDataRequest`.
- В настройках и safe DTO не хранить/не возвращать токен VTube Studio.
- При `Hotkeys` оба ID обязательны после `trim()`. Нельзя сохранить режим с
  пустым ID. В режиме `Parameter` пустое имя параметра также недопустимо.
- Строки не нормализовать и не менять регистр: `hotkeyID` и имена параметров
  могут быть чувствительны к регистру. Ограничить длину значением, допустимым
  VTS API (не более 64 символов); сообщение ошибки должно называть конкретное
  поле.

## UI панели VTube Studio

Разделить панель на два независимых блока, расположенных один под другим.

### 1. Подключение

Первый блок сохраняет существующие порт, Start/Stop/Restart и автозапуск.
Его единственная ответственность — открыть WebSocket и авторизовать TTSBard в
VTube Studio. Успешное подключение **не** создаёт custom parameter и не
запускает Hotkey: оно не зависит от выбранного способа typing.

Поэтому пользователь может подключиться к VTube Studio до настройки действия
набора. Этот блок имеет собственную кнопку сохранения для порта/автозапуска;
она не сохраняет настройки typing.

### 2. Действие при наборе

Второй блок заменяет фиксированную карточку «Статус набора». Он содержит
настройки доставки typing-state и собственную кнопку **«Сохранить действие»**.
Её нажатие сохраняет только выбранный способ и его поля; соединение не
перезапускается.

1. `select` «Способ управления»:
   - `Событие`;
   - `Горячие клавиши`.
2. Когда выбрано **«Событие»**, показать только поле «Имя события» с начальным
   значением `TTSBardTyping` и пояснение `1 = печатает, 0 = ожидание`. В
   протоколе это имя custom parameter; слово «событие» используется в UI,
   потому что описывает пользовательское назначение сигнала.
3. Когда выбраны **«Горячие клавиши»**, показать отдельную панель:
   - кнопку `Загрузить Hotkey` (ручное обновление списка);
   - состояние загрузки и понятную ошибку, если VTube Studio недоступна;
   - поле выбора «Hotkey начала набора»;
   - поле выбора «Hotkey окончания набора».
   Поле имени события в этом режиме не отображается.
4. Кнопка ручной загрузки запрашивает `HotkeysInCurrentModelRequest` только в уже активной
   аутентифицированной сессии. Выпадающие списки показывают
   `name — type`; выбор сохраняет уникальный `hotkeyID`, не имя. Ручного
   ввода технического ID в UI нет.
5. Если текущий сохранённый ID не найден в ответе, не очищать его молча:
   показать «Не найден в загруженной модели» и потребовать заново выбрать
   значение из обновлённого списка. Переключение модели после сохранения не
   переписывает настройки автоматически.
6. Существующая карточка тестирования сохраняет таймаут, повторы и все
   ограничения активной сессии, но переименовывается из «Тест параметра» в
   нейтральный **«Тест действия»**. Она зависит от выбранного способа:
   - параметр: как сейчас, посылает `1`, ждёт заданный интервал, посылает `0`;
   - Hotkey: один раз запускает Hotkey начала, ждёт интервал, затем запускает
     Hotkey окончания. Повторы сохраняют ту же последовательность.
7. Текст справки должен честно объяснять: для Hotkey пользователь заранее
   создаёт в VTube Studio два действия, например включение и выключение
   выражения. API запускает выбранные действия, но не знает их смысл и не
   восстанавливает состояние выражения самостоятельно.

UI не должен показывать поля неактивного способа и не должен сбрасывать их
значения при переключении: пользователь может вернуться к предыдущей
настройке без повторного ввода. Поля второго блока — черновик: выбранный
способ, имя события и оба Hotkey применяются к runtime и записываются в
настройки только кнопкой **«Сохранить действие»**. `Загрузить Hotkey`
запрашивает лишь список вариантов для селекторов, не меняет сохранённую
конфигурацию и не переключает runtime-режим.

Кнопка загрузки отключена вне состояния `Connected`, с подсказкой
«Сначала подключитесь к VTube Studio». Переключить черновик на режим Hotkey
можно и до подключения; сохранить его без двух выбранных Hotkey нельзя.
Обычный путь: **Подключение → выбрать „Горячие клавиши“ → Загрузить Hotkey →
выбрать начало и окончание → Сохранить действие**.

## Протокол и backend

### Parameter mode

Обобщить конструкторы сообщений: убрать жёсткую зависимость от
`TYPING_PARAMETER_NAME`, передавая `parameter_name: &str` в создание и inject.
`TTSBardTyping` остаётся только значением default-константы. Подставлять
пользовательское имя во все ветки: первичное подключение, повторное
подключение при typing, keepalive, stop и тест.

Существующий keepalive раз в 500 мс остаётся только здесь, поскольку
`InjectParameterDataRequest` в режиме `set` временно перехватывает параметр.

### Hotkey mode

Добавить типы запросов/ответов:

```json
{ "messageType": "HotkeysInCurrentModelRequest", "data": {} }
{ "messageType": "HotkeyTriggerRequest", "data": { "hotkeyID": "..." } }
```

`HotkeysInCurrentModelResponse.data.availableHotkeys` десериализуется в DTO с
минимумом `name`, `type`, `description` и `hotkeyID`. Новый Tauri command
возвращает этот DTO только после проверки `desired_running`, статуса
`Connected` и живого socket; не выполняет неявное подключение и не меняет
настройки.

В `set_typing` выбрать режим один раз для одного вызова:

- `typing=true` в `Hotkeys` запускает только `typing_started_hotkey_id`;
- `typing=false` запускает только `typing_stopped_hotkey_id`;
- keepalive в Hotkey-режиме останавливается/не создаётся;
- при Stop приложения best-effort отправить Hotkey окончания до закрытия
  socket, если typing был активен;
- transport/API error сбрасывает socket, очищает auth runtime state и
  публикует `Error` по тем же правилам, что и inject-путь.

Подключение и аутентификация не должны пытаться создать custom parameter ни в
одном режиме. В parameter-режиме параметр создаётся лениво только при первом
`typing=true` или по тесту; в Hotkey-режиме он не создаётся вовсе. Это важно
для модели без `TTSBardTyping` и позволяет сначала подключиться, затем
загрузить список Hotkey.

`HotkeyTriggerRequest` принимает ID или имя, но приложение сохраняет ID,
чтобы одинаковые имена не выбирали произвольный Hotkey. Технический ID не
показывается пользователю как поле ввода.

## Пользовательская настройка в VTube Studio

1. Загрузить модель, открыть её настройки и вкладку Hotkeys.
2. Создать два Hotkey с понятными названиями, например `TTS typing start` и
   `TTS typing stop`; назначить им нужные действия.
3. В TTSBard запустить подключение, выбрать `Горячие клавиши`, обновить список,
   выбрать оба Hotkey и сохранить.
4. В блоке «Тест действия» нажать «Проверить»: начало должно сработать сразу, окончание — после
   указанного интервала.

Многие действия VTS типа «Set/Unset Expression» являются toggle. Для них
начальный и конечный Hotkey часто будут двумя отдельными hotkey-конфигурациями
одного выражения; пользователь должен убедиться, что исходное состояние
выражения выключено и что оба действия дают ожидаемый результат. Stage не
выдаёт toggle за гарантированное «set on»/«set off»; если нужна строгая
синхронизация выражения, это отдельный будущий режим на
`ExpressionActivationRequest`.

## Приёмка и проверки

1. Имя parameter сохраняется, возвращается в DTO, отправляется в create,
   inject, keepalive, reset и test-pulse.
2. В Hotkey-режиме на `typing=true/false` уходят соответственно только два
   указанных `HotkeyTriggerRequest`; нет `ParameterCreationRequest`,
   `InjectParameterDataRequest` и keepalive.
3. Старт, переподключение и Stop сохраняют lifecycle-инварианты. Stop при
   активном typing посылает только Hotkey окончания best-effort.
4. `get_vtube_studio_hotkeys` не подключается сам, не меняет runtime/status и
   возвращает понятную ошибку при отключённой сессии или отсутствии модели.
5. UI не допускает сохранения пустых полей активного режима; ввод inactive
   режима сохраняется без требования валидности.
6. Ответ VTS с API error на получение/запуск Hotkey не приводит к ложному
   «успешно»; socket/status меняются консистентно с текущей обработкой ошибок.
7. Выполнить focused Rust unit tests для `messages`, `service`, VTS commands,
   frontend tests composable/panel, затем `cargo check` и `vue-tsc --noEmit`.
