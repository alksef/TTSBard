# Stage 37: Горячие клавиши приложения и возврат фокуса

**Дата:** 2026-07-17
**Статус:** реализовано; требуется ручная проверка runtime-сценария

## Цель

Расширить существующую систему горячих клавиш третьей моделью работы:

1. глобальные сочетания, работающие независимо от активного приложения;
2. экспериментальный общий перехват NumPad-клавиш;
3. локальные клавиши, работающие только внутри главного окна TTSBard.

Отдельно проработать действие «вернуться в предыдущее окно» после вызова окна TTSBard.

## Текущее состояние

- Глобальные сочетания регистрируются через `tauri-plugin-global-shortcut`.
- В настройках уже есть шесть глобальных действий: главное окно, звуковая панель,
  пауза, стоп, повтор и окно управления воспроизведением.
- `HotkeysPanel.vue` сейчас показывает их одним списком без визуального разделения.
- Экспериментальный NumPad Control — общий Windows low-level hook для NumPad/F-клавиш
  и назначаемых действий; он не является системой горячих клавиш саундпанели и не
  должен смешиваться ни с обычными глобальными сочетаниями, ни с локальными клавишами
  главного окна.
- Вызов `window.set_focus()` активирует окно TTSBard, но не сохраняет и не восстанавливает
  окно, которое было активно до этого.

## 1. Стандартная модель сочетаний

### Рекомендуемый формат

Хранить сочетание как структурированное значение:

```text
modifiers: [Ctrl, Shift]
key: F3
```

Это уже соответствует текущему `Hotkey` и удобно для проверки конфликтов, отображения
и преобразования в `tauri_plugin_global_shortcut::Shortcut`.

Для нового слоя следует разделять не формат клавиши, а область действия:

```text
scope: global | main_window
action: typed action id
binding: modifiers + physical key code
```

NumPad Control не включать в этот binding-слой: у него собственная экспериментальная
модель «перехваченная физическая клавиша → действие» и собственный low-level hook.

### Какие варианты поддерживать

- обычное сочетание: `Ctrl+Shift+F3` — основной вариант;
- одиночная функциональная клавиша: `F8` — допустимо для специальных режимов;
- одиночная буквенная/цифровая клавиша — только для явного режима перехвата, иначе
  она будет мешать вводу текста;
- последовательности вроде `Ctrl+K`, затем `Ctrl+C` — не включать в первый этап;
  это уже key-chord state machine и требует тайм-аута, отмены и отдельного UX;
- удержание клавиши, double-tap и mouse-кнопки — также отложить.

Для app-local клавиш предпочтительны `KeyboardEvent.code` и физические коды, а не
`KeyboardEvent.key`: `key` зависит от раскладки. Отображаемое имя можно строить
отдельно, а сравнение выполнять по canonical code.

### Правила записи и конфликтов

- Модификатор без основной клавиши не сохраняется.
- Порядок модификаторов нормализуется: `Ctrl`, `Shift`, `Alt`, `Win`.
- Одинаковое сочетание нельзя назначить двум действиям в одной области действия.
- Для global/global конфликт проверяется на backend до сохранения.
- Global и main-window-local могут иметь одинаковую комбинацию: глобальный обработчик
  будет иметь приоритет вне приложения, а внутри главного окна нужно заранее определить,
  какое действие считается приоритетным.
- `Escape` используется для отмены записи, а не как сохраняемое сочетание по умолчанию.
- Не регистрировать глобальные сочетания без основной клавиши или с пустым key.

## 2. UI панели горячих клавиш

В `HotkeysPanel.vue` сделать два визуально независимых блока:

### Глобальные

Подпись блока: **Глобальные**.

Сюда входят действия, которые должны срабатывать поверх других приложений:

- главное окно;
- звуковая панель;
- пауза / стоп / повтор;
- окно управления воспроизведением;
- в будущем — другие действия, которым нужен global scope.

Под блоком показать короткое пояснение: «Работают, когда TTSBard находится в фоне».

### Главное окно

Подпись блока: **Главное окно** или **Локальные клавиши**.

Сюда добавлять действия, применимые только когда сфокусировано именно главное окно
TTSBard:

- открыть/закрыть меню;
- переход между вкладками/панелями;
- фокус редактора;
- локальные команды редактора;
- переключение режимов и элементов основного UI.

Не включать сюда клавиши playback-control и soundpanel: это отдельные окна/модели.
Под блоком показать: «Работают только внутри главного окна TTSBard».

Экспериментальный NumPad Control должен оставаться отдельной панелью/настройкой
«Перехват», а не третьим UI-блоком в списке обычных горячих клавиш.

Лучше описывать строки через конфигурационный список `HotkeyActionDefinition`, а не
дублировать разметку и `if` для каждого нового имени. В definition должны быть:
`id`, `scope`, `label`, `icon`, `default_hotkey`, `description`. Backend должен
использовать тот же список action id для валидации и регистрации.

На первом подэтапе можно оставить существующие global-команды в текущем DTO и добавить
application-команды отдельным DTO/разделом, если это уменьшает риск миграции настроек.
Но UI-модель должна сразу иметь `scope`, чтобы не возвращаться к единому плоскому списку.

## 3. Возврат в предыдущее окно

### Почему снятия фокуса недостаточно

DOM-фокус (`element.blur()`) касается только элемента внутри WebView. Tauri-окно
остаётся Windows foreground window. Даже снятие фокуса с native-окна не задаёт Windows,
какое другое окно должно стать активным. Поэтому для возврата нужно либо скрыть окно,
либо явно активировать ранее сохранённое окно.

Windows ограничивает произвольный вызов `SetForegroundWindow`; приложение не всегда
может принудительно отобрать foreground у другого процесса. Это значит, что решение
должно иметь мягкий fallback: показать/скрыть окно и не считать невозможность явной
активации ошибкой.

### Основной сценарий: TTSBard на втором мониторе и игра на первом

Целевой flow:

1. Игра является foreground-окном на первом мониторе.
2. Пользователь нажимает глобальную клавишу TTSBard.
3. До `show/set_focus` TTSBard получает и сохраняет `GetForegroundWindow()` игры.
4. Главное окно TTSBard на втором мониторе становится активным; пользователь вводит текст.
5. Локальная клавиша главного окна, например `Ctrl+Enter` или отдельное назначаемое
   сочетание «Вернуть фокус», вызывает `SetForegroundWindow(saved_game_hwnd)`.
6. TTSBard остаётся видимым на втором мониторе, но игра снова получает клавиатурный ввод.

Это не toggle visibility и не закрытие окна. У действия должна быть отдельная семантика,
например `return_to_previous_window` / «Вернуть фокус в предыдущее окно».

Снимок нужно делать в `handle_main_window` до первого `show`, `set_always_on_top` и
`set_focus`: в этот момент foreground — игра. Если снимать снимок после активации TTSBard,
будет сохранено уже собственное окно приложения.

### Вариант A — рекомендуемый: вернуть foreground без скрытия

Окно TTSBard можно оставить видимым. При открытии нужно сохранить внешний
`previous_foreground_hwnd`, а при команде возврата:

1. убрать `always_on_top`, если он был включён для окна TTSBard;
2. вызвать `SetForegroundWindow(previous_foreground_hwnd)`;
3. проверить `GetForegroundWindow()` и результат вызова;
4. если Windows отказала, оставить TTSBard видимым, но показать его без принудительного
   перехвата ввода/foreground.

`SetActiveWindow` здесь недостаточен: он работает для окна, связанного с очередью
сообщений вызывающего потока, и не активирует background-приложение. Для другого
процесса нужен `SetForegroundWindow`. Вызов действительно активирует сохранённое окно
и направляет ему клавиатурный ввод, но Windows может отказать из-за foreground lock.

Это соответствует формулировке «вернуть фокус»: окно TTSBard не исчезает, но перестаёт
быть активным, а пользователь продолжает работу в прежнем приложении. Если TTSBard
должен остаться поверх всех окон как информационная панель, можно использовать режим
`TOPMOST | NOACTIVATE`, но тогда фокус вернётся, а панель останется видимой поверх игры.

### Вариант B — простой toggle с сокрытием

`hide()` — отдельный UX-вариант, а не реализация возврата фокуса. Он действительно
скрывает окно. После этого Windows часто активирует предыдущее окно из z-order, но
это побочный эффект скрытия, а не адресное восстановление нужного окна. Использовать
его только если желаемое поведение — «закрыть панель и вернуться к работе».

### Вариант C — надёжное явное восстановление

Если требуется именно «вернуть то окно, которое было перед TTSBard», хранить
`previous_foreground_hwnd` до активации окна приложения:

1. Получить `GetForegroundWindow()` перед `show/set_focus`.
2. Не сохранять собственные окна TTSBard и системные служебные окна.
3. При команде возврата проверить, что `HWND` ещё существует и относится к текущему
   пользовательскому desktop/session.
4. При необходимости восстановить minimized-окно.
5. Вызвать `SetForegroundWindow(hwnd)` и проверить результат.
6. Если вызов запрещён политикой foreground — сначала скрыть окно TTSBard и позволить
   Windows выбрать предыдущее окно; при неудаче не показывать ложное сообщение об ошибке.

Для случая, когда TTSBard уже был активен, одного снимка недостаточно. Нужен небольшой
стек/последний внешний foreground window, обновляемый при смене foreground окна
(например, через Windows event hook `EVENT_SYSTEM_FOREGROUND`) с фильтрацией окон
самого приложения. Это надёжнее, чем пытаться вычислять «предыдущее окно» через z-order
после факта.

`AttachThreadInput` не следует использовать как основной механизм: он связывает
очереди ввода потоков, сбрасывает состояние клавиш и имеет дополнительные ограничения.
Его можно рассматривать только как узкий fallback после измерений, если обычный
`SetForegroundWindow` не покрывает нужный сценарий.

### Рекомендация по реализации

Начать с варианта A для возврата foreground без скрытия окна. Параллельно заложить
backend abstraction `focus_history`/`window_activation`, чтобы вариант C можно было
добавить без изменения UI-контракта. Сначала проверить поведение на игре, браузере,
редакторе с несохранёнными данными и окне, запущенном от администратора.

## Предлагаемый порядок реализации

1. Зафиксировать три независимые модели: global, experimental NumPad Control,
   main-window local.
2. Разнести UI на блоки «Глобальные» и «Главное окно»; NumPad Control оставить отдельным
   разделом/режимом «Перехват».
3. Добавить main-window-local dispatcher на `keydown` с `code`, областью фокуса и
   `preventDefault` только после найденного binding.
4. Добавить toggle-hide для уже сфокусированного окна.
5. Провести ручную проверку возврата focus; `hide()` не использовать, если окно должно
   оставаться видимым.
6. Только если нужен fallback для Windows foreground lock, добавить сохранение `HWND` и
   Win32 foreground bridge.
7. Добавить backend-проверки конфликтов, тесты canonicalization и миграцию настроек.

## Зафиксированное решение для реализации

Локальное действие главного окна **«Вернуть фокус в предыдущее окно»** добавляется с
дефолтным сочетанием `Ctrl+F` и настраивается в панели горячих клавиш. Оно не является
глобальным shortcut и не регистрируется через `tauri-plugin-global-shortcut`.

При срабатывании глобальной клавиши показа главного окна backend до активации TTSBard
сохраняет текущий внешний foreground `HWND`. Локальный `Ctrl+F` вызывает native-команду,
которая снимает `always_on_top` и передаёт foreground сохранённому окну; главное окно
TTSBard остаётся видимым на втором мониторе.

## Реализовано

- `HotkeySettings.return_previous_window` с обратной совместимостью и дефолтом `Ctrl+F`;
- сохранение внешнего foreground `HWND` перед показом главного окна;
- Win32-команда `return_to_previous_window` без `hide()`;
- настройка, запись и сброс сочетания в `HotkeysPanel.vue`;
- локальный listener только в main WebView, с физическим `KeyboardEvent.code`.

## Дополнение: режим быстрого редактора

Раздел настроек **«Быстрый редактор»** должен явно описывать поведение редактора при
`Enter`, `Esc` и отправке текста в TTS. Вместо checkbox используются radio-варианты:

- **Отключено** — обычное поведение, окно не скрывается и foreground не меняется;
- **Сворачивать** — текущее поведение quick editor: после отправки/отмены окно скрывается;
- **Возвращать фокус** — после отправки/отмены окно остаётся видимым, но foreground
  передаётся сохранённому предыдущему окну через уже реализованный механизм возврата.

Старое поле `editor.quick: bool` нужно мигрировать: `false` → `disabled`, `true` →
`collapse`. Новое значение хранить как стабильную строку enum, чтобы настройки не
зависели от порядкового номера вариантов.
