# Stage 36 — Динамические Piper-провайдеры из AppData

**Дата:** 2026-07-15  
**Статус:** Backend (registry, scanner, lazy loading, settings DTO) и frontend (dynamic cards, selection, loading state) реализованы; focused tests добавлены, но Rust-тесты требуют Linux WebKitGTK dev-зависимости. Оставшаяся внешняя верификация — ручное Windows-тестирование пользователем.
**Источник:** обсуждение динамических локальных TTS-моделей + анализ текущего TTS-кода

---

## 1. Цель

Добавить в приложение локальные Piper-совместимые голосовые модели без кнопки
импорта и без ручной регистрации в настройках.

Пользователь кладёт пару файлов в каталог приложения и перезапускает TTSBard:

```text
%APPDATA%\\TTSBard\\models\\piper\\
  ru_RU-irina-medium-cloned.onnx
  ru_RU-irina-medium-cloned.onnx.json
```

При старте приложение сканирует каталог, валидные модели становятся отдельными
TTS-провайдерами, а выбранная модель загружается лениво только после выбора в UI.

---

## 2. Разделение провайдеров

Текущий `LocalTts` — это HTTP-клиент совместимого с TTSVoiceWizard локального
сервера. Он не имеет отношения к ONNX-моделям и должен быть переименован для
устранения неоднозначности:

```text
LocalTts        → LocalHttpServerTts
новый тип       → LocalModelTts
```

Семантика провайдеров:

```text
OpenAI
Silero
Локальный HTTP-сервер
Piper: Ирина cloned
Piper: Alex
```

`LocalHttpServerTts` сохраняет существующий URL/API-контракт и работает как
legacy-провайдер. `LocalModelTts` работает с конкретной Piper-моделью и не
использует HTTP-сервер TTSVoiceWizard.

Текущий `loca_tts/tts_server.py` — отдельный экспериментальный Piper HTTP-сервер:
он слушает порт 8020, загружает одну зашитую модель и не совпадает с текущим
LocalTts API (приложение ожидает URL 8124, GET с base64-ответом). Его не следует
делать обязательной частью новой архитектуры.

---

## 3. Каталог моделей и обнаружение

Первый этап не требует кнопки «Добавить модель».

При старте:

1. Создать каталог `%APPDATA%\\TTSBard\\models\\piper`, если его нет.
2. Найти `.onnx` файлы непосредственно в этом каталоге.
3. Для каждого файла найти соседний файл с суффиксом `.onnx.json`.
4. Прочитать и минимально провалидировать Piper-конфигурацию.
5. Для валидной пары создать runtime-описание `LocalModelTts`.
6. Модель не загружать в ONNX session на этом этапе.

Неполные или повреждённые пары не должны останавливать запуск приложения. Они
пропускаются, а причина пишется в лог. Серые панели и состояние «пропавшая
модель» в UI не нужны: отсутствующий файл просто не порождает провайдера.

Стабильный provider ID не должен зависеть от отображаемого имени. Предлагаемый
формат:

```text
local-piper:ru_RU-irina-medium-cloned
```

Имя в UI можно получить из metadata/config или из имени файла с последующей
нормализацией.

---

## 4. Восстановление выбранного провайдера

В настройках хранится provider ID, а не индекс позиции в списке:

```text
tts.provider = "local-piper:ru_RU-irina-medium-cloned"
```

После сканирования:

- если ID найден — он остаётся выбранным;
- если ID отсутствует — выбирается первый доступный провайдер в фиксированном
  порядке;
- новый выбор сохраняется;
- приложение показывает глобальное уведомление о том, что прежняя модель не
  найдена и выбран fallback.

Предпочтительный порядок встроенных провайдеров:

```text
OpenAI → Silero → LocalHttpServerTts → LocalModelTts-провайдеры
```

Если пользователь явно выбрал Piper-модель, её отсутствие не должно приводить к
ошибке каждого последующего TTS-запроса: fallback выполняется один раз во время
запуска.

---

## 5. Ленивое открытие модели

При обнаружении модели загружаются только metadata и пути к файлам. Реальная
ONNX session создаётся при выборе провайдера.

Состояния runtime-провайдера:

```text
Discovered  — пара файлов найдена, metadata прочитана
Loading     — создаётся runtime session
Ready       — модель готова к синтезу
Failed      — загрузка завершилась ошибкой
```

В UI карточка выбранного провайдера во время `Loading` показывает индикатор и
не позволяет начать новый конфликтующий выбор. После успеха отображается
состояние готовности, например «Модель загружена». После ошибки показывается
понятное сообщение с причиной; приложение остаётся работоспособным.

При переключении на уже загруженную модель повторная загрузка не нужна. Runtime
может держать session в кэше или выгружать неиспользуемые модели позже отдельной
оптимизацией.

---

## 6. Ограничение формата

Первый runtime явно поддерживает только Piper-compatible ONNX:

```text
model.onnx
model.onnx.json
```

Обычный произвольный ONNX-файл не является автоматически TTS-моделью. Для Piper
нужны не только ONNX graph, но и конфигурация, phoneme map, правила
фонемизации, параметры speaker/audio и ожидаемые tensor inputs.

Поэтому UI и документация должны говорить «Piper-модели» или «Piper-compatible
voices», а не «любые ONNX-модели».

Расширение на другие движки возможно позднее через отдельные адаптеры:

```text
models/piper/   → Piper runtime
models/kokoro/  → Kokoro runtime в будущем
```

---

## 7. Встроенный runtime и DLL

Целевой вариант — не запускать отдельный `piper.exe` и не поднимать HTTP-сервер.
`LocalModelTts` должен вызывать Piper/ONNX runtime внутри Tauri-приложения.

Целевой пользовательский артефакт — один `ttsbard.exe`. Piper и ONNX Runtime
должны быть статически залинкованы в приложение; отдельный `piper.exe`, Python
runtime и DLL рядом с exe не входят в целевую архитектуру. Динамическая DLL
остаётся только техническим fallback-вариантом при невозможности статической
линковки, а не частью принятого UX или установки.

Нужно отдельно проверить лицензию выбранной реализации Piper и ONNX Runtime до
финального способа линковки и распространения. Использование Python-сервера не
является обязательным и не должно определять архитектуру приложения.

---

## 8. Затрагиваемые домены реализации

Будущую реализацию лучше разделить на независимые задачи:

1. Backend: модель каталога, сканирование и стабильные provider ID.
2. Backend: `LocalHttpServerTts` и `LocalModelTts` в TTS-абстракции.
3. Backend: Piper/ONNX runtime с ленивой загрузкой и диагностикой.
4. Settings: сохранение выбранного динамического provider ID и fallback.
5. Frontend: динамический список провайдеров и loading/ready/error состояния.
6. Проверки: запуск, исчезновение модели, повреждённый JSON, ошибка runtime,
   переключение между двумя моделями.

---

## 9. Зафиксированные решения

- Старый `LocalTts` — отдельный HTTP-провайдер, переименовать в
  `LocalHttpServerTts`.
- Новый провайдер конкретной модели — `LocalModelTts`.
- Модели обнаруживаются автоматически при старте.
- Каталог первого этапа: `%APPDATA%\\TTSBard\\models\\piper`.
- Кнопки добавления и ручной регистрации моделей нет.
- Перезапуск после размещения новых файлов — часть первого UX.
- Все валидные Piper-модели появляются как отдельные TTS-провайдеры.
- ONNX session создаётся лениво при выборе модели.
- Во время загрузки показывается индикатор; после успеха — состояние готовности.
- Пропавшая выбранная модель заменяется первым доступным провайдером при старте.
- Невалидные/неполные модели не показываются в UI и пишутся в лог.
- Первый этап ограничен Piper-compatible ONNX + `.onnx.json`.
- Целевой runtime встроен в Tauri-приложение; отдельный Piper HTTP-сервер не
  является обязательной частью решения.
- Целевой артефакт распространения — один `ttsbard.exe`.
- Динамическая DLL допускается только как временный технический fallback при
  невозможности статической линковки.
