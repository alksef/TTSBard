# Stage: Переописание проекта — позиционирование и аудитория

**Дата:** 2026-07-08
**Статус:** research / ✅ РЕШЕНО (на уровне stage; правка README/PROBLEMS — отдельным планом)
**Решение:** переработать описание проекта — уйти от абстрактного «TTS для общения» к чёткому
**двойному позиционированию**: (А) люди с ограничениями речи, использующие TTS для общения;
(Б) стримеры/контент-мейкеры, использующие TTS и звуковую панель для трансляций. Толерантный
язык («ограничения речи», не «инвалидность»). Добавить историческую справку (из интервью со
стримером aleksa).
**Связано:** `README.md`, `PROBLEMS.md`, stage `14` (саундпанель для стримеров), stage `15` (AI
для двух аудиторий).

## Цель (запрос пользователя)
- Текущее описание (`README.md`, `PROBLEMS.md`) — абстрактное «TTS-приложение для общения с
  минимальным отвлечением». Не отражает **реальную аудиторию** и **историю возникновения**.
- Пользователь: проект вырос из **интервью со стримером aleksa**, которому нужен TTS-инструмент;
  потом выяснилось, что и другие стримеры, и люди с ограничениями речи нуждаются в таком инструменте.
- Требование: уйти от изначальной «проблематики инвалидности» → **толерантно** «для людей с
  ограничениями речи», с историческим контекстом (интервью).

## Исторический контекст (для описания)
- **Начало:** интервью со стримером **aleksa** — стример, использующий TTS для общения на стриме.
  Из его потребностей родились ключевые фичи: глобальный перехват клавиш, звуковую панель, быстрый
  редактор, препроцессор пресетов, OBS-интеграция.
- **Расширение:** оказалось, что те же потребности (быстрый ввод → озвучка → минимум отвлечения)
  актуальны и для других стримеров, и для людей, использующих TTS в повседневном общении из-за
  ограничений речи.

## Двойное позиционирование

### Аудитория А — люди с ограничениями речи (assistive)
- Используют TTS как основной способ общения в мессенджерах/браузере/повседневно.
- Ключевые потребности: **скорость и удобство ввода** (печатают сами каждый день), автокомплит,
  история фраз, коррекция орфографии, минимум когнитивной нагрузки.
- Язык: **толерантный** — «люди с ограничениями речи», «использующие синтез речи для общения».
  НЕ «инвалиды», НЕ «больные». Person-first language.
- AI-ценность: ⭐⭐⭐ (автокомплит, коррекция, грамматика — печатают много).

### Аудитория Б — стримеры / контент-мейкеры (broadcast)
- Используют TTS + звуковую панель для трансляций (Twitch/YouTube), озвучки чата, реакций.
- Ключевые потребности: **озвучка в реальном времени**, звуковая панель (наборы звуков),
  OBS-интеграция (WebView), Twitch-чат, экспрессивность голоса, скорость реакции во время игры.
- AI-ценность: ⭐⭐ (перефразирование, экспрессивная разметка), плюс отдельное направление
  (авто-озвучка чата — не сейчас).

> **Общий знаменатель:** оба кейса — «быстро превратить мысль в голос с минимумом отвлечения от
> основного занятия». Это и есть ядро продукта. Разные акценты в фичах, общая инфраструктура.

## Толерантный язык (гайдлайн для всех текстов)
- ✅ «люди с ограничениями речи», «использующие синтез речи для общения», «assistive use»
- ✅ «человек, который общается через TTS» (person-first)
- ❌ «инвалиды», «больные», «те, кто не может говорить» (как приговор)
- ❌ жалостливые/медицинские формулировки

## Риски / нюансы
1. **Не «выбросить» assistive-направление** — пользователь явно сказал «про инвалидность не надо,
   но ограничения в отсутствие речи — толерантно». Значит: **сохранить assistive-кейс**, но
   подать через толерантный язык, не делать его единственным/жалостливым.
2. **Двойное позиционирование — риск «размытия»** — README может стать «всем для всех». Решение:
  единое ядро («быстрый мысль → голос, минимум отвлечения») + две аудитории как акценты, не два
  разных продукта.
3. **Историческая справка** — упомянуть интервью с aleksa как **origin story** (коротко, 1-2
   предложения), не превращать README в мемуары. Возможно — в отдельный раздел «История» внизу.
4. **Актуальность после stages 12-15** — README править **после** реализации (или хотя бы после
   планов), чтобы описать фичи как «есть», а не «планируется». Сейчас в stage — зафиксировать
   направление; саму правку README — отдельным планом, ближе к релизу.
5. **Локализация** — README на русском (как сейчас); если есть английская версия — синхронизировать.
