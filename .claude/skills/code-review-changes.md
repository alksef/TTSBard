# Code Review Changes Skill

Проверяйте результат задачи относительно baseline и только в согласованном
scope. В dirty worktree команда `git diff HEAD` без ограничения путей смешивает
работу агента с пользовательскими изменениями и не задаёт корректную границу.

## Процесс

1. Прочитать цель, разрешённые пути и критерии приёмки task-файла.
2. Сравнить текущий `git status --short` с сохранённым baseline status.
3. Просмотреть diff разрешённых путей: `git diff -- <path...>`; отдельно учесть
   новые и удалённые файлы.
4. Прочитать изменённые файлы целиком и трассировать затронутый сценарий.
5. Проверить correctness, regression risks, безопасность, ошибки, тесты и
   соответствие архитектурным decisions.
6. Запустить проверки из `build-validation` по риску.
7. Сначала перечислить findings с `file:line` и severity; затем дать краткий
   verdict. При отсутствии findings сказать это явно и назвать непроверенные
   риски.

Не принимать checklist автора изменений как доказательство и не включать
несвязанные изменения в commit. При замечаниях формулируется следующий узкий
corrective task; исправлять их можно только в соответствии с текущей ролью и
разрешённым scope.
